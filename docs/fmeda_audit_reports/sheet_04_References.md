## 1. Sheet 概述

此工作表名稱為 **References**，在整個 FMEDA 分析報告中位於第 4 個位置。其維度為 32 行 × 2 列。此工作表的核心目的在於建立一份權威性的參考文獻清單，作為整個 FMEDA 分析所依據的標準、技術報告、數據手冊及方法論的基礎。在功能安全稽核中，這份清單是追溯性與方法論正確性的基石，它向稽核員展示了所有計算、假設和決策皆基於公認的行業標準與數據來源，而非憑空臆測。

此工作表扮演著 FMEDA 報告「證據鏈」的起點。它不僅列出了所遵循的標準（如 ISO 26262），也指明了失效率數據的來源（如 IEC TR 62380、SN 29500）。稽核員會透過此頁面，快速評估分析團隊所使用的方法論是否具備行業公信力，並確認所有後續的量化分析是否建立在一個穩固且可被驗證的基礎之上。

## 2. 內容結構分析

此工作表的結構極為簡潔且目標明確，旨在提供清晰的參考文獻索引。

- **欄位標頭**: 工作表使用 A2 與 B2 儲存格作為標頭，分別是「**Ref. No**」（參考編號）與「**Standard / Document Name**」（標準/文件名稱）。「Ref. No」提供了一個唯一的數字索引，便於在報告的其他部分（如 FMEDA 主表或失效率計算表）中引用特定的參考文獻。「Standard / Document Name」則詳細說明了該文獻的完整標題、版本及發布年份。

- **合併儲存格**: 儲存格 `A1:B1` 被合併，並以粗體、深藍色背景標示出標題「**References**」，在視覺上統整了整個工作表的內容，使其目的性一目了然。

- **資料組織邏輯**: 資料從第 3 行開始，以表格形式逐行條列。每一行代表一份獨立的參考文獻，A 列是其序號，B 列是其詳細名稱。這種線性且簡單的組織方式確保了資訊的易讀性與可追溯性，讓任何閱讀者都能迅速找到特定標準或數據手冊的全名。

根據提供的資料，此工作表中**不包含任何 Excel 公式**。所有內容均為靜態文本輸入，其功能純粹是資訊展示與索引，不涉及任何動態計算。

## 5. 稽核合規性分析

此「References」工作表雖然不直接進行計算，但它對整個 FMEDA 的合規性至關重要，是稽核的起點。它主要對應以下 ISO 26262 的條款與概念：

- **ISO 26262-2:2018, Clause 6 - Management of functional safety**: 此工作表是展現「安全文化」與「品質管理系統」的證據之一。一個管理良好、有系統的專案，其所有分析方法與數據來源都應被明確記錄和追溯，此工作表正是此類記錄的體現。

- **ISO 26262-5:2018, Annex D - Quantitative analysis of hardware architectural metrics**: 附錄 D 提供了計算 SPFM、LFM 和 PMHF 的詳細指引。要正確執行這些計算，必須依賴可靠的元件失效率數據。此工作表列出的 **IEC TR 62380** 和 **SN 29500** 等失效率手冊，正是滿足此條款要求的數據來源基礎。稽核員會檢查 FMEDA 中使用的失效率是否與這些手冊中的數據一致或基於其模型進行推算。

- **ISO 26262-8:2018, Clause 9 - Supporting processes**: 此條款要求對所使用的工具與軟體進行信心評估。雖然此處列出的是參考標準而非軟體工具，但其邏輯相通：稽核員需要確認分析所依賴的「輸入」（即標準與數據）是可靠且被行業認可的。

- **滿足 ASIL B 等級的要求**: 對於 ASIL B 等級，雖然對硬體度量的要求不如 ASIL C/D 嚴格，但仍然要求有系統化的方法來進行硬體安全分析。提供一份清晰的參考文獻列表，證明分析是基於如 ISO 26262、IEC 61709 等公認標準，是滿足 ASIL B 所需的「系統性方法」的有力證據。

**稽核員可能會問的問題**: 
1. 「你們在計算元件失效率時，主要依據哪一份參考文件？是 IEC TR 62380 還是 SN 29500？兩者的適用範圍和假設有何不同？」
2. 「你們使用的 ISO 26262 是 2011 年版還是 2018 年版？兩者在硬體度量分析上有何差異，你們是如何應對的？」
3. 「IEC TR 62380 已被撤銷，為何你們仍將其列為參考文獻？是否有替代或更新的數據來源？」
4. 「請展示你們是如何應用 DIN EN 61709 中的轉換模型，將參考條件下的失效率轉換到你們產品的實際工作環境中的？」

**需要準備的佐證文件**: 
1. **所有列出的標準與文件的電子或紙本副本**：稽核員可能隨時要求查閱原文，以確認團隊對標準的理解是否正確。
2. **版本控制記錄**: 證明使用的是特定版本的文件，並解釋為何選擇該版本。
3. **失效率數據選擇報告**: 一份內部文件，說明為何選擇特定的數據手冊（如 SN 29500），以及如何根據產品特性應用其中的數據。
4. **工具評估報告**: 如果使用了軟體工具來實現這些標準中的模型（例如，一個實現了 IEC TR 62380 計算模型的內部腳本），則需要提供該工具的評估與驗證報告。

## 6. 稽核檢查清單

以下是稽核員針對此「References」工作表會逐項檢查的項目：

| 檢查項目 | 檢查內容 | Pass/Fail 判定標準 |
| :--- | :--- | :--- |
| **完整性** | 是否所有在 FMEDA 分析中使用的主要標準、方法論和數據來源都已被列出？ | **Pass**: 清單全面，涵蓋了 ISO 26262、失效率數據手冊等所有關鍵文件。<br>**Fail**: 遺漏了關鍵的參考文件，例如在計算中使用了某個數據庫的失效率，但未在此處列出。 |
| **準確性** | 列出的文件名稱、編號和版本是否準確無誤？ | **Pass**: 所有文件資訊均準確，與官方發布一致。<br>**Fail**: 文件名稱、版本號或發布年份存在錯誤，可能導致引用混淆。 |
| **一致性** | FMEDA 其他部分（如失效率計算表）引用的參考文獻編號或名稱，是否與此處的清單完全一致？ | **Pass**: 整個報告中的引用保持一致，可雙向追溯。<br>**Fail**: 其他工作表引用了未在此清單中定義的文獻，或引用的編號與此處不符。 |
| **時效性** | 是否使用了最新或最合適的標準版本？對於已撤銷的標準（如 IEC TR 62380），是否有合理的理由支持其繼續使用？ | **Pass**: 使用了現行有效的標準版本，或對使用過時標準給出了充分且可接受的解釋（例如，行業慣例或客戶指定）。<br>**Fail**: 仍在使用已被取代的過時標準，且沒有提供合理解釋，可能導致分析方法落後。 |
| **可獲得性** | 開發團隊是否能夠在稽核員要求時，立即提供所有列出的參考文件以供查閱？ | **Pass**: 團隊擁有所有參考文件的副本，並能隨時提供。<br>**Fail**: 團隊無法提供所列出的文件，表明其可能並未真正依據該文件進行分析。 |
