# FMEDA 深度稽核分析報告：Sheet 11 - SRAM Tran FIT

## 1. Sheet 概述

本文件為針對 FMEDA (Failure Modes, Effects, and Diagnostic Analysis) 工作簿中第 11 個工作表「SRAM Tran FIT」的深度稽核分析報告。此工作表在整個功能安全分析中扮演著關鍵的角色，專注於評估靜態隨機存取記憶體 (SRAM) 的瞬態故障率。

| 項目 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `SRAM Tran FIT` |
| **在 FMEDA 的位置** | 工作簿中的第 11 個 Sheet |
| **維度** | 50 行 × 35 列 (A1:AI50) |
| **核心目的** | 計算和匯總由單粒子效應 (Single Event Effects, SEE)，特別是中子 (Neutron) 和 Alpha 粒子引發的 SRAM 瞬態故障（或稱軟錯誤，Soft Errors）的失效率 (FIT, Failure in Time)。 |

此工作表的目的在於將底層的物理現象（輻射導致的位元翻轉）與 ISO 26262 的量化安全指標（如 PMHF）聯繫起來。它詳細列出了設計中使用的每一個 SRAM 實例，並根據其物理特性、運作環境和已實施的安全機制（如 ECC），來量化其對整體系統失效率的貢獻。這對於證明硬體設計能夠滿足目標 ASIL 等級所要求的硬體架構度量至關重要。

## 2. 內容結構分析

「SRAM Tran FIT」工作表的結構經過精心設計，以系統化的方式從單個記憶體實例的參數，逐層匯總至整個晶片的瞬態總失效率。其結構邏輯清晰，便於追溯與審核。

### 欄位標頭的含義與組織方式

工作表的主要欄位 (A-Q) 用於定義和計算每個 SRAM 實例的失效率，而次要欄位 (S-Y) 則用於匯總和計算不同類型記憶體的總體失效率。主要欄位可分為以下幾類：

| 類別 | 欄位 | 含義 |
| :--- | :--- | :--- |
| **識別與屬性** | `Island`, `Instance`, `Type`, `Library` | 定義了 SRAM 的層次結構位置、具體實例名稱、類型（如 one port SRAM）及所使用的標準單元庫。 |
| **物理參數** | `Depth(Bit)`, `Bus Width`, `Area(um^2)` | 描述了 SRAM 的物理規模，包括深度、寬度（位元）和在晶片上佔據的面積。 |
| **失效率計算** | `neutron SER`, `Alpha SER`, `SUM` | 核心計算欄位，分別計算由中子和 Alpha 粒子引發的原始失效率（單位：FIT/Mbit），並加總。 |
| **安全機制** | `Sram Protection`, `neutron SER with ECC`, `Alpha SER with ECC` | 記錄了該 SRAM 是否有錯誤修正碼 (ECC) 等保護機制，並計算啟用保護後降低的失效率。`SECDED` (Single Error Correction, Double Error Detection) 是常見的保護類型。 |
| **輔助計算** | `size(bit)`, `Szie (KB)` | 將 SRAM 的深度和寬度轉換為總位元數和 KB 數，方便參考。 |

### 合併儲存格的用途

工作表中多處使用了合併儲存格，其主要目的是為了對資料進行分組和歸類，增強可讀性。

- **`A2:A3`, `A4:A7`, `A8:A50`**: 縱向合併，將具有相同「Island」編號的 SRAM 實例歸為一組，體現了物理佈局或邏輯分區上的關聯性。
- **`B27:O27`, `B28:O28`**: 橫向合併，用於創建視覺分隔線或標題，標示出工作表的總結或特殊計算區域。

### 資料的組織邏輯

資料以「由細到粗」的邏輯進行組織。每一行（從第 2 行開始）代表一個具體的 SRAM 實例。工作表首先計算每個實例的個體失效率，然後在工作表的右側區域 (S-Y)，透過 `SUMIF` 等公式，從另一個名為 `BlockList` 的工作表中提取資訊，對不同類型的記憶體（如 SRAM、Flip-Flop）的總失效率進行分類匯總。這種結構使得稽核員既可以深入到單個元件的細節，也可以快速掌握整個系統的宏觀指標。

## 4. 公式深度分析

此工作表包含 231 個公式，其中有 10 種獨特的計算模式。這些公式是實現瞬態故障率量化分析的核心。以下將對關鍵的公式模式進行深度解析。

| 公式模式 | 物理意義 / 計算目的 | 引用關係 | 對應標準 & 驗證要點 |
| :--- | :--- | :--- | :--- |
| **`=D2*E2`** | 計算單個 SRAM 實例的總位元數 (Total Bits)。 | `D2`: Depth, `E2`: Bus Width | **驗證要點**: 檢查深度和寬度的單位是否正確，乘積是否代表了完整的記憶體陣列大小。 |
| **`=P2/(8*1024)`** | 將總位元數 (來自 P 欄) 轉換為千位元組 (Kilobytes)。 | `P2`: size(bit) | **驗證要點**: 檢查轉換因子是否正確（1 KB = 8 * 1024 bits）。 |
| **`=D2*E2/1000000*AA3`** | 計算中子引發的軟錯誤率 (Neutron SER)。 | `D2*E2`: 總位元數<br>`1000000`: 轉換為百萬位元 (Mbit)<br>`AA3`: 基礎中子 SER 率 (FIT/Mbit) | **ISO 26262-11:2018, Clause 6**: 提供了關於半導體輻射效應的指導。<br>**驗證要點**: 必須驗證 `AA3` 所引用的基礎失效率數據的來源與合理性，通常來自 JEDEC JESD89A 標準或晶圓廠提供的數據。 |
| **`=D2*E2/1000000*Z3`** | 計算 Alpha 粒子引發的軟錯誤率 (Alpha SER)。 | `Z3`: 基礎 Alpha SER 率 (FIT/Mbit) | **同上**。<br>**驗證要點**: 驗證 `Z3` 數據的來源，通常與封裝材料的純度相關。 |
| **`=SUM(H2:I2)`** | 加總中子和 Alpha 粒子引發的 SER，得到該實例的總瞬態失效率。 | `H2`: Neutron SER<br>`I2`: Alpha SER | **驗證要點**: 確保加總範圍包含了所有相關的瞬態故障源。 |
| **`=SUMIF(...)`** | 從 `BlockList` 工作表按條件匯總特定類型元件的總位元數。 | `[1]BlockList!R:R`: 條件範圍<br>`[1]BlockList!AO:AO`: 加總範圍 | **驗證要點**: 這是跨表計算的關鍵。必須核實 `BlockList` 中的數據是否準確，`SUMIF` 的條件（如 `Si MOS: High speed SRAM, FIFO`）是否能正確匹配所有目標元件。樣本中的 `#VALUE!` 錯誤表明存在匹配失敗或數據錯誤，這是稽核的**重點關注對象**。 |
| **`=(U2+V2)*T2/1024/1024`** | 計算一類元件（如 SRAM）的總瞬態 FIT。 | `U2+V2`: 總 FIT/Mbit<br>`T2`: 總位元數 | **驗證要點**: 檢查 `T2`（來自 `SUMIF`）是否正確，以及 `U2` 和 `V2` 是否代表了正確的基礎失效率。分母 `1024*1024` 是將 bit 轉換為 Mbit。 |

## 5. 稽核合規性分析

此工作表的內容與 ISO 26262 的多個部分緊密相關，是證明硬體設計滿足 ASIL B 等級要求的關鍵證據之一。

### 對應的 ISO 26262 條款

- **ISO 26262-5:2018, Clause 8 & 9**: 關於硬體架構度量 (SPFM, LFM) 和 PMHF 的評估。此表計算的瞬態失效率是 PMHF 計算的直接輸入。雖然軟錯誤通常被視為多點故障（因為它們通常能被 ECC 等機制檢測和糾正），但其殘餘失效率仍需計入 PMHF。
- **ISO 26262-11:2018, Clause 6**: 提供了關於在半導體中應用 ISO 26262 的詳細指南，特別是附錄 B 中提到了單粒子效應 (SEE) 的處理方法。此工作表正是該條款的具體實踐。
- **ISO 26262-9:2018, Clause 8**: 關於相依故障分析 (DFA)。雖然此表主要計算隨機故障，但如果多個 SRAM 實例共享相同的電源或時鐘，則需要考慮其瞬態故障的共因故障 (CCF) 可能性。

### 滿足 ASIL B 等級的具體要求

對於 ASIL B，PMHF 的目標值為 < 100 FIT。此工作表計算出的總瞬態失效率，將與其他永久性故障（Stuck-at, etc.）的失效率一同匯總，最終用於證明 PMHF < 100 FIT。此外，如果 SRAM 的軟錯誤被聲稱由安全機制（如 ECC）覆蓋，則該機制的診斷覆蓋率 (DC) 需要被量化，並用於計算 LFM 指標。

### 稽核員可能會問的問題

1.  用於計算中子和 Alpha SER 的基礎失效率數據 (FIT/Mbit) 的來源是什麼？是否有第三方認證報告或晶圓廠的官方數據支持？
2.  `SUMIF` 公式出現 `#VALUE!` 錯誤的原因是什麼？請展示 `BlockList` 工作表的相關內容，並解釋為何無法正確匹配。
3.  對於聲稱有 `SECDED` 保護的 SRAM，其診斷覆蓋率 (DC) 是如何確定的？是否有故障注入測試 (Fault Injection Testing) 的證據來支持聲稱的 DC 值？
4.  此處計算的瞬態故障率是如何整合到最終的 PMHF 計算中的？請展示從此表的輸出到 PMHF 總表的數據流。
5.  是否考慮了不同操作模式（Mission Profiles）對瞬態故障率的影響？

### 需要準備的佐證文件

- 晶圓廠或第三方實驗室提供的 SER 測試報告。
- JEDEC JESD89A 等行業標準文件。
- 故障注入測試的計劃、報告和日誌。
- 完整的 FMEDA 報告，特別是 `BlockList` 和最終的 PMHF 匯總表。
- 設計規格書，描述 ECC 等安全機制的詳細實現。

## 6. 稽核檢查清單

基於以上分析，稽核員將使用以下清單對「SRAM Tran FIT」工作表進行逐項檢查。稽核風險等級被評估為**中等**，主要風險點在於跨表引用的數據準確性和基礎失效率數據的來源有效性。

| # | 檢查項目 | Pass/Fail 判定標準 |
|:-:|:---|:---|
| 1 | **數據來源可靠性** | **Pass**: 所有基礎失效率（如 `AA3`, `Z3`）均有可追溯的、可靠的數據來源（如 JEDEC 標準、晶圓廠報告）。<br>**Fail**: 使用了無依據的、假設的或過時的數據。 |
| 2 | **公式準確性** | **Pass**: 所有計算公式（如 SER 計算、單位轉換）均符合其物理意義和標準定義。<br>**Fail**: 公式中存在邏輯錯誤、引用錯誤或不正確的常數。 |
| 3 | **跨工作表引用** | **Pass**: `SUMIF` 等跨表引用公式能夠正確、完整地從 `BlockList` 等源工作表中提取數據，無 `#VALUE!` 等錯誤。<br>**Fail**: 存在錯誤或無法解析的跨表引用，導致匯總數據不準確。 |
| 4 | **安全機制有效性** | **Pass**: 對於聲稱有 ECC 保護的 SRAM，其診斷覆蓋率有合理的量化依據（如故障注入測試）。<br>**Fail**: 聲稱了保護機制但無證據支持其有效性或覆蓋率。 |
| 5 | **數據一致性** | **Pass**: 工作表內的數據（如 SRAM 列表、尺寸）與設計規格書、`BlockList` 中的數據完全一致。<br>**Fail**: 不同文件或工作表之間存在數據矛盾。 |
| 6 | **完整性** | **Pass**: 設計中所有的 SRAM 實例都已包含在此工作表中進行分析。<br>**Fail**: 遺漏了部分 SRAM 實例，導致計算結果偏低。 |
| 7 | **與 PMHF 的整合** | **Pass**: 此工作表計算出的總瞬態失效率被正確地計入了最終的 PMHF 匯總計算中。<br>**Fail**: 數據流中斷，此表的計算結果未被或被錯誤地用於最終的安全度量評估。 |
