# FMEDA 深度稽核分析報告：Sheet 9 - FailureModes

## 1. Sheet 概述

此工作表為 FMEDA 分析中的 **第 9 個 sheet**，名為 **FailureModes**。

| 屬性 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `FailureModes` |
| **在報告中位置** | 第 9 個 Sheet |
| **維度** | 338 行 × 3 列 (A1:C338) |

### 角色與目的

此 `FailureModes` 工作表在整個 FMEDA 分析流程中，扮演著一個**基礎知識庫與參考指南**的角色。它並非用於執行計算，而是根據 **ISO 26262-11:2018** 標準（特別是其 Annex B），系統性地列舉了數位半導體元件中可能出現的各種失效模式（Failure Modes）。

其核心目的在於為執行 FMEDA 的工程師提供一個權威且標準化的起點，確保在分析硬體元件（如 CPU、記憶體、匯流排等）時，能夠全面地考慮到業界公認的潛在失效情境。這有助於：

1.  **標準化分析過程**：確保不同團隊或工程師在進行失效模式分析時，有一致的參考依據。
2.  **提高分析完整性**：防止遺漏關鍵的、已知的失效模式，從而提升 FMEDA 的品質與可信度。
3.  **作為稽核證據**：在稽核過程中，此表單可作為證據，證明團隊的 FMEDA 分析是基於國際標準所建議的實踐。

總結來說，`FailureModes` sheet 是 FMEDA 分析的「字典」，定義了分析所使用的「詞彙」，確保了整個分析的嚴謹性與合規性。

## 2. 內容結構分析

此工作表的結構清晰，旨在以層次化的方式呈現硬體元件及其對應的失效模式。

### 欄位標頭

| 欄位 | 標頭名稱 | 意義與解釋 |
| :--- | :--- | :--- |
| **A** | `Part / sub-part` | 定義硬體元件的層級。可以是一個大的功能區塊（如 CPU），也可以是其中的子模組（如 CPU Interrupt Handler）。 |
| **B** | `Function` | 詳細描述該元件或子模組所應執行的預期功能。這是判斷「失效」的基準。 |
| **C** | `Aspects to be considered for Failure mode` | 列出針對該元件功能可能發生的各種失效模式。這些描述非常具體，例如「指令未執行 (omission)」、「非預期指令被執行 (commission)」、「執行延遲 (too late)」等。 |

### 資料組織邏輯

資料以**元件為中心**進行組織。Column A 中的一個主要元件（如 `CPU`）會對應多個子元件（如 `CPU Interrupt Handler circuit`, `CPU Memory Management Unit`）。每個子元件在 Column B 中有其特定的功能描述，並在 Column C 中列出其專屬的失效模式清單。

這種結構使得使用者可以快速地根據所分析的硬體架構，找到對應元件的潛在失效模式，並將其應用到主要的 FMEDA 計算表中。

### 合併儲存格的用途

合併儲存格在此工作表中主要有兩種用途：

1.  **標題與分組**：如 `A1:C1` 用於整個工作表的標題，而 `A3:C3`、`A9:C9` 等則用於標示一個大的元件類別（如 `CPU`, `Memory`），使其下的子元件一目了然。
2.  **對齊與可讀性**：當一個元件（如 `A55:A59` 的 `Memory`）包含多個子功能區塊時，透過垂直合併儲存格，可以清晰地表明這些子區塊都隸屬於同一個上層元件，增加了報告的可讀性。

## 3. 圖片分析

此 `FailureModes` 工作表中**不包含任何圖片**。這符合其作為一個純文字參考資料庫的角色定位。其所有資訊均透過結構化的文字來傳達。

## 4. 公式深度分析

此 `FailureModes` 工作表中**不包含任何 Excel 公式**，其 `formula_count` 為 0。這再次確認了它的功能是一個靜態的資料庫，而非動態的計算表。所有內容均為預先填寫的標準文字資訊。

## 5. 稽核合規性分析

此工作表雖然不涉及計算，但在 ISO 26262 稽核中卻是**合規性的重要基石**。稽核員會透過此表來評估團隊是否對「失效模式」有著標準化且全面的理解。

### 對應的 ISO 26262 條款

| 相關條款 | 關聯性說明 |
| :--- | :--- |
| **ISO 26262-11:2018, Annex B** | **直接來源**：此工作表的內容是該附錄「Examples of failure modes for digital components」的具體實踐。稽核員會核對此表的內容是否與標準建議一致。 |
| **ISO 26262-5:2018, Clause 8.4.3** | **分析基礎**：該條款要求在 FMEDA 中識別硬體元件的失效模式。此工作表為滿足該要求提供了必要的輸入和指南。 |
| **ISO 26262-9:2018, Clause 5** | **ASIL 分解的基礎**：相依失效分析（DFA）中，需要理解元件的失效模式以評估其傳播路徑。此表提供了這些失效模式的標準化描述。 |

### 滿足 ASIL B 等級的要求

對於 ASIL B 等級的產品，雖然其要求不如 ASIL C/D 嚴格，但仍然需要一個**系統化且有文件記錄的** FMEDA 流程。使用此標準化的 `FailureModes` 工作表，可以向稽核員證明：

*   團隊並非憑空想像失效模式，而是基於國際標準進行分析。
*   分析過程具有可追溯性與一致性。
*   即使是 ASIL B，團隊也採用了嚴謹的方法論來確保硬體可靠性分析的品質。

### 稽核員可能會問的問題

1.  「你們在 FMEDA 中所使用的失效模式，其來源是什麼？如何確保其完整性？」
    *   **回答策略**：展示此 `FailureModes` 工作表，並說明其內容是基於 ISO 26262-11:2018 Annex B。
2.  「對於你們產品中一個特定的元件（例如，DMA 控制器），你們是如何確定其失效模式的？請展示分析過程。」
    *   **回答策略**：在 `FailureModes` 表中找到 DMA 對應的條目，並解釋如何將這裡列出的標準失效模式（如 `DMA_FM1` 到 `DMA_FM6`）應用到主 FMEDA 分析表中。
3.  「如果一個元件在此列表中不存在，你們的處理流程是什麼？」
    *   **回答策略**：說明團隊會依循 ISO 26262-11 的原則，結合元件的 datasheet 和設計規格，定義新的、合理的失效模式，並將其記錄下來以備查。

### 需要準備的佐證文件

*   **硬體架構圖**：用於將 `FailureModes` 表中的元件與實際設計對應起來。
*   **主 FMEDA 分析表**：展示 `FailureModes` 表中的內容是如何被實際引用的。
*   **ISO 26262-11:2018 標準文件**：在被問及來源時，可以隨時查閱。

## 6. 稽核檢查清單

稽核員在審查此工作表及其應用時，會關注以下項目。此清單可作為內部準備的參考。

| 檢查項目 | 檢查內容 | Pass/Fail 判定標準 |
| :--- | :--- | :--- |
| **1. 來源可追溯性** | 內容是否基於公認的標準（如 ISO 26262-11）？ | **Pass**: 能明確指出內容來源於 ISO 26262-11 Annex B。<br>**Fail**: 無法說明來源，或來源為非標準化的內部文件。 |
| **2. 應用一致性** | 主 FMEDA 表中使用的失效模式，是否與此表的命名和定義一致？ | **Pass**: 主 FMEDA 表中的失效模式能在此表中找到對應條目，或遵循了相同的定義原則。<br>**Fail**: 主 FMEDA 表中的失效模式描述混亂、不標準，且與此表無關。 |
| **3. 完整性** | 對於一個給定的元件，是否考慮了此表中所有相關的失效模式？ | **Pass**: FMEDA 分析涵蓋了此表中列出的適用失效模式。<br>**Fail**: 明顯遺漏了關鍵的失效模式（例如，只分析了功能遺漏，未分析功能誤執行）。 |
| **4. 客製化與擴展** | 對於表中未列出的客製化元件，是否有流程來定義其失效模式？ | **Pass**: 有文件化的流程來分析和定義新元件的失效模式，且定義合理。<br>**Fail**: 對於新元件，沒有分析其失效模式，或分析過程沒有記錄。 |

### 稽核風險等級評估

此工作表本身的內容是靜態且源於標準，因此其**固有風險為「低」（Low）**。然而，它在整個 FMEDA 流程中的**應用風險為「高」（High）**，因為如果此表沒有被正確地應用，將直接導致整個 FMEDA 分析的基礎出現嚴重缺陷。稽核員會將重點放在「應用」而非「內容」本身。
