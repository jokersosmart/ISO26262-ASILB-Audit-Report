# FMEDA 深度稽核分析報告：Sheet 15 - FMEDA

## 1. Sheet 概述

此份文件為針對 FMEDA (Failure Modes, Effects, and Diagnostic Analysis) 報告中第 15 個工作表「FMEDA」的深度稽核分析報告。此工作表是整個 FMEDA 分析的核心，其目的在於系統性地識別硬體元件的失效模式，並根據其對安全目標的影響以及現有安全機制的覆蓋率，進行量化分析與分類。

| 項目 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | FMEDA |
| **在報告中的位置** | 第 15 個 Sheet |
| **維度** | 463 欄 × 1659 列 (A1:QU1659) |
| **角色與目的** | 此工作表是 FMEDA 方法論的具體實踐，用於對每一個硬體元件的每種失效模式進行詳細評估。它將失效模式分類為單點故障 (SPF)、殘餘故障 (RF) 或多點故障 (MPF)，並計算相關的失效率，最終匯總成 SPFM、LFM 和 PMHF 等頂層安全度量，以證明系統符合 ISO 26262 所要求的 ASIL 等級。 |

## 2. 內容結構分析

「FMEDA」工作表的結構極為複雜，旨在引導分析人員逐步完成對每個失效模式的評估。其結構設計緊密圍繞 ISO 26262-5 的要求展開。

- **欄位標頭的含義與組織方式**
  - 標頭分為多個層級，利用合併儲存格將相關欄位分組，例如 `AO1:BL1` 下的「Diagnosis type to be considered」群組。
  - 從左至右，欄位邏輯依序為：
    1.  **元件識別** (A-E欄): 標識硬體元件的層次結構。
    2.  **基本失效率** (F-G欄): 輸入元件的永久和瞬態失效率 (λ)。
    3.  **失效模式分析** (H-K欄): 定義失效模式及其分佈，並描述其影響。
    4.  **安全目標違反評估** (L-AD欄): 針對多達 18 個安全目標 (SGV)，評估此失效模式是否會導致違規。
    5.  **安全機制與診斷覆蓋率** (AE-MD欄): 針對不同的診斷類型（內部、UseCase1-3），定義安全機制及其診斷覆蓋率 (DC)。
    6.  **故障分類與失效率計算** (ME-QU欄): 根據前面的輸入，自動計算並將失效率分配到不同的故障類別（SPF, RF, MPF, Safe）。

- **合併儲存格的用途**
  - 合併儲存格主要用於創建層次化的標頭，提高可讀性。例如，`AG1:AG3` 標示了「SPF DC gets MPF (int.)」這一概念，其下的子欄位則進行具體評估。
  - 這種結構使得複雜的表格在視覺上更易於管理，將數百個欄位組織成有邏輯關聯的區塊。

- **資料的組織邏輯**
  - 每一橫列代表一個獨立的失效模式分析。
  - 資料從左側的元件基本資訊開始，向右逐步深入，進行安全分析、診斷覆蓋率評估，最終在最右側得出該失效模式的量化安全指標。
  - 這種從因到果的水平鋪陳方式，完全對應了 `Sheet 8: FlowChart` 中定義的 FMEDA 計算流程。

## 3. 圖片分析

根據提供的 JSON 資料，此「FMEDA」工作表本身**不包含任何圖片**。所有與方法論相關的關鍵流程圖和定義圖，均位於 `Process`、`Definitions` 和 `FlowChart` 等輔助工作表中。稽核員在審查此表時，會頻繁參照那些工作表中的圖片來驗證此處公式的正確性。

## 4. 公式深度分析

此工作表包含高達 **60,921** 個公式，是整個 FMEDA 報告的計算核心。雖然公式數量龐大，但其模式具有高度的重複性，主要可歸納為以下幾類獨特的模式。

| 公式模式 (以代表性儲存格為例) | 物理意義 / 計算目的 | 引用的儲存格/Sheet | 對應的 ISO 26262 / IEC TR 62380 條款 |
| :--- | :--- | :--- | :--- |
| `=SUM(I4:I1653)` | 計算所有失效模式分佈的總和，用於驗證分佈比例是否為 100%。 | `I4:I1653` (同一欄) | ISO 26262-5, Annex D: FMEDA 的基本 sanity check。 |
| `=SafetyGoalViolations!D3` | 從 `SafetyGoalViolations` 工作表引用安全目標的名稱。 | `SafetyGoalViolations!D3` | ISO 26262-3: Hazard analysis and risk assessment. |
| `=IF(SUM(CL2,CM2)=0,"Not calculated",1-CM2/CO2)` | 計算診斷覆蓋率 (DC)。如果總失效率為零，則不計算。 | `CL2`, `CM2`, `CO2` (同一工作表內的匯總值) | ISO 26262-5, 8.4.6: Diagnostic coverage. |
| `=IF(SUM(CR2,CT2,CV2)=0,"Not calculated",1-CT2/(CU2-CS2))` | 另一個 DC 計算的變體，結構類似但引用不同欄位。 | `CR2`, `CT2`, `CV2`, `CU2`, `CS2` | ISO 26262-5, 8.4.6: Diagnostic coverage. |

**公式的正確性驗證要點**：
1.  **分母確效**：稽核員會特別關注所有除法公式的分母，確保在分母為零或無效時（如 `CO2-CM2`），有適當的錯誤處理機制（如 `IF` 判斷式），以避免 `#DIV/0!` 錯誤。
2.  **跨表引用**：所有對 `SafetyGoalViolations` 和其他工作表的引用必須準確無誤，確保分析基於正確的假設。
3.  **邏輯一致性**：公式的計算邏輯必須嚴格遵循 `Sheet 8: FlowChart` 中定義的流程。例如，一個被歸類為 SPF 的故障，其失效率不應出現在 MPF 的計算中。
4.  **匯總範圍**：`SUM` 公式的範圍（如 `I4:I1653`）必須涵蓋所有有效的資料列，不能遺漏或包含標頭。

## 5. 稽核合規性分析

此「FMEDA」工作表是證明產品符合 ISO 26262 功能安全要求的核心證據之一，其內容直接對應標準的多個部分。

- **對應的 ISO 26262 條款**
  - **Part 5 (Product development at the hardware level)**: 整個工作表就是 Part 5 要求的硬體安全分析的具體實踐，特別是第 8 節「Hardware architectural metrics」和第 9 節「Evaluation of violation of the safety goal due to random hardware failures」。
  - **Part 9 (Automotive safety integrity level (ASIL)-oriented and safety-oriented analyses)**: 提供了相依失效分析 (DFA) 的輸入，因為此處評估了安全機制對特定失效模式的覆蓋情況。
  - **Part 11 (Guidelines on application of ISO 26262 to semiconductors)**: 對於半導體元件，此處的失效率輸入（λ）和失效模式分佈（FM distribution）應基於 Part 11 的指導原則或公認的半導體可靠性數據手冊（如 IEC TR 62380）。

- **滿足 ASIL B 等級的具體要求**
  - **SPFM (Single-Point Fault Metric)**: 對於 ASIL B，SPFM 必須 ≥ 90%。此工作表通過將故障分類為 SPF、RF 和 MPF，為計算 SPFM 提供了所有必要的失效率分量。
  - **LFM (Latent Fault Metric)**: 對於 ASIL B，LFM 必須 ≥ 60%。此工作表通過評估多點故障的診斷覆蓋率，計算出潛伏故障 (MPF,L) 的失效率，是 LFM 計算的基礎。
  - **PMHF (Probabilistic Metric for random Hardware Failures)**: 對於 ASIL B，PMHF 必須 < 100 FIT。此工作表計算出的 SPF 和 RF 失效率是 PMHF 計算的主要輸入。

- **稽核員可能會問的問題**
  1.  「請解釋 A-E 欄中的硬體元件層次劃分依據是什麼？它是否與您的硬體架構圖一致？」
  2.  「F 和 G 欄的失效率數據來源是什麼？您是如何根據 IEC TR 62380 或其他標準計算出這些值的？」
  3.  「針對失效模式 H4，您是如何判斷其在 SGV 01 (L欄) 下的分類為 'S' (Safe) 的？請提供您的技術判斷依據。」
  4.  「在 AS2 欄中，您為這個失效模式設定的診斷覆蓋率為 99%，請展示對應的安全機制設計文檔以及故障注入測試報告來佐證這個數值。」
  5.  「請走一遍從 G4 欄的失效率到 QU4 欄最終故障分類的完整計算路徑，並解釋每一步是如何對應 FlowChart 上的決策節點的。」

- **需要準備的佐證文件**
  - 硬體架構設計圖
  - 元件失效率計算報告 (基於 IEC TR 62380 或類似標準)
  - 安全概念 (Safety Concept) 文件
  - 軟硬體介面規格書 (HIS)
  - 故障注入測試計畫與報告
  - 安全機制設計與驗證文件

## 6. 稽核檢查清單

以下是稽核員在審查此「FMEDA」工作表時，可能會使用的檢查清單。

| 檢查項目 | Pass/Fail 判定標準 |
| :--- | :--- |
| **1. 元件與架構一致性** | **Pass**: FMEDA 中的元件列表與硬體架構圖完全對應。 **Fail**: 存在遺漏或不一致的元件。 |
| **2. 失效率數據來源** | **Pass**: 所有基礎失效率 (λ) 都有明確、可追溯的來源（如 IEC TR 62380 計算表或供應商數據）。 **Fail**: 失效率數據無法提供來源或計算過程。 |
| **3. 失效模式完整性** | **Pass**: 失效模式列表完整，涵蓋了元件所有可能的物理失效，並有合理的參考依據（如 IEC 62380, FMD-97 等）。 **Fail**: 失效模式列表不完整或分佈比例不合理。 |
| **4. 安全目標違反評估** | **Pass**: 對每個失效模式影響的評估都有清晰的技術理由，並與安全概念一致。 **Fail**: 評估理由模糊、矛盾或缺乏技術深度。 |
| **5. 診斷覆蓋率 (DC) 佐證** | **Pass**: 所有聲稱的 DC 值都有對應的安全機制設計和故障注入測試報告支持。 **Fail**: DC 值僅為估計值，缺乏客觀證據支持。 |
| **6. 公式與流程圖一致性** | **Pass**: 表格中的所有計算公式都嚴格遵循了 `FlowChart` 工作表中定義的計算邏輯。 **Fail**: 發現公式與標準流程不符，或存在計算錯誤。 |
| **7. 數據完整性與無錯誤** | **Pass**: 表格中沒有 `#DIV/0!`, `#REF!`, `#N/A` 等錯誤值，所有加總驗證（如 FM distribution）均通過。 **Fail**: 表格中存在計算錯誤或不一致的數據。 |
| **8. ASIL 指標計算正確性** | **Pass**: 基於此表數據計算出的 SPFM, LFM 和 PMHF 均符合目標 ASIL 等級的要求。 **Fail**: 計算出的指標未達標，或計算過程本身存在錯誤。 |

此份報告基於提供的 JSON 數據進行分析，旨在為 ISO 26262 功能安全稽核提供深度參考。
