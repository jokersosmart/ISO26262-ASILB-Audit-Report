# FMEDA 深度稽核分析報告：Sheet 17 - DiagnosisExternal

## 1. Sheet 概述

- **Sheet 名稱**: `DiagnosisExternal`
- **在 FMEDA 報告中的位置**: 第 17 個 sheet
- **維度**: 28 行 × 17 列 (A1:Q28)
- **角色與目的**: 此工作表的核心目的在於**記錄和管理所有應用於被分析硬體（在此情境下為 IC）之外的外部診斷機制**。在功能安全的脈絡下，單靠晶片內部的安全機制往往不足以達到所需的安全等級（ASIL）。因此，系統層級或由其他 ECU 執行的診斷程序，對於偵測和控制潛在故障至關重要。本表單即是集中管理這些外部診斷措施的資料庫，詳細說明每個機制的運作方式、偵測能力（診斷覆蓋率）、反應時間及故障反應行為，為整體 FMEDA 分析中的多點故障（Multiple-Point Faults）和潛伏故障（Latent Faults）分析提供關鍵輸入。

## 2. 內容結構分析

`DiagnosisExternal` 工作表的結構清晰，旨在條列化管理每一個外部診斷機制。

- **欄位標頭的含義與組織方式**: 標頭從 A2 到 H2，定義了描述一個完整診斷機制所需的核心屬性：
  - `ID Diagnosis`: 每個診斷機制的唯一識別碼（例如 DX001），便於在 FMEDA 主表中被引用。
  - `Short description of diagnostic procedure`: 對診斷程序的簡潔描述，讓人快速了解其功能。
  - `Detailed description of diagnostic procedure`: 詳細的技術說明，解釋診斷的具體步驟、原理和範圍。
  - `Detection and reaction time`: 定義了從故障發生到被此診斷機制偵測並觸發反應所需的總時間。這是計算故障處理時間間隔（Fault Handling Time Interval, FHTI）的關鍵參數。
  - `Reaction in case diagnostics detects a fail`: 描述當偵測到故障時，系統會採取的應對措施，例如進入安全狀態、警示駕駛員或執行降級操作。
  - `Covers transient faults`: 指出此診斷機制是否能有效偵測暫態故障（瞬時的、非永久性的故障）。這對評估系統的整體穩健性非常重要。
  - `Diagnostic Coverage estimated`: 預估的診斷覆蓋率（DC），以百分比表示。此數值直接影響潛伏故障度量（Latent Fault Metric, LFM）的計算，是 FMEDA 中的核心量化指標。
  - `Comments`: 提供額外的備註或說明。

- **合併儲存格的用途**: 儲存格 `A1:H1` 被合併並標示為 "IC external diagnosis"，作為整個工作表的總標題，清晰地揭示了此頁的用途。

- **資料的組織邏輯**: 資料以行為單位，每一行（從第 3 行開始）代表一個獨立的外部診斷機制。範例資料中的第一筆 `ID Diagnosis = "-"` 且描述為 `"No mechanism available"`，這是一個重要的基線設定，代表「無任何診斷機制」的情況，其診斷覆蓋率為 0%。後續的 `DX001`, `DX002` 等則應詳細填寫各自的診斷細節。

## 3. 圖片分析

此工作表中**沒有包含任何圖片**。所有與 FMEDA 方法論相關的參考圖片（如流程圖、定義圖）均位於其他工作表（如 Sheet 3, 6, 8）中。

## 4. 公式深度分析

此工作表中**沒有包含任何 Excel 公式**。本表的功能是作為一個純粹的資料輸入表，其核心價值在於提供結構化的資料，供 FMEDA 主表或其他計算表單中的公式引用。例如，此處定義的「Diagnostic Coverage estimated」值，將會被 FMEDA 主表中計算 `λ_MPF,det`（已偵測多點故障）和 `λ_MPF,pl`（潛在多點故障）的公式所使用。

## 5. 稽核合規性分析

此工作表雖然不含複雜計算，但在 ISO 26262 稽核中扮演著不可或缺的角色，直接關聯到安全機制的完整性與有效性。

- **對應的 ISO 26262 條款**:
  - **ISO 26262-5, Clause 8 (Hardware architectural metrics)**: 本表定義的診斷覆蓋率（DC）是計算潛伏故障度量（LFM）的直接輸入。LFM 是 Part 5 中針對高 ASIL 等級（B, C, D）的強制要求。
  - **ISO 26262-5, Annex D (Evaluation of diagnostic coverage)**: 附錄 D 提供了估算 DC 的指導原則與範例。稽核員會期望本表中的 DC 估計值是有依據的，最好能追溯到 Annex D 的方法論或更詳細的故障注入測試報告。
  - **ISO 26262-9, Clause 5 (Analysis of dependent failures)**: 外部診斷機制本身也可能存在相依故障。稽核員可能會探討這些外部機制與被診斷元件之間的獨立性。

- **滿足 ASIL B 等級的具體要求**: 
  對於 ASIL B，LFM 的目標值為 ≥ 60%。要達到此目標，單靠晶片內部診斷往往不足。`DiagnosisExternal` 中定義的強大外部診斷機制（例如，由主處理器執行的記憶體端對端 CRC 校驗、通訊匯流排的 E2E 保護等）是貢獻 DC、提升 LFM 數值的關鍵。若沒有此表中的有效機制，LFM 很可能無法達標。

- **稽核員可能會問的問題**:
  - "針對診斷機制 DX001，其宣稱的 90% DC 是如何得出的？請提供估算報告或故障注入測試的證據。"
  - "此處列出的『Detection and reaction time』是否考慮了最壞情況下的系統負載？這是否滿足安全目標定義的 FTTI 要求？"
  - "診斷機制 DX002 是由哪個硬體或軟體單元執行的？它與被診斷的 IC 是否存在共因失效（Common Cause Failures）的風險？"
  - "對於暫態故障的覆蓋，你們的依據是什麼？是否有數據支持『Partly』或『Yes』的結論？"

- **需要準備的佐證文件**:
  - **系統架構圖**: 清晰標示執行外部診斷的單元及其與被診斷 IC 的關係。
  - **軟體設計文件**: 詳細描述診斷演算法的實現細節。
  - **故障注入測試報告**: 證明診斷覆蓋率的實際測試結果。
  - **時序分析報告 (Timing Analysis)**: 驗證偵測與反應時間滿足 FTTI 要求。
  - **相依故障分析 (DFA) 報告**: 分析診斷機制與目標元件之間的獨立性。

## 6. 稽核檢查清單

以下是稽核員在審查 `DiagnosisExternal` 工作表時會使用的檢查清單：

| 檢查項目 | Pass/Fail 判定標準 |
| :--- | :--- |
| **完整性** | **Pass**: 所有應用於此 IC 的外部診斷機制均已列出，無遺漏。<br>**Fail**: 存在未被記錄的系統級診斷措施。 |
| **唯一識別碼** | **Pass**: 每個診斷機制都有唯一的 `ID Diagnosis`，且在 FMEDA 主表中有對應的引用。<br>**Fail**: ID 重複、缺失或與主表不一致。 |
| **描述清晰度** | **Pass**: `Short` 和 `Detailed` 描述清晰準確，足以讓第三方理解其工作原理。<br>**Fail**: 描述含糊不清，無法判斷其具體功能。 |
| **時間參數合理性** | **Pass**: `Detection and reaction time` 被明確定義，且有證據支持其滿足 FTTI。<br>**Fail**: 時間參數為空、不合理（過長），或缺乏分析支持。 |
| **故障反應明確** | **Pass**: `Reaction` 描述了具體的系統行為，且該行為能有效過渡到安全狀態。<br>**Fail**: 故障反應未定義或無效。 |
| **DC 估計依據** | **Pass**: `Diagnostic Coverage estimated` 的數值有合理的技術分析或測試報告作為支撐。<br>**Fail**: DC 值僅為拍腦袋的猜測，缺乏任何佐證。 |
| **暫態故障覆蓋** | **Pass**: 對於是否覆蓋暫態故障的聲明（Yes/No/Partly）有合理解釋。<br>**Fail**: 聲明與診斷機制的物理原理相悖。 |
| **與主表一致性** | **Pass**: 本表中定義的 DC 和 ID 與 FMEDA 主表中引用的數據完全一致。<br>**Fail**: 數據不一致，導致計算結果錯誤。 |
