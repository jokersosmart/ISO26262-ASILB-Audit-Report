## 1. Sheet 概述

本報告針對 FMEDA 工作簿中的第 26 個工作表「BlockTypeEval-IEC61508_opt1」進行深度稽核分析。此工作表是整個 FMEDA 分析的核心成果彙總之一，其重要性在於它從硬體元件類型（Block Type）的視角，對整個系統的失效率指標進行了整合與計算，並直接關聯到國際功能安全標準 IEC 61508 及 ISO 26262 的合規性要求。

| 項目 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `BlockTypeEval-IEC61508_opt1` |
| **在報告中的位置** | 第 26 個 Sheet |
| **維度** | 9 行 × 34 列 (A1:AH9) |
| **角色與目的** | 此工作表的主要目的是根據 IEC 61508 的計算假設，對不同類型的硬體元件（如數位電路、SRAM、ROM 等）進行失效率評估。它從更底層的 `BlockList` 和 `FMEDA` 工作表中提取原始數據，將其分類、加總，並計算出關鍵的硬體架構度量指標，如 **安全故障分數 (SFF)** 和 **每小時危險失效機率 (PFH)**。其產出是判斷系統硬體設計是否滿足特定安全完整性等級 (SIL/ASIL) 的關鍵依據。 |

## 2. 內容結構分析

此工作表的結構設計旨在提供一個清晰、全面的硬體失效率概覽。其組織邏輯圍繞著「永久性失效 (permanent)」與「暫態性失效 (transient)」這兩個維度展開，並對每種失效模式進行了詳細的分類計算。

- **欄位標頭 (第一列)**: 共有 34 個欄位，詳細定義了從「初始總失效率」到最終「SFF」和「PFH」的完整計算鏈路。欄位可以分為以下幾個邏輯群組：
  1.  **初始失效率 (B-D)**: 從 `BlockList` 引入的該元件類型的總失效率。
  2.  **安全相關失效率 (E-G)**: 屬於安全功能範疇內的失效率。
  3.  **非安全相關失效率 (H-J)**: 與安全功能無關的失效率。
  4.  **安全相關失效率分佈 (K-M)**: 各元件類型在總安全相關失效率中的佔比。
  5.  **安全失效 (Safe Fault) (N-P)**: 不會導致安全目標被違反的失效率。
  6.  **危險失效 (Dangerous Fault) (Q-S)**: 有潛力導致安全目標被違反的失效率。
  7.  **已偵測到的危險失效 (Dangerous Detected) (T-V)**: 可被安全機制偵測到的危險失效率。
  8.  **未偵測到的危險失效 (Dangerous Undetected) (W-Y)**: 無法被安全機制偵測到的危險失效率，這是計算 PFH 的關鍵輸入。
  9.  **未偵測危險失效率分佈 (Z-AB)**: 各元件類型在總未偵測危險失效率中的佔比。
  10. **安全指標計算 (AC-AH)**: 計算 SFF 和 PFH 的最終結果。

- **資料組織邏輯 (第二列及以下)**: 以「元件類型」作為基本單位進行組織。每一行（從第 2 行開始）代表一類特定的半導體元件（如 `Si MOS: Digital circuits`）。所有計算都圍繞著該元件類型在整個系統中的表現進行，數據主要引用自 `BlockList`、`FMEDA`、`BlockTypeEvaluation` 和 `SafetyGoalViolations` 等工作表，體現了 FMEDA 分析中數據的層層關聯與追溯性。

## 5. 稽核合規性分析

此工作表是證明硬體設計滿足 ISO 26262 安全目標的核心證據之一。基於其內容，稽核風險被評估為 **高 (High)**，因為任何計算錯誤都將直接影響最終的 ASIL 等級宣告。

- **對應的 ISO 26262 條款**:
  - **Part 5, Clause 8**: *Hardware architectural metrics*。此工作表直接實作了 SFF 和 PFH 的計算，是滿足該條款的核心。
  - **Part 5, Annex E**: *Guideline on carrying out failure mode and effects analysis (FMEA)*。本表是 FMEDA 分析方法的具體產出。
  - **Part 9, Clause 5**: *ASIL-dependent and safety-oriented analyses*。提供了基於 ASIL 等級的量化分析結果。

- **滿足 ASIL B 等級的具體要求**:
  - **SFF**: 計算結果（AC2:AE2）必須顯示 SFF ≥ 90%。
  - **PFH**: 計算結果（AF2:AH2）必須顯示 PFH < 100 FIT。
  稽核員會嚴格審查達到這些目標值所依賴的數據和假設。

- **稽核員可能會問的問題**:
  1.  「請解釋 `BlockTypeEvaluation` 工作表中 PFH 基值的計算邏輯。它基於哪些假設？」
  2.  「在計算『已偵測到的危險失效』(T2) 時，你們所使用的診斷覆蓋率 (DC) 數據從何而來？請提供故障注入測試報告或其他佐證。」
  3.  「對於『暫態失效 (transient failure)』的失效率數據（C列），其來源是什麼？你們如何驗證這些數據的準確性？」
  4.  「請展示從 `BlockList` 到本表的數據追溯路徑，並證明數據在傳遞過程中沒有失真或遺漏。」
  5.  「`SafetyGoalViolations!$F$37` 這個參數代表什麼？其數值的設定依據是什麼文件或分析？」

- **需要準備的佐證文件**:
  - 硬體元件的失效率數據手冊（如 SN 29500, IEC 62380）。
  - 故障注入（Fault Injection）測試計畫與報告。
  - 軟體安全機制（用於偵測硬體故障）的設計文檔與驗證報告。
  - 系統架構設計文檔，說明各元件與安全功能的對應關係。
