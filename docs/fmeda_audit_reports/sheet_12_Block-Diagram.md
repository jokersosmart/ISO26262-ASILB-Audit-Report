# FMEDA 深度稽核分析報告：Sheet 12 - Block-Diagram

---

## 1. Sheet 概述

本章節提供 Block-Diagram 工作表的概觀，說明其在整個 FMEDA 分析中的基礎角色與目的。

| 項目 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `Block-Diagram` |
| **在報告中的位置** | 第 12 個工作表 |
| **維度** | 2 行 × 4 列 |
| **角色與目的** | 此工作表的目的在於提供一個高層次的系統架構視圖，通常以方塊圖的形式呈現。在 FMEDA 分析流程中，它扮演著**視覺化上下文**的角色，幫助所有利害關係人（包括設計、測試、安全團隊及稽核員）快速理解系統的邊界、主要硬體元件（Hardware Elements）及其相互關係。雖然此特定工作表中未直接繪製圖表，但其標題明確指出此處應為系統方塊圖的所在地。這個方塊圖是後續進行失效模式分析的基礎，因為它定義了「什麼」是系統的一部分，以及各部分之間如何互動。 |

## 2. 內容結構分析

本章節分析工作表的內部結構，包括儲存格的組織方式與資料邏輯。

- **欄位標頭**：此工作表僅包含一個標頭，位於儲存格 `B2`，內容為「**Block Diagram**」。該儲存格的字體被加粗，並使用深藍色（`FF1F497D`）進行填充，表明這是一個標題儲存格，用於定義整個工作表的主題。

- **合併儲存格**：根據儲存格 `B2` 的標題性質與常見的 Excel 實踐，此儲存格很可能橫跨 `B2:D2`，以在視覺上置中標題。然而，檔案元數據中 `merged_cells` 陣列為空，這可能意味著合併是透過欄寬調整而非實際的儲存格合併來達成視覺效果，或是元數據提取上的小疏漏。無論如何，其功能是作為一個跨越多欄的標題。

- **資料組織邏輯**：此工作表不包含任何表格化的數據或資料樣本。其唯一的內容就是標題。這表明它是一個**佔位符 (Placeholder)** 或**展示性 (Presentational)** 的工作表。其組織邏輯極為簡單：宣告此處應用於展示系統的總體架構圖。在實際的專案交付中，這裡通常會嵌入一張圖片或使用 Excel 的繪圖工具繪製系統方塊圖。

## 5. 稽核合規性分析

儘管此工作表本身不包含計算或詳細的失效模式，但它在稽核中扮演著關鍵的**起點**和**上下文提供者**角色。稽核員會利用此處的（預期中的）方塊圖來框架整個 FMEDA 的審查。

- **對應的 ISO 26262 條款**：
  - **ISO 26262-5:2018, Clause 7 (Hardware safety requirements)**：此處的系統架構圖是定義硬體設計的起點，所有硬體安全需求的分配都基於此架構。
  - **ISO 26262-8:2018, Clause 6 (System design)**：方塊圖是系統設計規格的一部分，必須清晰地描述元件及其介面。
  - **ISO 26262-4:2018, Clause 6 (System-level design)**：在系統層級，此圖應能展示系統元件如何共同作用以實現所需功能。

- **滿足 ASIL B 等級的要求**：
  對於 ASIL B 等級，系統架構的清晰度至關重要。雖然標準未對圖表的格式有強制要求，但要求設計必須是**可理解的 (understandable)** 和**可追溯的 (traceable)**。一個好的方塊圖能直接滿足此要求，它顯示了：
  1.  系統邊界內的所有硬體元件。
  2.  元件之間的依賴關係和通訊路徑。
  3.  安全相關元件的識別。

- **稽核員可能會問的問題**：
  1.  「這個方塊圖是否代表了最新版本的系統架構？請提供對應的設計文件版本號。」
  2.  「圖中標示的所有硬體元件，是否都已包含在 FMEDA 主分析表中？」
  3.  「系統的外部介面（例如電源、感測器輸入、致動器輸出）在哪裡？它們是否被正確地識別為潛在的故障來源？」
  4.  「此架構圖如何對應到硬體線路圖（Schematics）？請展示兩者之間的一致性。」
  5.  「是否有任何在此圖中未顯示，但卻屬於分析範圍的硬體元件？」

- **需要準備的佐證文件**：
  1.  最新版本的系統架構設計文件。
  2.  硬體規格書（Hardware Specification）。
  3.  硬體線路圖（Schematics）。
  4.  元件清單（Bill of Materials, BOM）。
  5.  版本控制歷史，證明此架構圖與其他設計文件同步更新。

## 6. 稽核檢查清單

稽核員將根據以下清單，評估此工作表及其代表的系統架構是否滿足功能安全的要求。

| 檢查項目 | Pass/Fail 判定標準 |
| :--- | :--- |
| **1. 架構圖存在性** | **Pass**: 工作表包含一個清晰、易於理解的系統方塊圖。<br>**Fail**: 工作表為空或僅有標題，沒有實際的圖表。 |
| **2. 元件完整性** | **Pass**: 圖中包含了所有在 FMEDA 中分析的硬體元件，無遺漏。<br>**Fail**: FMEDA 中分析的元件在圖中缺失，或圖中有 FMEDA 未分析的元件。 |
| **3. 介面清晰度** | **Pass**: 元件之間的介面（資料流、控制信號、電源）被清晰標示。<br>**Fail**: 介面模糊不清或未標示，無法判斷元件間的依賴關係。 |
| **4. 邊界定義** | **Pass**: 系統的分析邊界（System Boundary）被明確定義。<br>**Fail**: 無法從圖中判斷哪些屬於系統內部，哪些屬於外部。 |
| **5. 文件一致性** | **Pass**: 方塊圖與系統設計文件、線路圖保持一致。<br>**Fail**: 圖表與其他設計文件存在矛盾或版本不匹配。 |
| **6. 可讀性** | **Pass**: 圖表佈局合理，標示清晰，無需額外解釋即可理解。<br>**Fail**: 圖表混亂、標示不清或使用過多難懂的縮寫。 |

### 稽核風險評估

基於當前工作表的狀態（僅有標題），此工作表的**稽核風險等級為「高」**。原因是缺乏核心內容（實際的方塊圖），導致無法驗證系統架構的正確性與完整性，這是整個 FMEDA 分析的基礎。在實際稽核中，這將是一個重大的發現（Major Finding），需要立即補正。
