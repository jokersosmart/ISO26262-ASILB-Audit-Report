# ISO 26262 深度稽核分析報告：BlockEvaluation

## 1. Sheet 概述

- **Sheet 名稱：** `BlockEvaluation`
- **在 FMEDA 報告中的位置：** 第 22 個 sheet
- **維度：** 211 行 × 63 列 (A1:BK211)
- **角色與目的：**

此工作表在 FMEDA 分析流程中扮演著**承上啟下**的關鍵角色，其核心目的是對整個硬體系統的各個功能區塊（Block）進行故障率的彙總、分類與評估。它從更底層的工作表（如 `BlockList` 和 `FMEDA`）提取基礎故障數據，並根據 ISO 26262 的硬體架構度量指標（Hardware Architectural Metrics）要求，計算出單點故障（SPF）、殘餘故障（RF）和多點故障（MPF）等關鍵指標的故障率貢獻。

簡單來說，`BlockEvaluation` 將分散在數千個元件級別的故障分析結果，提升到系統架構的「區塊」層級進行審視，為最終計算整體的單點故障度量（SPFM）、潛在故障度量（LFM）以及隨機硬體失效機率度量（PMHF）提供清晰、結構化的數據基礎。此工作表是驗證硬體設計是否滿足目標 ASIL 等級所要求的架構容錯能力的關鍵證據。

## 2. 內容結構分析

- **欄位標頭的含義與組織方式：**

第一行標頭欄位定義了整個分析的框架，從總體故障率開始，逐步分解到與安全相關的各個類別。欄位可以分為幾個邏輯群組：

  1.  **總體故障率 (D-F):** 區分永久性（Permanent）和暫態（Transient）故障，計算初始總故障率。
  2.  **安全相關性分類 (G-L):** 將總故障率分解為「安全相關」與「非安全相關」兩部分。
  3.  **安全相關故障分佈 (M-O):** 計算各區塊的安全相關故障率在總體中所佔的比例。
  4.  **安全故障分類 (P-U):** 進一步將安全相關故障分為「安全故障（Safe）」和「不安全故障（Not Safe）」。
  5.  **單點/多點故障分析 (V-AM):** 此為核心區域。將不安全故障細分為直接違反安全目標的「單點故障（SPF）」和不會直接違反的「多點故障（MPF）」。同時也計算了「殘餘故障（RF）」。
  6.  **診斷覆蓋影響分析 (AN-AV):** 分析安全機制的診斷覆蓋（DC）如何影響故障分類，例如部分被檢測到的 SPF 可能會轉化為 MPF 或安全故障。

- **合併儲存格的用途：**

  - `B205:C211`: 此區域用於顯示一個註解或圖例，解釋圖表中使用的顏色或符號所代表的意義，這有助於快速理解圖表內容。

- **資料的組織邏輯：**

資料以「一個硬體區塊佔據一行」的方式組織。每一行從 `BlockList` 工作表獲取區塊的基本資訊，然後通過一系列公式，從 `FMEDA` 主工作表中抓取並加總對應區塊下所有元件的各類故障率。這種結構使得每一行都是對一個獨立硬體區塊的完整故障特性評估，邏輯清晰，易於追蹤和審查。

## 3. 圖片分析

此工作表中包含一張圖片，位於 `B212` 儲存格，是對整個 `BlockEvaluation` 工作表數據的圖形化總結。

- **圖片內容描述：**
該圖片是一個堆疊條形圖（Stacked Bar Chart），標題為「Failure Rate Evaluation per Block [FIT]」。橫軸代表不同的故障率類別（永久性 vs. 暫態），縱軸是故障率的數值（單位：FIT）。圖表中的每一個彩色區段代表一種特定的故障分類，例如：
  - `λ_SPF,nd` (SPF, not detected)
  - `λ_SPF,d` (SPF, detected)
  - `λ_RF` (Residual Fault)
  - `λ_MPF,nd` (MPF, not detected)
  - `λ_MPF,d` (MPF, detected)
  - `λ_safe` (Safe Fault)

- **代表意義：**
此圖表將複雜的表格數據轉化為直觀的視覺呈現。它清晰地展示了系統總故障率的構成，讓稽核員和工程師可以一目了然地看出：
  - 哪一類故障（SPF、MPF、Safe）佔主導地位。
  - 安全機制的有效性如何（被檢測到的故障 `d` vs. 未被檢測到的 `nd` 的比例）。
  - 永久性故障和暫態故障對系統風險的貢獻分佈。

- **與 ISO 26262 的對應關係：**
此圖表直接對應 **ISO 26262-5:2018, Annex D 和 Annex E** 中關於硬體架構度量計算的指導。它將標準中定義的 `λ_SPF`, `λ_RF`, `λ_MPF,L`, `λ_S` 等故障率分量進行了視覺化彙總，是計算 SPFM 和 LFM 指標的直接輸入。

- **對稽核的重要性：**
這張圖是**極其重要**的稽核證據。稽核員可以通過此圖快速評估設計的健全性。例如，如果代表「未被檢測到的單點故障」（`λ_SPF,nd`）的區塊佔比過高，將會是一個重大的警示信號（Red Flag），表明系統可能無法滿足目標 ASIL 等級對單點故障的嚴格要求。

## 4. 公式深度分析

此工作表雖然沒有在 `formulas` 欄位中列出公式，但其資料行（第 2 行起）大量使用了公式進行計算。以下是幾個關鍵且具有代表性的公式模式分析：

1.  **`=D2+E2` (F列)**
    - **計算目的：** 將永久性故障率和暫態故障率相加，得到總故障率。
    - **物理意義：** 體現了總體風險是兩類故障模式的總和。
    - **對應條款：** ISO 26262-5, Clause 8.4.4 強調需要考慮不同原因導致的隨機硬體失效。
    - **正確性驗證：** 檢查 D 列和 E 列的數據來源是否可靠，加法本身很難出錯。

2.  **`=SUMIFS(FMEDA!$HB$4:$HB$1653,FMEDA!$B$4:$B$1653,B2,FMEDA!$HJ$4:$HJ$1653,"N")` (P列)**
    - **計算目的：** 從 `FMEDA` 主工作表中，條件式地加總所有屬於當前區塊（`B2`）且被分類為「安全故障（Safe, 'N'）」的永久性故障率。
    - **物理意義：** 彙總指定硬體區塊內所有不會違反安全目標的元件級故障率。
    - **引用：** 引用了 `FMEDA` 工作表的 `B`, `HB`, `HJ` 列。
    - **對應條款：** ISO 26262-5, Annex D.2.2 中對安全故障 `λ_S` 的定義。
    - **正確性驗證：** 必須驗證 `FMEDA` 表中 `HJ` 列對「安全故障」的分類標準是否正確，以及 `SUMIFS` 的範圍是否覆蓋了所有元件。

3.  **`=IF(SUM($AE$2:$AE$201)=0,"-",AE2/SUM($AE$2:$AE$201))` (AH列)**
    - **計算目的：** 計算當前區塊的 SPF & RF 永久性故障率（`AE2`）佔所有區塊總和的比例。
    - **物理意義：** 評估單個區塊對系統總體單點/殘餘風險的貢獻度，有助於識別關鍵風險模組。
    - **引用：** 引用了本工作表的 `AE` 列。
    - **對應條款：** 雖然沒有直接對應的計算公式，但這符合 ISO 26262 強調的風險分解與貢獻度分析思想。
    - **正確性驗證：** 檢查分母的加總範圍（`$AE$2:$AE$201`）是否正確覆蓋了所有活動的硬體區塊。

4.  **`=V2-Y2-AB2` (AN列)**
    - **計算目的：** 從總的 SPF 故障率（`V2`）中減去未被安全機制覆蓋的 SPF（`Y2`）和被覆蓋的 RF（`AB2`），得到被檢測到的 SPF 故障率。
    - **物理意義：** 此公式體現了故障在被安全機制檢測後的重新歸類邏輯，是計算診斷覆蓋率影響的核心步驟。
    - **對應條款：** 緊密關聯 ISO 26262-5, Annex E 中關於診斷覆蓋率如何影響故障分類的描述。
    - **正確性驗證：** 驗證 V, Y, AB 各列的計算邏輯是否符合 ISO 26262 的定義，這是非常容易出錯的環節。

## 5. 稽核合規性分析

- **對應的 ISO 26262 條款：**
  - **ISO 26262-5:2018 (硬體層面):**
    - **Clause 8:** 評估硬體架構度量指標（SPFM, LFM）。此工作表是計算這些指標的基礎。
    - **Clause 9:** 評估因隨機硬體失效導致違反安全目標的機率（PMHF）。此工作表計算的各類故障率是 PMHF 計算的輸入。
    - **Annex D & E:** 提供了計算 SPFM 和 LFM 的詳細方法，本表的公式和結構正是這些附件的具體實踐。
  - **ISO 26262-9:2018 (ASIL 導向和安全導向分析):** Clause 6 的依賴性失效分析（DFA）也可能參考此處的區塊級故障數據。

- **滿足 ASIL B 等級的具體要求：**
  對於 ASIL B，ISO 26262 要求：
  - **SPFM ≥ 90%**
  - **LFM ≥ 60%**
  此工作表的最終輸出（特別是 `λ_SPF`, `λ_RF`, `λ_MPF,L` 的總和）將被用於驗證是否達到這些目標值。稽核員會嚴格審查此表的數據，以確認指標計算的正確性。

- **稽核員可能會問的問題：**
  1.  「請解釋 `BlockEvaluation` 表中『SPF』、『RF』和『MPF』的區分標準是什麼？它與 `FMEDA` 主表中的分類邏輯如何保持一致？」
  2.  「在公式 P2 中，你們是如何確保 `FMEDA` 表中 `HJ` 列的『N』分類是準確無誤的？請提供分類的準則文件。」
  3.  「圖表中『λ_SPF,nd』（未檢測到的單點故障）的佔比較高，請解釋原因，並說明你們將採取何種措施來降低這部分風險。」
  4.  「請展示從這個表的數據到最終 SPFM 和 LFM 指標的完整計算過程。」

- **需要準備的佐證文件：**
  1.  **硬體架構設計文檔：** 用於解釋表中各個「Block」的劃分依據。
  2.  **FMEDA 分析報告：** 完整的 FMEDA 主表，用於追溯本表數據的來源。
  3.  **故障分類準則：** 詳細定義如何區分安全故障、SPF、MPF 的內部文件。
  4.  **安全機制設計文檔：** 解釋各診斷措施的原理和預期診斷覆蓋率（DC）。

## 6. 稽核檢查清單

| # | 檢查項目 | Pass/Fail 判定標準 |
|---|---|---|
| 1 | **數據一致性** | **Pass:** `BlockEvaluation` 表中各區塊的總故障率與 `BlockList` 及 `FMEDA` 表中的數據完全一致。<br>**Fail:** 存在任何數據差異或無法追溯的數據。 |
| 2 | **公式正確性** | **Pass:** 所有關鍵公式（如 `SUMIFS`, 分佈比例, 故障重分類）的邏輯與 ISO 26262-5 附件 D/E 的要求一致。<br>**Fail:** 公式邏輯錯誤，或引用範圍不完整。 |
| 3 | **故障分類合理性** | **Pass:** SPF, RF, MPF 的分類有明確、合理的準則支持，並在整個分析中保持一致。<br>**Fail:** 分類標準模糊、不一致或不符合標準定義。 |
| 4 | **圖表呈現清晰度** | **Pass:** 圖表清晰地展示了各類故障率的構成，圖例明確，能夠支持快速的風險評估。<br>**Fail:** 圖表混亂、難以解讀，或與表格數據不符。 |
| 5 | **與 ASIL 目標的關聯** | **Pass:** 能夠清晰地展示此表的輸出如何被用於計算 SPFM 和 LFM，並能證明計算結果滿足 ASIL B 的要求。<br>**Fail:** 無法建立與 ASIL 指標的明確聯繫，或計算結果未達標。 |
| 6 | **文件可追溯性** | **Pass:** 所有數據和結論都有對應的設計文檔、分析報告或準則文件作為支撐。<br>**Fail:** 任何關鍵假設或數據缺乏佐證文件。 |
