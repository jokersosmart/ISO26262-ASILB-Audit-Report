# FMEDA 深度稽核分析報告：Sheet 3 - Process

---

## 1. Sheet 概述

本章節提供 `Process` 工作表的概觀，說明其在整個 FMEDA 分析中的基礎性角色。

| 屬性 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `Process` |
| **在 FMEDA 報告中的位置** | 第 3 個工作表 |
| **維度** | 97 行 × 2 列 (A1:B97) |

此 `Process` 工作表是整個 FMEDA 模板的**方法論核心與使用指南**。它並不包含實際的計算資料，而是扮演一個資訊中心的角色，其主要目的有二：

1.  **指導使用者**：透過 A、B 兩欄的問答形式，詳細說明了模板中每一個工作表（如 `Cover`, `References`, `FMEDA` 等）的用途與填寫方式，確保使用者能夠正確地運用此模板進行分析。
2.  **定義核心流程**：此工作表內嵌了一張關鍵的流程圖，該圖視覺化了 ISO 26262 標準中定義的硬體元件失效模式分類方法。這不僅是理論指導，更是後續 `FMEDA` 主表中所有量化計算的根本依據。

對於稽核員而言，此工作表是理解該 FMEDA 報告是否基於一個合規、標準化流程的起點。它確立了整個分析的「遊戲規則」。

## 2. 內容結構分析

`Process` 工作表的結構設計以清晰傳達資訊為目的，其組織邏輯如下：

- **主要標題**：儲存格 `A1:B1` 進行了合併，以置中顯示整個工作表的主題：「FMEDA Process / How to fill this template」，明確揭示其雙重功用——流程定義與模板教學。

- **欄位標頭**：`A2` 和 `B2` 分別定義了「Worksheet」和「Description」，建立了一個直觀的鍵值對（Key-Value Pair）結構。`A` 欄列出模板中的各個工作表名稱，`B` 欄則提供對應的詳細描述。

- **資料組織邏輯**：從第 3 行開始，每一行都代表對一個特定工作表的解釋。例如，第 5 行解釋了 `Process` 工作表自身的功能，第 8 行解釋了 `FMEDA` 主表的功能。這種逐行解釋的方式，使得使用者可以像查閱字典一樣快速找到所需資訊。

- **合併儲存格的用途**：除了標題 `A1:B1` 外，內文中也出現了如 `A15:A16`、`A19:A21` 等縱向合併的儲存格。這些合併的主要目的是為了在視覺上將屬於同一大段落的文字（例如對 `FMEDA` 主表的詳細描述）歸為一組，使其在 `B` 欄跨越多行的長篇描述能對應到 `A` 欄的單一項目，增強了可讀性。

## 3. 圖片分析

此工作表包含一張至關重要的流程圖，是整個 FMEDA 方法論的核心。

- **圖片位置**：位於第 33 行下方，佔據了工作表的大部分視覺空間。
- **圖片檔案**：`sheet_3_Process_img_0.png`

### 3.1. 圖片內容描述

該圖片是一張詳細的決策流程圖，標題為「**Failure mode classification**」。它從一個硬體元件的總失效率（λ）出發，透過一系列二元判斷問題，最終將該元件的各種失效模式（Failure Modes）歸類到五個不同的「失效率桶」中。這五個桶分別是：

1.  `λ_SPF` (Single-Point Fault Rate)
2.  `λ_RF` (Residual Fault Rate)
3.  `λ_MPF,L` (Latent Multiple-Point Fault Rate)
4.  `λ_MPF,DP` (Detected/Perceived Multiple-Point Fault Rate)
5.  `λ_S` (Safe Fault Rate)

### 3.2. 代表意義

此流程圖定義了**硬體架構度量（Hardware Architectural Metrics）**計算的基礎。它回答了一個核心問題：「當一個硬體元件發生失效時，我們應如何根據其對安全目標的影響、以及有無安全機制覆蓋，來判定其風險等級？」

- **單點故障 (SPF)** 和 **殘餘故障 (RF)** 是最危險的，因為它們能直接或在缺少安全機制覆蓋時違反安全目標。
- **多點潛伏故障 (MPF,L)** 雖然本身不直接違反安全目標，但它可能與其他故障組合導致危險，且無法被駕駛員或診斷機制察覺，是隱藏的風險。
- **安全故障 (S)** 則是不會導致任何安全目標被違反的故障。

### 3.3. 與 ISO 26262 的對應關係

這張流程圖是 **ISO 26262-5:2018 中圖 5 (Figure 5) — *Hardware element failure mode classification*** 的一個忠實且詳細的實作版本。ISO 26262 標準提供了此分類的原則性框架，而這張圖將其轉化為一個可執行的、按部就班的決策過程。圖中的每一個判斷節點（例如「Is the HW element safety-related?」、「Potential to violate the safety goal?」）都直接源自於標準的要求。

### 3.4. 對稽核的重要性

對於稽核員來說，這張圖是**稽核的藍圖**。它的重要性體現在：

1.  **方法論合規性**：稽核員會首先確認開發團隊是否理解並遵循此流程。這是判斷整個 FMEDA 分析是否有效的基礎。
2.  **可追溯性**：在稽核 `FMEDA` 主表時，稽核員會隨機抽取幾個硬體元件的失效模式，並反向追溯，檢查其分類是否嚴格遵循了此流程圖的每一個決策步驟。例如，如果一個失效模式被歸類為 `λ_RF`，稽核員會驗證它是否確實「有潛力違反安全目標」，且其對應的「安全機制覆蓋率不足」。
3.  **工具驗證**：由於此流程圖被嵌入在官方模板中，它也證明了所使用的分析工具（即此 Excel 模板）本身的方法論是符合標準的。

## 4. 公式深度分析

此 `Process` 工作表本身**不包含任何計算公式**。其 `formula_count` 為 0。它的角色是為其他工作表（特別是 `FMEDA` 主表）中的公式提供理論依據和計算邏輯，而非執行計算。

## 5. 稽核合規性分析

此工作表雖然不含數據，但其內容與 ISO 26262 的多個部分緊密相關，是稽核的重點之一。

- **對應的 ISO 26262 條款**：
    - **ISO 26262-5:2018, Clause 8 (Evaluation of the hardware architectural metrics)**：此工作表的流程圖是計算 SPFM 和 LFM 兩個核心指標的基礎。該條款要求對硬體架構進行量化評估，而流程圖正是實現這一量化評估的第一步。
    - **ISO 26262-5:2018, Annex E (Classification of failures)**：該附錄提供了關於失效分類的詳細指導，流程圖的內容是該附錄的具體化。
    - **ISO 26262-9:2018, Clause 5 (ASIL-dependent requirements)**：雖然是間接關聯，但正確的失效分類是判斷 ASIL 等級是否達成的前提。如果分類錯誤，所有後續的 ASIL 等級評估都將是無效的。

- **滿足 ASIL B 等級的具體要求**：
    對於 ASIL B，ISO 26262-5 為 SPFM（單點故障度量）和 LFM（潛在故障度量）設定了目標值（SPFM ≥ 90%, LFM ≥ 60%）。`Process` 工作表中的流程圖是計算這兩個指標的起點。要達到 ASIL B，就必須嚴格遵循此流程對每一個失效模式進行分類，任何偏離都可能導致指標計算錯誤，從而無法證明達到 ASIL B 的要求。

- **稽核員可能會問的問題**：
    1.  「請解釋一下你們團隊是如何理解這張失效分類流程圖的？請以 `FMEDA` 表中的某個具體元件為例，帶我走一遍分類過程。」
    2.  「在流程圖的判斷節點 'd'（是否有潛力違反安全目標），你們的判斷標準是什麼？是否有文件化的指引？」
    3.  「對於被歸類為『安全故障』（Safe Fault）的失效模式，你們如何確保它們在任何情況下都不會導致安全目標被違反？」
    4.  「此流程圖與 ISO 26262-5:2018 圖 5 之間是否存在差異？如果有，請解釋原因。」

- **需要準備的佐證文件**：
    1.  **FMEDA 分析指南**：一份內部文件，詳細說明團隊如何應用此流程圖，並定義每個決策節點的具體判斷準則。
    2.  **安全目標定義文檔 (Safety Goals)**：用於在流程圖節點 'd' 進行判斷，證明某個失效模式是否會違反安全目標。
    3.  **系統/硬體架構設計文檔**：用於識別安全機制（節點 'e'），並評估其有效性。
    4.  **團隊培訓記錄**：證明負責執行 FMEDA 的工程師都接受過關於此流程圖和 ISO 26262 的充分培訓。

**稽核風險等級評估**: **低 (Low)**

此工作表為資訊展示和方法論定義，本身不涉及計算，出錯的可能性較低。然而，由於其內容是整個 FMEDA 分析的基礎，任何對其內容的**誤解**都將導致後續所有工作表出現系統性風險。因此，雖然工作表本身的風險為「低」，但其引發的潛在風險為「高」。稽核的重點在於驗證「人的理解」而非「表的數據」。
