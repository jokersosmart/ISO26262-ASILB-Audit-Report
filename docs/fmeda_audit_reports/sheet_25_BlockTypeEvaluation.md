# FMEDA 深度稽核分析報告：BlockTypeEvaluation

## 1. Sheet 概述

- **Sheet 名稱：** `BlockTypeEvaluation`
- **在 FMEDA 報告中的位置：** 第 25 個 sheet
- **維度：** A1:BI17 (17 行 × 61 列)
- **角色與目的：** 此工作表在 FMEDA 分析流程中扮演著**數據匯總與分類統計**的關鍵角色。其主要目的在於將來自 `BlockList` 和 `FMEDA` 主工作表的詳細故障數據，按照不同的「區塊/組件類型（Block / Component Type）」進行歸納與聚合。這使得分析團隊與稽核員能夠從一個更高維度的視角，審視不同類型組件（如 SRAM、ROM、邏輯電路等）的整體故障率貢獻、安全相關性以及在單點故障（SPF）、殘餘故障（RF）和多點故障（MPF）中的分佈情況。此表是驗證硬體架構度量（Hardware Architectural Metrics）如 SPFM 和 LFM 計算正確性的重要樞紐。

## 2. 內容結構分析

此工作表的結構設計旨在對故障率進行多維度的切割與檢視。

- **欄位標頭的含義與組織方式：**
  - **A 欄（Block / Component Type）：** 定義了組件的分類，例如 `Si MOS: High speed SRAM, FIFO`。這是數據聚合的基礎。
  - **B-D 欄（Initial / Total Failure Rate）：** 分別記錄了永久（permanent）、暫態（transient）以及兩者總和的初始總故障率，單位為 FIT（Failure in Time）。這些數據通常直接引用自 `BlockList` 工作表。
  - **E-J 欄（Safety related / Not safety related Failure Rate）：** 將總故障率切割為「安全相關」與「非安全相關」兩部分，並同樣區分永久與暫態故障。這是判斷故障是否影響安全目標（Safety Goal）的第一層分類。
  - **K-M 欄（Safety related Failure Rate Distribution）：** 計算各組件類型的安全相關故障率佔總安全相關故障率的百分比分佈，用於識別關鍵組件。
  - **N-S 欄（Failure Rate (Safe) / (Not Safe)）：** 進一步將安全相關故障分為「安全故障（Safe）」和「不安全故障（Not Safe）」。此處的「不安全故障」是指所有可能違反安全目標的故障，包括 SPF、RF 和 MPF。
  - **T-AH 欄（SPF & RF Analysis）：** 此區塊是核心，專注於分析「直接違反安全目標的潛在故障」，即單點故障（SPF）和殘餘故障（RF）。它詳細列出了 SPF 和 RF 的永久、暫態故障率，並計算其分佈。
  - **AI-AW 欄（MPF Analysis）：** 此區塊專注於分析「不會直接違反安全目標的故障」，即多點故障（MPF）。它區分了被安全機制檢測到的 MPF、未被檢測到的 MPF（即 Latent Fault），以及因安全機制失效而轉化為 MPF 的故障。

- **合併儲存格的用途：**
  - `A11:B17` 的合併儲存格在此 JSON 數據中未明確其內容，但通常在此類報告中，合併儲存格可能用於註釋、說明、或放置圖例，解釋特定計算的假設或數據來源。

- **資料的組織邏輯：**
  - **橫向邏輯：** 每一行代表一個特定的組件類型。從左到右，數據的顆粒度越來越細。始於該組件的總故障率，然後逐步分解為安全相關/非相關、安全/不安全，最終歸類到 SPF、RF、MPF 等具體的故障類別中。
  - **縱向邏輯：** 每一列代表一種特定的故障率指標。從上到下，數據是不同組件類型在該指標上的具體數值。工作表底部的匯總行（在此樣本中未完全顯示，但從公式中可推斷）會對所有組件的各項指標進行總計，這些總計值是計算整個系統 SPFM 和 LFM 的基礎。

## 4. 公式深度分析

此工作表包含 366 個公式，主要集中在第 2 行，並向下填充。以下是關鍵公式模式的分析：

1.  **數據引用與加總：**
    - **公式範例：** `=BlockList!C2` (B2), `=B2+C2` (D2)
    - **物理意義：** 從 `BlockList` 工作表引用特定組件類型的初始永久和暫態故障率，然後將它們相加以獲得該組件的總故障率。
    - **對應條款：** 這是 FMEDA 的基礎數據輸入，其準確性直接關乎整個分析的有效性。數據來源的可靠性需遵循 ISO 26262 Part 11（對半導體的應用）或行業標準數據手冊（如 IEC TR 62380）。
    - **正確性驗證：** 稽核員會抽查並驗證 `BlockList` 中的數據是否與組件供應商提供的數據手冊或可靠性報告一致。

2.  **條件求和 (SUMIFS)：**
    - **公式範例：** `=SUMIFS(BlockList!$F$10:$F$209,BlockList!$R$10:$R$209,A2)` (E2)
    - **物理意義：** 此公式計算特定組件類型（由 A2 單元格定義）的所有安全相關故障的總和。它在 `BlockList` 工作表中，根據 R 列的組件類型標籤，對 F 列的安全相關故障率進行條件求和。
    - **對應條款：** ISO 26262 Part 5, 8.4.4 要求對故障進行分類。此公式是實現該分類的具體技術手段。
    - **正確性驗證：** 需驗證 `SUMIFS` 的條件範圍（`$R$10:$R$209`）和求和範圍（`$F$10:$F$209`）是否完整覆蓋了 `BlockList` 中的所有相關條目。同時，條件（A2）的匹配邏輯必須正確。

3.  **故障率分佈計算：**
    - **公式範例：** `=E2/SUM($E$2:$E$7)` (K2)
    - **物理意義：** 計算當前組件類型（第 2 行）的永久性安全相關故障率（E2）佔所有組件類型總永久性安全相關故障率（`SUM($E$2:$E$7)`）的百分比。
    - **對應條款：** 雖然標準未明確要求此百分比計算，但這是進行設計優化和風險評估的常用實踐，有助於識別「壞蘋果」。
    - **正確性驗證：** 驗證分母的求和範圍（`$E$2:$E$7`）是否正確地包含了所有參與計算的組件類型。絕對引用（`$`）的使用是防止公式填充時出錯的關鍵。

4.  **跨 Sheet 匯總與複雜邏輯：**
    - **公式範例：** `=SUMIFS(FMEDA!$HZ$4:$HZ$1653,FMEDA!$HJ$4:$HJ$1653,"S")-FMEDA!$HX$2` (AL2)
    - **物理意義：** 此公式計算「被檢測到的單點故障（SPF detected）」的故障率。它首先在 `FMEDA` 主工作表中，對所有標記為安全機制（`HJ` 列為 "S"）所能檢測到的故障率（`HZ` 列）進行求和，然後再減去一個修正項（`HX2`），該修正項可能是安全機制本身的故障率，以避免重複計算。
    - **對應條款：** 這是計算診斷覆蓋率（Diagnostic Coverage, DC）和最終硬體度量的核心步驟，直接關聯 ISO 26262 Part 5 的 SPFM 和 LFM 計算要求。
    - **正確性驗證：** 這是稽核的重點和難點。稽核員會深入質詢：
        - `HJ` 列 "S" 的判定標準是什麼？是否有文件化的規則？
        - `HZ` 列的數據是如何從底層故障模式分析中得出的？
        - 修正項 `HX2` 的物理意義是什麼？為何需要減去它？其計算依據何在？

## 5. 稽核合規性分析

- **對應的 ISO 26262 條款：**
  - **Part 5, Clause 8 (Hardware architectural metrics):** 此工作表是計算單點故障度量（SPFM）和潛在故障度量（LFM）的數據基礎。表中的 SPF, RF, MPF 分類匯總值將直接用於這兩個指標的計算。
  - **Part 5, Clause 9 (Evaluation of hardware architectural metrics):** 稽核員會使用此表的結果來評估是否達到了目標 ASIL 等級所需的 SPFM 和 LFM 閾值（例如，ASIL B 要求 SPFM ≥ 90%, LFM ≥ 60%）。
  - **Part 11, Clause 4.5 (Failure rate analysis of semiconductor):** 此表引用的基礎故障率數據（來自 `BlockList`）必須遵循 Part 11 中關於半導體可靠性預測的指導原則。

- **滿足 ASIL B 等級的具體要求：**
  - **數據來源：** 用於計算的基礎故障率應基於公認的數據手冊（如 IEC TR 62380）、現場回饋數據或符合 Part 11 的分析方法。
  - **故障分類：** 必須提供清晰、一致的標準來區分安全相關/非相關、安全/不安全、SPF/RF/MPF。這些標準應被文件化並經過同行評審。
  - **度量計算：** 從此表匯總的數據計算出的 SPFM 和 LFM 必須達到或超過 ASIL B 的目標值。

- **稽核員可能會問的問題：**
  1.  「請解釋 A3 單元格 `Si MOS: High speed SRAM, FIFO` 的總故障率是如何從 `BlockList` 工作表的 `C3` 和 `L3` 繼承的？請展示 `BlockList` 中的對應條目。」
  2.  「在公式 `=SUMIFS(BlockList!...)` 中，你們是如何確保 `BlockList` 中的組件類型標籤（R 列）與本表 A 列的名稱完全一致的？是否有自動化檢查機制？」
  3.  「請解釋公式 `=...-FMEDA!$HX$2` 中 `HX2` 的含義。它代表什麼，為什麼要減去它？請提供計算該值的詳細追溯路徑。」
  4.  「數據樣本中出現了大量的 `#DIV/0!` 和 `#VALUE!` 錯誤，這表明某些基礎數據缺失或格式錯誤。你們將如何處理這些錯誤以完成最終的度量計算？目前的計算結果是否可信？」

- **需要準備的佐證文件：**
  1.  **FMEDA 方法論文件：** 詳細說明故障分類標準、數據來源選擇、以及 SPFM/LFM 計算公式的文件。
  2.  **組件數據手冊/可靠性報告：** 所有 `BlockList` 中引用的基礎故障率的來源文件。
  3.  **設計文檔：** 描述安全機制（Safety Mechanisms）如何檢測和控制故障的文檔，用於支持 `SUMIFS` 公式中的條件判斷。
  4.  **FMEDA 審查記錄：** 證明 FMEDA 已經過內部或外部專家審查的會議記錄和問題追蹤列表。
