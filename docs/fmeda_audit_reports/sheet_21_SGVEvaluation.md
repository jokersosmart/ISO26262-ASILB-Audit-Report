# FMEDA 深度稽核分析報告：Sheet 21 - SGVEvaluation

---

## 1. Sheet 概述

本文件為針對 FMEDA (Failure Modes, Effects, and Diagnostic Analysis) 工作簿中第 21 個工作表 **`SGVEvaluation`** 的深度稽核分析報告。

| 項目 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `SGVEvaluation` |
| **在報告中的位置** | 第 21 個 Sheet |
| **維度** | 33 行 × 57 列 (A1:BE33) |
| **公式數量** | 1034 |
| **圖片數量** | 1 |

此工作表在整個 FMEDA 分析流程中扮演著**高階結果彙總與驗證**的關鍵角色。其核心目的在於，從先前 `FMEDA` 主工作表中分析的數千個底層硬體元件故障模式中，根據其對系統級「安全目標 (Safety Goal)」的影響進行分類和匯總。它將複雜的底層數據提煉成直接對應於 ISO 26262 安全指標（SPFM, LFM, PMHF）的頂層失效率數據，是判斷整個設計是否滿足目標 ASIL 等級的最終依據之一。稽核員會將此表作為起點，來追蹤與驗證整個 FMEDA 分析的正確性與完整性。

## 2. 內容結構分析

`SGVEvaluation` 工作表的結構設計旨在提供一個清晰、多維度的安全目標違反 (Safety Goal Violation, SGV) 情況概覽。

- **欄位標頭 (Row 1)**: 標頭是本表的核心，佔據了從 A 到 BE 共 57 個欄位。這些欄位將失效率 (Failure Rate) 進行了精細的切割與分類，其組織方式圍繞以下幾個維度展開：
  - **故障類型**: 永久性 (Permanent) vs. 瞬態 (Transient) 故障。
  - **安全相關性**: 安全相關 (Safety related) vs. 非安全相關 (Not safety related)。
  - **故障影響**: 安全故障 (Safe) vs. 不安全故障 (Not Safe)。
  - **ISO 26262 故障分類**: 單點故障 (Single-Point Fault, SPF)、殘餘故障 (Residual Fault, RF)、多點故障 (Multiple-Point Fault, MPF)。
  - **多點故障狀態**: 已偵測 (Detected)、潛在 (Latent)。

- **資料組織邏輯**: 表格的每一行（從 Row 2 開始）代表一個獨立的**安全目標 (Safety Goal)**。例如，`SGV 01`, `SGV 02` 等。每一行中的各個欄位則透過公式，從 `FMEDA`、`BlockList` 等其他工作表中提取與該安全目標相關的失效率數據，並進行加總與分類。這種結構使得分析師和稽核員能夠清晰地看到每個安全目標的風險貢獻分佈，例如，某個安全目標的失效率主要是由單點故障還是潛在多點故障貢獻的。

- **合併儲存格**: 報告中提及的合併儲存格 `B23:C23` 和 `B27:C33`，通常用於標示總計或摘要資訊，在此表中可能用於顯示所有安全目標的總體失效率指標。

## 3. 圖片分析

此工作表中包含一張關鍵的流程圖，該圖是理解和稽核 FMEDA 計算邏輯的核心依據。

- **圖片檔案**: `sheet_21_SGVEvaluation_img_0.png`
- **圖片內容描述**: 此圖為 ISO 26262-5:2018 標準中定義的硬體元件故障模式分類流程圖。它從一個硬體元件的失效率（λ）開始，經過一系列的判斷菱形（例如，是否為安全相關、是否違反安全目標、是否有安全機制覆蓋等），最終將該失效率分配到不同的「桶」中，如安全故障（λ_S）、單點故障（λ_SPF）、殘餘故障（λ_RF）或多點故障（λ_MPF,L 或 λ_MPF,DP）。

- **代表意義與稽核重要性**: 與其他工作表中的類似流程圖不同，這張圖的**獨特之處在於增加了黃色的欄位標記**（例如 D, G, S, V, Y, AT, AQ 等）。這些標記直接將流程圖中的每一個決策點或終點，**映射到 `FMEDA` 主工作表的特定欄位**。

  > 這張圖相當於稽核員手中的「藏寶圖」或「驗證地圖」。稽核員可以透過這張圖，快速理解 `FMEDA` 工作表中複雜公式的設計邏輯。例如，圖中的標記 `V` 指向「單點故障 (SPF)」的輸出，稽核員可以立即去 `FMEDA` 表中檢查標記為 `V` 的欄位，驗證其計算公式是否確實只包含了那些「違反安全目標」且「無安全機制覆蓋」的故障模式。這極大地提高了稽核的效率和準確性，確保了 FMEDA 的計算嚴格遵循了 ISO 26262 的方法論。

- **與 ISO 26262 的對應關係**: 此圖完全對應於 ISO 26262-5:2018, Annex D 中描述的硬體架構度量計算方法。圖中的每一個分支和輸出都直接關係到單點故障度量 (SPFM) 和潛在故障度量 (LFM) 的計算。

## 4. 公式深度分析

此工作表的公式主要集中在第 2 行，用於匯總各個安全目標的失效率。雖然 JSON 中 `formulas` 陣列為空，但實際公式存在於儲存格值中。以下是關鍵公式模式的分析：

| 公式模式 | 範例 | 計算目的與物理意義 | 引用來源 | 對應 ISO 26262 條款 |
| :--- | :--- | :--- | :--- | :--- |
| **總失效率匯總** | `=IF(B2=0,0,SUM(BlockList!$C$2:$C$7))` | 如果該安全目標存在 (B2 > 0)，則從 `BlockList` 工作表中加總所有硬體模組的總失效率。這代表了與此安全目標相關的所有硬體元件的基礎失效率之和。 | `BlockList` | ISO 26262-5, 8.4.2 |
| **安全相關失效率** | `=IF(B2=0,0,BlockList!$D$2)` | 從 `BlockList` 中提取與此安全目標相關的「安全相關」部分的失效率。 | `BlockList` | ISO 26262-5, Annex E |
| **安全/不安全故障** | `=FMEDA!CL2` (Safe), `=FMEDA!CO2-M2` (Not Safe) | 從 `FMEDA` 主工作表的總計行中，直接引用已經計算好的「安全故障」總和，以及用「總不安全故障」減去「安全故障」來得到「非安全故障」。 | `FMEDA` | ISO 26262-5, 8.4.4 |
| **SPF/RF/MPF 匯總** | `=SUMIFS(FMEDA!..., FMEDA!..., ...)` | 使用 `SUMIFS` 函數，根據 `FMEDA` 主工作表中的分類標記（例如，判斷是否為 SPF 的欄位為 "No"），對不同類別的故障（SPF, RF, MPF）進行條件加總。 | `FMEDA` | ISO 26262-5, Annex D & E |
| **度量指標引用** | `=SafetyGoalViolations!E3` | 直接從 `SafetyGoalViolations` 工作表中引用已經計算完成的 SPFM 指標。 | `SafetyGoalViolations` | ISO 26262-5, 8.4.5 |

**公式正確性驗證要點**: 
1.  **跨表引用完整性**: 必須驗證所有 `SUMIFS` 的範圍 (`$4:$1653`) 是否完整覆蓋了 `FMEDA` 主表中的所有故障模式條目。
2.  **條件邏輯**: 必須仔細核對 `SUMIFS` 中的條件（例如 `\"S\"`, `\"No\"`, `\"Yes\"`）是否與 `FMEDA` 表中對應欄位的標記邏輯完全一致。
3.  **加減運算**: 檢查如 `=P2-S2` (總不安全 - 潛在違反 = MPF) 這類減法運算的邏輯是否符合 ISO 26262 的故障分類原則，確保沒有重複計算或遺漏。
4.  **與流程圖的一致性**: 最重要的，是將此處的每一個公式與「圖片分析」章節中提到的帶有黃色標記的流程圖進行比對，確保公式的計算路徑與圖中的邏輯分支完全吻合。

## 5. 稽核合規性分析

此工作表是證明產品符合 ISO 26262 硬體開發要求的核心證據之一。

- **對應的 ISO 26262 條款**:
  - **ISO 26262-5, Clause 8**: *Hardware architectural metrics*。此表直接匯總了計算 SPFM 和 LFM 所需的各類故障的失效率。
  - **ISO 26262-5, Clause 9**: *Evaluation of violation of the safety goal due to random hardware failures*。此表計算的 PMHF（或其組成部分）是此條款的關鍵輸入。
  - **ISO 26262-9, Clause 5**: *ASIL-dependent requirements*。此表匯總的結果必須與目標 ASIL 等級的要求進行比較（例如 ASIL B 的 SPFM > 90%）。
  - **ISO 26262-11, Part 11**: *Application of ISO 26262 to semiconductors*。此表所依賴的底層失效率計算（來自 `FailureRateCalc` 等表）需遵循 Part 11 的指導原則。

- **滿足 ASIL B 等級的具體要求**: 
  - **SPFM (Single-Point Fault Metric)**: 匯總的 SPFM 值必須 **≥ 90%**。
  - **LFM (Latent Fault Metric)**: 匯總的 LFM 值必須 **≥ 60%**。
  - **PMHF (Probabilistic Metric for random Hardware Failures)**: 匯總的 PMHF 值必須 **< 100 FIT**。
  此工作表是驗證這些指標是否達標的最終計算台。

- **稽核員可能會問的問題**:
  1.  「請解釋 `SGVEvaluation` 表是如何從 `FMEDA` 主表中匯總數據的？請展示一個具體的故障模式，追蹤它從底層到此表的完整路徑。」
  2.  「此表中瞬態故障 (Transient Fault) 的失效率是如何得出的？你們是否進行了瞬態故障分析？依據是什麼？」
  3.  「圖 `sheet_21_SGVEvaluation_img_0.png` 中的黃色標記 `Y` 代表殘餘故障 (Residual Fault)，請在 `FMEDA` 主表中指出對應的欄位，並解釋其計算公式為何能正確代表殘餘故障。」
  4.  「如果某個安全目標的 SPFM 未達標，你們的應對流程是什麼？如何使用此表來指導設計優化？」

- **需要準備的佐證文件**:
  1.  **安全計畫 (Safety Plan)**: 需定義 FMEDA 的執行方法、範圍和所使用的工具/模板。
  2.  **安全需求規格 (Safety Requirements Specification)**: 包含所有安全目標的詳細定義。
  3.  **設計規格書 (Design Specification)**: 用於追溯硬體元件及其功能。
  4.  **失效率數據來源報告**: 例如，基於 IEC TR 62380 或其他行業標準的失效率預測報告，用於證明 `BlockList` 中失效率數據的合理性。
  5.  **安全機制有效性分析報告**: 證明在 FMEDA 中為安全機制分配的診斷覆蓋率 (DC) 是有依據的。
