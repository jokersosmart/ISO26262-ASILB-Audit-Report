# FMEDA 深度稽核分析報告：Sheet 16 - DiagnosisInternal

---

## 1. Sheet 概述

本文件為針對 FMEDA (Failure Modes, Effects, and Diagnostic Analysis) 工作簿中第 16 個工作表「DiagnosisInternal」的深度稽核分析報告。此工作表是整個功能安全分析的核心部分，專門用於記錄與評估積體電路 (IC) 內部的診斷機制。

| 項目 | 內容 |
| :--- | :--- |
| **Sheet 名稱** | `DiagnosisInternal` |
| **在 FMEDA 的位置** | 第 16 個 Sheet (共 27 個) |
| **維度** | 29 行 × 17 列 |
| **角色與目的** | 此工作表的核心目的在於**系統性地登錄、描述並量化 IC 內部所有的安全診斷機制**。在 ISO 26262 的框架下，安全機制是控制硬體隨機失效、避免其違反安全目標的關鍵。本表單作為一個診斷機制的目錄，詳細記錄了每個機制的工作原理、偵測能力、反應時間與措施，其最終產出的「診斷覆蓋率 (Diagnostic Coverage, DC)」是計算「潛在失效度量 (Latent Fault Metric, LFM)」與評估整體硬體架構安全性的關鍵輸入。 |

## 2. 內容結構分析

「DiagnosisInternal」工作表的結構經過精心設計，旨在全面捕捉一個診斷機制所需的所有屬性，以滿足 ISO 26262 的稽核要求。

### 欄位標頭的含義與組織方式

工作表的主要內容由 A2 到 H2 的標頭定義，以下逐一說明其含義與在稽核中的重要性：

| 欄位標頭 | 英文原文 | 含義與稽核重點 |
| :--- | :--- | :--- |
| **ID Diagnosis** | ID Diagnosis | **唯一識別碼**：為每一個獨立的診斷機制分配一個獨一無二的 ID。稽核員會檢查此 ID 是否在整個安全文件中保持一致性，以便追溯。 |
| **診斷程序簡述** | Short description of diagnostic procedure | **功能摘要**：用一句話簡潔描述該機制的功能，例如「記憶體區塊的 ECC 錯誤校正」。 |
| **診斷程序詳述** | Detailed description of diagnostic procedure | **詳細工作原理**：深入解釋機制如何偵測失效。稽核員會關注此描述是否清晰、準確，並與設計規格書一致。 |
| **偵測與反應時間** | Detection and reaction time | **時間參數**：定義了從失效發生到系統完成安全反應所需的總時間。此時間必須小於安全目標所定義的「故障容錯時間間隔 (FTTI)」。這是稽核的絕對重點。 |
| **偵測到失效時的反應** | Reaction in case diagnostics detects a fail | **安全措施**：描述系統在偵測到失效後的具體行為，例如「進入安全狀態」、「重置系統」或「通知駕駛員」。 |
| **是否涵蓋暫態失效** | Covers transient faults | **暫態失效覆蓋能力**：標明此機制是否能偵測到暫時性的、非永久性的硬體錯誤。這對於評估系統在真實環境下的穩健性至關重要。 |
| **預估診斷覆蓋率 (DC)** | Diagnostic Coverage estimated | **量化指標**：此機制能偵測到特定元件特定失效模式的百分比。此數值直接影響 LFM 的計算結果，其估算必須有合理的依據（例如，來自 ISO 26262-5 附錄 D 的參考值）。 |
| **備註** | Comments | **補充資訊**：任何與該機制相關的額外說明或假設。 |

### 合併儲存格的用途

- **A1:H1**: 標題「IC internal diagnosis」橫跨整個內容區域，起到了分區標題的作用，清晰地標示了此工作表的用途範疇。

### 資料的組織邏輯

資料以**一行一個診斷機制**的方式進行組織。從第 3 行開始，每一行都代表一個獨立的安全機制。範例資料中的第一筆（第 3 行）顯示為「No mechanism available」，這是一個佔位符或預設項目，表示在某些情況下可能不存在對應的診斷機制。在實際分析中，此處應被具體的機制所取代。

## 3. 圖片分析

根據提供的 JSON 資料，**「DiagnosisInternal」工作表中不包含任何圖片**。所有與 FMEDA 方法論相關的圖示（如流程圖、定義圖）均位於其他輔助性工作表中（如 Sheet 3, 6, 8 等）。

## 4. 公式深度分析

根據提供的 JSON 資料，**「DiagnosisInternal」工作表中不包含任何 Excel 公式**。此工作表本質上是一個資料登錄表，其核心產出「預估診斷覆蓋率 (DC)」是一個需要專家判斷或引用標準的**輸入值**，而非計算結果。這些 DC 值後續會被 FMEDA 主工作表中的公式所引用，以計算 SPFM 和 LFM 等安全度量。

## 5. 稽核合規性分析

此工作表是證明產品符合 ISO 26262 硬體設計要求的核心證據之一。

### 對應的 ISO 26262 條款

| ISO 26262 條款 | 內容摘要 | 與本表的關聯 |
| :--- | :--- | :--- |
| **Part 5, Clause 8** | Hardware design | 本表是對硬體設計中安全機制的具體實現的詳細記錄。 |
| **Part 5, Clause 9** | Evaluation of the hardware architectural metrics | 本表提供的 DC 值是計算 SPFM (Single-Point Fault Metric) 和 LFM (Latent Fault Metric) 的直接輸入。 |
| **Part 5, Annex D** | Estimation of diagnostic coverage | 本表「預估診斷覆蓋率」欄位的數值應參考此附錄給出的指導和範例值。 |
| **Part 9, Clause 8** | Analysis of dependent failures | 本表描述的機制可能也用於應對相依失效（如時鐘或電源監控）。 |

### 滿足 ASIL B 等級的具體要求

對於 ASIL B 等級，ISO 26262 要求：
- **SPFM**: ≥ 90%
- **LFM**: ≥ 60%

要達到這些目標，必須實施有效的診斷機制。此工作表的重要性在於，它證明了開發團隊已經系統性地識別了失效模式，並設計了對應的診斷機制來覆蓋它們。一個內容詳實、DC 估算合理的「DiagnosisInternal」表是達成 ASIL B 指標的基礎。

### 稽核員可能會問的問題

1.  **DC 值的來源？** 針對「預估診斷覆蓋率」一欄，稽核員會追問每一個 DC 值的估算依據是什麼？是基於 ISO 26262-5 附錄 D 的表格，還是有更詳細的故障注入測試 (Fault Injection Testing) 結果來支撐？
2.  **時間參數的驗證？** 「偵測與反應時間」是如何得出的？是否有時序分析報告或測試結果可以證明此時間小於 FTTI？
3.  **機制的獨立性？** 表中列出的多個診斷機制之間是否存在相依性？一個機制的失效是否會影響另一個？
4.  **覆蓋範圍的完整性？** 如何證明表中列出的診斷機制已經涵蓋了所有相關的硬體元件及其關鍵失效模式？

### 需要準備的佐證文件

- **硬體設計規格書**：用以證明診斷機制的設計與實現細節。
- **故障注入測試報告**：如果執行了故障注入，此報告是證明 DC 值最有利的證據。
- **安全手冊 (Safety Manual)**：其中應包含對系統整合者使用這些診斷機制的要求。
- **時序分析報告**：證明滿足 FTTI 要求。

## 6. 稽核檢查清單

以下是稽核員在審查「DiagnosisInternal」工作表時會使用的檢查清單。

| 檢查項目 | Pass/Fail 判定標準 |
| :--- | :--- |
| **1. 唯一 ID** | **Pass**: 所有診斷機制都有唯一且格式一致的 ID。<br>**Fail**: 存在重複或遺漏的 ID。 |
| **2. 描述清晰度** | **Pass**: 簡述和詳述內容清晰、準確，與設計規格一致。<br>**Fail**: 描述模糊不清，或與其他文件矛盾。 |
| **3. 時間參數合理性** | **Pass**: 所有時間參數均有定義，且有證據支持其小於 FTTI。<br>**Fail**: 時間參數缺失，或無法提供其滿足 FTTI 的證據。 |
| **4. DC 估算依據** | **Pass**: 所有 DC 值都有明確的、可接受的依據（引用標準或測試數據）。<br>**Fail**: DC 值僅為隨意設定，缺乏任何支持證據。 |
| **5. 安全反應措施** | **Pass**: 所有機制都定義了清晰的失效反應措施。<br>**Fail**: 未定義反應措施，或措施不符合安全概念要求。 |
| **6. 暫態失效考慮** | **Pass**: 明確標註了對暫態失效的覆蓋能力。<br>**Fail**: 未考慮暫態失效。 |
| **7. 內容完整性** | **Pass**: 涵蓋了 FMEDA 分析中所需的所有關鍵診斷機制。<br>**Fail**: 明顯遺漏了關鍵元件（如 CPU、Memory）的診斷機制。 |

**稽核風險等級評估**: **高 (High)**

此工作表雖然沒有複雜的公式，但其內容的正確性、合理性和完整性直接決定了整個 FMEDA 的可信度以及硬體安全度量的計算結果。任何在 DC 估算或時間參數上的錯誤都可能導致對產品安全等級的嚴重誤判，因此將其稽核風險評定為「高」。
