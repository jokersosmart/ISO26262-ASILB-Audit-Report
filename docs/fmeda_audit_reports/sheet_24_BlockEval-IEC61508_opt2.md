# FMEDA 深度稽核分析報告：BlockEval-IEC61508_opt2

## 1. Sheet 概述

- **Sheet 名稱：** `BlockEval-IEC61508_opt2`
- **在 FMEDA 報告中的位置：** 第 24 個 sheet
- **維度：** 201 行 × 36 列 (A1:AJ201)
- **角色與目的：** 此工作表的核心目的是根據 IEC 61508 標準，對系統中的各個硬體「區塊 (Block)」或「子區塊 (Sub-block)」進行詳細的失效率評估。它整合了來自其他工作表（如 `BlockList`、`BlockEvaluation` 和 `SafetyGoalViolations`）的基礎失效率數據，並計算一系列關鍵的功能安全指標，包括安全失效率 (SFF) 和每小時危險失效機率 (PFH)。此表單是從底層元件失效率到頂層系統級安全指標的關鍵橋樑，其計算結果直接用於判斷系統是否滿足特定的安全完整性等級 (SIL) 或汽車安全完整性等級 (ASIL) 要求。

## 2. 內容結構分析

此工作表的結構設計旨在對每個硬體區塊的各種失效模式進行全面量化分析。

- **欄位標頭的含義與組織方式：**
  - **A-C 欄：** 識別區塊的基本資訊（編號、名稱、類型），通常引用自 `BlockList` 工作表。
  - **D-L 欄：** 分解初始總失效率。首先區分「永久性 (permanent)」和「暫態 (transient)」失效，然後進一步將其歸類為「安全相關 (Safety related)」與「非安全相關 (Not safety related)」。這是 FMEDA 分析的基礎，確保只有影響安全目標的失效才會被納入後續的量化分析。
  - **M-O 欄：** 計算安全相關失效率在整個系統中的分佈比例。
  - **P-AA 欄：** 這是分析的核心部分。它將「安全相關」的失效率進一步細分為「安全失效 (Safe)」、「危險失效 (Dangerous)」，並將危險失效再細分為「可被偵測到的危險失效 (Dangerous Detected)」和「無法被偵測到的危險失效 (Dangerous Undetected)」。這個分類是計算 SFF 和 PFH 的基礎。
  - **AB-AD 欄：** 計算「無法被偵測到的危險失效」在所有未檢測到的危險失效中的分佈比例。
  - **AE-AG 欄：** 計算安全失效率 (SFF)，這是一個關鍵指標，用於衡量安全機制的有效性。公式通常為 `(λ_S + λ_DD) / (λ_S + λ_D)`。
  - **AH-AJ 欄：** 計算每小時危險失效機率 (PFH)，這是評估系統違反安全目標風險的最終指標，直接對應 ASIL 等級的要求。

- **合併儲存格的用途：** 在此工作表中，未發現使用合併儲存格。所有資料都以單一儲存格的形式進行組織，有利於公式的複製和資料的清晰呈現。

- **資料的組織邏輯：** 每一橫列代表一個獨立的硬體區塊。資料從左到右呈現了一個清晰的分析流程：從總失效率開始，逐步分解、歸類，並最終計算出 SFF 和 PFH 兩個核心安全指標。這種邏輯緊密貼合 ISO 26262 和 IEC 61508 的要求。

## 4. 公式深度分析

此工作表包含大量公式 (7200個)，且公式之間存在複雜的相依性。以下列出幾個關鍵且具代表性的公式模式：

1.  **資料引用公式 (Data-linking Formulas):**
    - **公式範例：** `=BlockList!B10` (B2), `=BlockEvaluation!P2` (P2)
    - **物理意義 / 計算目的：** 從其他專門的工作表（如 `BlockList`, `BlockEvaluation`）中引用基礎數據，例如區塊名稱、初始失效率或更底層的分析結果。這確保了數據的一致性和可追溯性。
    - **引用的 Sheet/儲存格：** `BlockList`, `BlockEvaluation`
    - **對應的 ISO 26262 條款：** ISO 26262-5: D.2 (FMEDA)，強調輸入數據的來源和準確性。
    - **正確性驗證要點：** 稽核員必須驗證所引用的工作表和儲存格是否正確，且來源數據是否已經過審核和批准。任何對來源工作表的修改都會直接影響此處的計算結果。

2.  **失效率加總/減法公式 (Failure Rate Aggregation/Subtraction):**
    - **公式範例：** `=D2+E2` (F2), `=D2-G2` (J2)
    - **物理意義 / 計算目的：** 計算永久性和暫態失效的總和，或從總失效率中減去安全相關的部分以得到非安全相關的失效率。這是基本的失效率分解運算。
    - **引用的 Sheet/儲存格：** 同一列中的其他儲存格。
    - **對應的 ISO 26262 條款：** ISO 26262-5: Annex D, Table D.3。
    - **正確性驗證要點：** 檢查加減法的邏輯是否符合 FMEDA 的定義。例如，`λ_total = λ_permanent + λ_transient`，`λ_not_safety_related = λ_total - λ_safety_related`。

3.  **條件判斷與數據查找公式 (Conditional Logic & Lookup):**
    - **公式範例：** `=IF(BlockList!L10=0,0,IF(BlockList!P10="",0,...))` (D2)
    - **物理意義 / 計算目的：** 根據 `BlockList` 中的特定條件（例如，失效率是否為0，或是否有指定的元件類型）來決定如何計算初始失效率。這通常用於處理不同類型元件或不同數據來源的情況。
    - **引用的 Sheet/儲存格：** `BlockList`
    - **對應的 ISO 26262 條款：** ISO 26262-5: D.2.3，要求分析過程具有系統性。
    - **正確性驗證要點：** 這是稽核的重點。必須仔細審查 `IF` 條件的邏輯，確保其完整覆蓋了所有可能情況，並且查找的目標值是正確的。此公式的複雜性使其成為潛在的錯誤來源。

4.  **安全指標計算公式 (Safety Metric Calculation):**
    - **SFF 公式範例：** `=IF(G2=0,"-",IF(P2+S2=0,"Not calculated",(P2+V2)/(P2+S2)))` (AE2)
    - **PFH 公式範例：** `=IF(G2=0,0,BlockEvaluation!AE2+SafetyGoalViolations!$F$37*BlockEvaluation!AZ2)` (AH2)
    - **物理意義 / 計算目的：**
        - **SFF (安全失效率):** 計算安全失效和被偵測到的危險失效佔總安全相關失效率的比例。公式本質是 `(λ_safe + λ_dangerous_detected) / (λ_safe + λ_dangerous)`。此處的 `P2` (Safe) 和 `V2` (Dangerous Detected) 分別對應 `λ_safe` 和 `λ_dangerous_detected`。
        - **PFH (每小時危險失效機率):** 計算導致違反安全目標的殘餘風險。它通常基於「無法被偵測到的危險失效 (λ_DU)」和「被偵測到但延遲修復的危險失效 (λ_MPF,latent)」計算得出。此處公式直接引用了 `BlockEvaluation` 和 `SafetyGoalViolations` 的計算結果，顯示了高度的模組化。
    - **引用的 Sheet/儲存格：** `BlockEvaluation`, `SafetyGoalViolations`，以及同列的其他儲存格。
    - **對應的 ISO 26262 / IEC TR 62380 條款：**
        - **SFF:** ISO 26262-5: 8.4.4
        - **PFH:** ISO 26262-5: 9.4.2.1
    - **正確性驗證要點：**
        - **SFF:** 驗證分子和分母的組成是否正確。稽核員會特別關注 `IF` 條件，例如當總失效率為零時的處理方式，確保不會出現除以零的錯誤 (`#DIV/0!`)。從 `data_sample` 中看到的大量錯誤表明，此處的錯誤處理機制可能不足或輸入數據有問題。
        - **PFH:** 驗證公式是否完整考慮了所有導致違反安全目標的失效模式。對 `SafetyGoalViolations!$F$37` 這類常數的引用需要追溯其定義，確保其代表的意義（例如，多點失效的診斷覆蓋率）是正確且有依據的。

## 5. 稽核合規性分析

- **對應的 ISO 26262 條款：**
  - **Part 5: Product Development at the Hardware Level:** 這是最主要的關聯部分，特別是第 8 節「Hardware architectural metrics」和第 9 節「Evaluation of violation of the safety goal due to random hardware failures」。此工作表正是這兩節要求的直接實踐。
  - **Part 9: Automotive Safety Integrity Level (ASIL)-oriented and safety-oriented analyses:** 特別是第 6 節「Failure mode and effects analysis (FMEA)」，FMEDA 是 FMEA 的一種量化擴展。
  - **Part 11: Guidelines on application of ISO 26262 to semiconductors:** 對於半導體元件，此部分提供了更詳細的失效率預測和失效模式分類的指導。

- **滿足 ASIL B 等級的具體要求：**
  - **PFH:** 對於 ASIL B，單點和殘餘失效指標 (SPFM/LFM) 的目標值是建議性的，但 PFH 的目標值是強制性的。系統的總 PFH 必須小於 10⁻⁷ /h。此工作表計算了每個區塊的 PFH，最終需要將所有相關區塊的 PFH 加總，以驗證是否滿足此要求。
  - **SFF:** 雖然 ISO 26262 對 ASIL B 的 SFF 沒有強制要求，但通常會要求達到一個合理的數值（例如 >60%），以證明安全機制的有效性。此工作表計算了 SFF，可作為佐證。

- **稽核員可能會問的問題：**
  1.  「請解釋 `BlockList`、`BlockEvaluation` 和 `SafetyGoalViolations` 這三個工作表的數據來源和可信度。你們如何確保輸入數據的準確性？」
  2.  「在 `data_sample` 中觀察到大量的 `#DIV/0!` 和 `#VALUE!` 錯誤。請解釋原因，並說明這對最終的 PFH/SFF 計算結果有何影響？你們是否有應對計畫？」
  3.  「請詳細說明公式 `AH2` 中引用的 `SafetyGoalViolations!$F$37` 這個常數的物理意義是什麼？它是如何得出的？請提供依據文件。」
  4.  「你們是如何區分『永久性失效』和『暫態失效』的？其失效率的比例是如何決定的？請提供參考的標準或數據手冊（如 IEC TR 62380）。」
  5.  「對於公式 `D2` 中複雜的 `IF` 巢狀結構，你們是否進行過單元測試或模擬，以確保其在所有邊界條件下都能正確計算？」

- **需要準備的佐證文件：**
  1.  **元件失效率數據手冊：** 用於證明 `BlockList` 中初始失效率數據來源的權威性（例如，SN 29500, FIDES, 或供應商提供的數據）。
  2.  **安全手冊 (Safety Manual):** 針對所使用的複雜元件（如微控制器），其中應包含元件級的失效模式分佈。
  3.  **設計文檔：** 描述每個硬體區塊的功能及其與安全目標的關係。
  4.  **安全機制設計文檔：** 詳細說明用於偵測危險失效的診斷機制，及其診斷覆蓋率 (Diagnostic Coverage, DC) 的計算依據。
  5.  **本 FMEDA 文件的版本控制記錄和審查記錄。**
