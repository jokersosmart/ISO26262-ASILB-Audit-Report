# FMEDA 深度稽核分析報告：Sheet 20 - CCF Calc. Logic

## 1. Sheet 概述

- **Sheet 名稱**: `CCF Calc. Logic`
- **在 FMEDA 報告中的位置**: 第 20 個 sheet
- **維度**: 61 行 × 19 列 (A1:S61)
- **角色與目的**: 此 sheet 的核心目的是根據 IEC 61508 Ed. 2 附錄 D 中定義的 β-factor 模型，對邏輯硬體（如微控制器、ASIC）的**共因失效（Common Cause Failures, CCF）**進行量化評估。β-factor 代表了在一個冗餘系統中，由單一根因導致多個通道同時失效的機率。此工作表透過一個結構化的問卷，引導使用者評估其設計在防禦共因失效方面的強度，最終計算出一個具體的 β 值，該值將被用於 FMEDA 主表中計算多點故障的失效率，是硬體架構度量（如 LFM）計算的關鍵輸入。

## 2. 內容結構分析

- **欄位標頭與組織**: 
  - 標題 `A1` 明確指出此表的計算依據是「IEC 61508 Ed. 2 for Logic」。
  - `A3:I3` 定義了核心評估區的欄位：`XLS`, `YLS`, `XSF`, `YSF` 是針對不同條件下的權重分數，`X` 和 `Y` 是根據使用者輸入計算出的加權分數，`yes = 1 no = 0` 指示使用者應如何在 `H` 欄填寫，`Remark` 則用於補充說明。
  - `N` 欄及之後的欄位提供了額外的上下文，例如 `N5` 指出權重值的來源是「IEC 61508 Table D.2」。

- **合併儲存格**: 大量使用水平合併儲存格（如 `A4:I4`, `A10:I10`）將一系列具體的防禦措施問題（例如 A5, A6, A7）歸類到一個大的防禦主題下（例如 A4: 「Separation/segregation」）。這種結構使得問卷層次分明，易於理解和填寫。

- **資料組織邏輯**: 此表本質上是一個計分卡。`A` 欄提出問題，這些問題代表了 IEC 61508 建議的、能有效降低共因失效的設計措施。使用者在 `H` 欄回答「是 (1)」或「否 (0)」，表示其設計是否實作了該措施。`B` 到 `E` 欄是預設的權重分數，`F` 和 `G` 欄的公式會將權重與使用者的回答相乘，得到該項的得分。最後，所有得分會被加總，並透過一個轉換公式計算出最終的 β-factor。

## 4. 公式深度分析

此 sheet 包含 67 個公式，但它們遵循幾個非常一致的模式。

- **獨特公式模式**: 
  1. **加權分數計算**: `=[Weight_Cell]*[Answer_Cell]`
     - **範例**: `F5` 中的 `=B5*H5`，`G5` 中的 `=C5*H5`
     - **物理意義/計算目的**: 此公式用於計算單一防禦措施的得分。`B5` (1.5) 是該措施的基礎權重，`H5` (0 或 1) 是使用者對「是否實施」的回答。如果回答為「是」(1)，則該項得分為 1.5；如果為「否」(0)，則得分為 0。這實現了一個自動化的計分過程。
     - **引用的儲存格**: 僅引用同一行內的權重和答案儲存格，無跨 sheet 引用。
     - **對應的 ISO 26262 / IEC TR 62380 條款**: 整個方法論基於 **IEC 61508-6:2010, Annex D** 的 β-factor 模型。這個模型被 ISO 26262 所接受，作為進行相依失效分析（Dependent Failure Analysis, DFA）的量化方法之一（**ISO 26262-9:2018, Clause 6**）。
     - **正確性驗證要點**: 
       - 驗證每個計分公式是否都正確地將權重與 `H` 欄的對應儲存格相乘。
       - 抽查權重值（B-E欄）是否與 IEC 61508 Table D.2 中建議的值一致。

  2. **總分與 β-factor 轉換 (未在樣本中顯示，但為必要邏輯)**: 
     - **預期公式**: `SUM(F5:F_last)` 和 `SUM(G5:G_last)`，以及一個基於總分的查表或轉換公式來得到最終的 β 值。
     - **物理意義/計算目的**: 將所有單項得分加總，得到一個綜合評分。這個總分越高，代表設計的 CCF 防禦能力越強，對應的 β-factor 就越低。β-factor 通常是透過一個非線性公式或查表從總分轉換而來。
     - **正確性驗證要點**: 稽核員必須找到並驗證這個最終的轉換公式或查表，確保其符合 IEC 61508 的規範。任何對轉換模型的修改都必須有充分的理由和文件支持。

## 5. 稽核合規性分析

- **對應的 ISO 26262 條款**:
  - **ISO 26262-9:2018 (ASIL-dependent and safety-oriented analyses)**: 此 sheet 直接實作了第 6 條「Analysis of dependent failures」的要求，特別是針對共因失效的量化分析。
  - **ISO 26262-5:2018 (Product development at the hardware level)**: β-factor 的計算結果會直接影響硬體架構度量（LFM）的計算，這是第 8.4.5 條的要求。
  - **ISO 26262-11:2018 (Guidelines on application of ISO 26262 to semiconductors)**: 對於 ASIC 或邏輯晶片，此表提供了評估其內部冗餘設計 CCF 穩健性的方法。

- **滿足 ASIL B 等級的要求**: 對於 ASIL B，標準允許進行定性的 CCF 分析。然而，採用像 β-factor 這樣的量化模型被視為更嚴謹、更強健的作法。使用此工作表並提供合理的輸入，不僅滿足甚至超越了 ASIL B 的基本要求，為安全論證提供了強有力的證據。

- **稽核員可能會問的問題**:
  1. 「對於 `H` 欄中每一個回答『是』(1) 的項目，請提供具體的設計文件、程式碼或測試報告作為證據。」
  2. 「填寫這份評估表的人員是誰？他/她是否具備足夠的系統架構和硬體設計知識來做出準確的判斷？」
  3. 「此處使用的權重值直接取自 IEC 61508，你們是否評估過這些通用權重對於你們特定應用的適用性？」
  4. 「最終計算出的 β-factor 是多少？這個值是否在你們專案的安全計畫（Safety Plan）中預先設定的可接受範圍內？」

- **需要準備的佐證文件**:
  - 針對問卷中每個「是」的回答，準備對應的設計規格書、電路圖、佈線規則文件、軟體架構文件等。
  - 執行相依失效分析（DFA）的報告。
  - 團隊成員的資格和經驗記錄。
  - 專案的安全計畫，其中應定義 CCF 分析的策略和目標。
