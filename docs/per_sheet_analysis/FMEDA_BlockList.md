### 公式 1：Si MOS: Digital circuits, Micros, DSP - Base Failure Rate Permanent (per Block Type)  [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | C2 |
| 公式原文 | `=IF($AJ$2="No",FailureRateCalc!E23,(FailureRateCalc!E23+FailureRateCalcIC!$D$129*(BlockList!E2/SUM(BlockList!$E$2:$E$7))+FailureRateCalcIC!$D$130*(BlockList!E2/SUM(BlockList!$E$2:$E$7))))` |
| 應用範圍 | C2 到 C7（共 6 個儲存格） |
| 計算值 | 0 |

**參照來源**：None 的 $AJ$2、FailureRateCalc 的 E23、FailureRateCalc 的 E23、FailureRateCalcIC 的 $D$129、BlockList 的 E2、BlockList 的 $E$2:$E$7、FailureRateCalcIC 的 $D$130、BlockList 的 E2、BlockList 的 $E$2:$E$7

**計算邏輯（白話文）**：這個公式用來計算一個「區塊類型（Block Type）」的基礎永久性故障率（Base Failure Rate, Permanent）。計算方式是一個條件判斷：

1.  **如果儲存格 $AJ$2 的值是 "No"**：那麼這個區塊的故障率就直接等於 `FailureRateCalc` 工作表中 `E23` 儲存格的值。
2.  **如果儲存格 $AJ$2 的值不是 "No"**：那麼計算會變得複雜。它會以 `FailureRateCalc` 的 `E23` 為基礎，再加上兩個額外的故障率。這兩個額外的故障率是從 `FailureRateCalcIC` 工作表的 `$D$129` 和 `$D$130` 來的，但會根據目前這個區塊的電晶體數量（`BlockList!E2`）佔總電晶體數量（`SUM(BlockList!$E$2:$E$7)`）的比例來做加權分配。

簡單來說，這個公式在決定要用一個簡單的、固定的基礎故障率，還是要用一個考慮了更多因素（來自 `FailureRateCalcIC`）並按電晶體比例分配的、更精細的故障率。

**物理意義**：這個公式計算的是 SSD 控制器晶片中，某一個特定功能區塊（例如，NAND Flash 控制器、DRAM 控制器、處理器核心等）因為物理缺陷（例如，製程變異、材料老化）而導致永久性失效的頻率，單位是 FIT（Failure In Time），即十億小時內發生一次失效。這個值是整個 FMEDA 分析的起點，代表了該區塊的固有不可靠性。

`$AJ$2` 的開關，物理意義上代表了分析的兩種不同情境或假設：

*   **"No" 的情境**：可能代表一個簡化的分析模型，假設所有區塊的基礎故障率都相同，或者採用一個通用的、保守的估計值。這在早期設計階段或缺乏詳細資料時很常見。
*   **非 "No" 的情境**：代表一個更精細的分析模型。它不僅考慮了通用的基礎故障率，還把另外兩種特定的失效模式（可能來自 `FailureRateCalcIC` 中定義的 Die 級或 Package 級的額外風險）按比例分攤到各個功能區塊。這個比例的依據是電晶體數量，因為通常電晶體越多的區塊，其佔據的晶片面積越大，發生隨機硬體失效的機率也越高。

**標準依據**：這個計算方法本身並非直接來自某個標準的單一公式，而是融合了多個標準的原則和實務做法，主要依據如下：

1.  **IEC TR 62380 (RDF 2000)**：這份文件提供了在不同操作條件下，電子元件（包括積體電路）的故障率預測模型。公式中參照的 `FailureRateCalc!E23` 的基礎 FIT 值，很可能就是基於 IEC TR 62380 的模型計算得出的，它會考慮到製程技術、溫度、電壓等因素。
2.  **SN 29500**：這是西門子的內部標準，但已被廣泛業界採用作為 FIT 計算的參考。與 IEC TR 62380 類似，它也提供了一套完整的電子元件故障率計算方法。`FailureRateCalc!E23` 的值也可能來自此標準。
3.  **ISO 26262-5, Annex D (Quantitative analysis of residual failure)**：雖然 ISO 26262 本身不提供具體的 FIT 計算公式，但它在附錄中闡述了如何使用 FMEDA（Failure Modes, Effects, and Diagnostic Analysis）方法來進行硬體架構度量的量化分析。這個公式正是 FMEDA 實作的一部分，它計算出的 λ (Failure Rate) 是後續計算 SPFM、LFM 和 PMHF 的基礎。

將額外故障率按電晶體數量比例分配的方法，是功能安全工程中的一種常見實踐，旨在將晶片級別的整體風險（如封裝或電源域的風險）合理地分配到各個子模組。

**為什麼要這樣算**：從安全工程的角度來看，這樣計算有以下幾個關鍵原因：

1.  **風險量化與分配**：為了評估整個晶片是否能達到特定的汽車安全完整性等級（ASIL），我們必須先將風險量化到每一個最小的功能單元。如果沒有這個計算，我們就無法知道每個區塊對整體安全目標的貢獻和風險。
2.  **設計優化指導**：計算出每個區塊的故障率後，ASIC 設計團隊可以識別出哪些是「最脆弱」的區塊（即故障率最高的區塊）。這些高風險區塊就需要投入更多的設計資源來增加其可靠性，例如，為其增加錯誤修正碼（ECC）、冗餘設計（Redundancy）或內建自我測試（Built-in Self-Test, BIST）等安全機制。
3.  **支援多種分析情境**：`IF` 條件的存在，使得這份 FMEDA 表格非常靈活。分析師可以根據專案階段、可用資料的精確度，或客戶的特定要求，在簡化模型和精細模型之間切換，而無需修改大量的公式。
4.  **建立追溯性**：這個公式清晰地展示了每個區塊的故障率是如何從更基礎的數據（`FailureRateCalc`）和晶片級的數據（`FailureRateCalcIC`）推導出來的，並明確了分配的邏輯（按電晶體數量）。這在後續的審核（Audit）和認證（Certification）過程中至關重要，因為它證明了分析的系統性和合理性。

**ASIC 部門注意事項**：**ASIC 部門注意事項**：

1.  **確認電晶體數量的準確性**：公式的核心是按電晶體數量比例分配故障率。因此，`BlockList!E2` 到 `E7` 的電晶體數量（Gate Count）必須盡可能準確。這需要 ASIC 前端（RTL 設計）和後端（實體設計）團隊緊密合作，在設計的不同階段（如 Pre-synthesis, Post-synthesis, Post-layout）提供最新的估計值或精確值。錯誤的電晶體數量會導致故障率分配不均，可能使某些區塊的風險被低估。
2.  **理解 `FailureRateCalcIC` 的來源**：ASIC 團隊需要與功能安全團隊溝通，徹底理解 `FailureRateCalcIC!$D$129` 和 `$D$130` 這兩個值的物理意義。它們代表的是什麼樣的額外風險？是與特定製程節點相關的 Die-level 風險，還是與封裝（Package）相關的風險？理解這些，才能在設計中採取針對性的緩解措施。
3.  **區塊劃分的合理性**：`BlockList` 中區塊的劃分（Block Partition）應與晶片的實際架構和功能邊界保持一致。如果一個被定義為單一「區塊」的模組，在物理上跨越了不同的電壓域（Voltage Domain）或時脈域（Clock Domain），那麼用單一的故障率來描述它可能就不夠準確。ASIC 團隊應參與到區塊劃分的討論中，確保其合理性。
4.  **關注高故障率區塊**：在 FMEDA 分析結果出來後，ASIC 團隊應立即將注意力集中在那些被計算出具有高基礎故障率的區塊上。需要評估現有的設計是否足以應對這些風險。如果不足，則必須考慮增加硬體安全機制。

### 公式 2：Si MOS: Digital circuits, Micros, DSP - Sum (all Block Failure Rates) Permanent [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | D2 |
| 公式原文 | `=SUM($F$10:$F$9981)` |
| 應用範圍 | D2 (共 1 個儲存格) |
| 計算值 | #DIV/0! |

**參照來源**：None 的 $F$10:$F$9981

**計算邏輯（白話文）**：這個公式非常直接，它計算 F 欄從第 10 行到第 9981 行所有數值的總和。在 `BlockList` 這個工作表的上下文中，F 欄代表的是「每個單獨區塊（Block）的故障率」。因此，這個公式計算的是整個晶片所有已定義區塊的永久性故障率總和。

**物理意義**：這個計算結果代表了整個 SSD 控制器晶片的「總基礎永久性故障率（Total Base Failure Rate, Permanent）」。它是構成晶片的所有功能區塊（列在 F10:F9981 範圍內）各自故障率的簡單疊加。這個值是評估晶片整體可靠性的關鍵指標之一。如果這個總和值過高，意味著即使為每個區塊都設計了安全機制，晶片整體的失效風險依然可能超出安全目標（如 ASIL B 的 PMHF 目標值）。

**標準依據**：**ISO 26262-5, Annex D**：此處的求和運算，是遵循 FMEDA 方法論的自然結果。在 FMEDA 中，整個硬體元件（Hardware Part，此處指 SSD 控制器晶片）的總故障率 (λ) 是其所有組成部分（此處指功能區塊）故障率之和。公式為：λ_total = Σ λ_i，其中 λ_i 是第 i 個子部分的故障率。

**Part-Count Failure Prediction**：這種直接將各部分故障率相加的方法，是可靠性工程中最基礎的「元件計數法」失效率預測模型。它假設各個部分的失效是相互獨立的。

**為什麼要這樣算**：1.  **宏觀風險評估**：計算總故障率是為了從宏觀上把握整個晶片的可靠性水平。它為安全經理和專案經理提供了一個頂層的、易於理解的數字，用以判斷設計是否在正確的軌道上。
2.  **作為後續計算的輸入**：這個總故障率（D2 的值）會被其他公式引用，用於計算整個晶片的平均故障診斷覆蓋率（DC_avg）或作為計算 PMHF（Probabilistic Metric for Hardware Failure）指標的一部分。
3.  **設定預算和目標**：在設計初期，可以先為整個晶片設定一個總故障率「預算」，然後再將這個預算分配給各個功能區塊。在設計過程中，可以透過這個公式來追蹤所有區塊故障率的總和是否超出了最初設定的預算。

**ASIC 部門注意事項**：**ASIC 部門注意事項**：

1.  **注意計算範圍的完整性**：公式的範圍是 `$F$10:$F$9981`。ASIC 團隊在新增或刪除功能區塊時，必須確保 `BlockList` 的結構保持一致，即所有區塊的故障率都落在這個範圍內。如果有區塊被錯誤地放在了第 9982 行或之後，它的故障率將不會被計入總和，導致對整個晶片的風險低估。
2.  **理解 `#DIV/0!` 錯誤**：目前計算值顯示為 `#DIV/0!`。這通常意味著被加總的範圍內（$F$10:$F$9981）存在除以零的錯誤。ASIC 團隊應與功能安全團隊合作，追查 F 欄中哪個或哪些區塊的故障率計算出了問題，並予以修正。在所有區塊的故障率都有有效值之前，這個總故障率是沒有意義的。
3.  **F 欄的資料來源**：ASIC 團隊需要知道 F 欄的每個值是如何計算出來的。它們是直接輸入的，還是由其他更複雜的公式計算得出的？理解這一點有助於判斷當某個區塊的故障率過高時，應該從何處著手進行優化。
