**物理意義**：

**參照來源**：
- `SafetyGoalViolations` 工作表的 `C3` 儲存格

**計算邏輯（白話文）**：
這個公式從 `SafetyGoalViolations` 工作表中，複製儲存格 `C3` 的內容到目前的儲存格。此處的 `0` 可能表示一個特定的安全目標違規（SGV）編號，或是代表「無違規」。

此欄位代表與特定硬體元件相關聯的安全目標違規（SGV）的識別碼。在功能安全分析中，SGV 是指因系統故障而可能導致的危害事件。這個公式將一個特定的SGV與一個或多個硬體區塊連結起來，表示這些硬體區塊的失效可能會導致該SGV的發生。

**為什麼要這樣算**：
為了建立從系統級安全目標到具體硬體元件的追溯鏈。透過將SGV分配給特定的硬體區塊，功能安全工程師可以分析該區塊的失效模式將如何影響系統安全，並評估其風險等級。

**ASIC 部門注意事項**：
ASIC 設計團隊必須清楚了解分配給他們設計的硬體區塊的SGV。這有助於他們在設計過程中，針對性地導入安全機制（Safety Mechanisms），以降低因硬體隨機失效導致SGV發生的機率。例如，如果一個SGV與資料毀損有關，ASIC團隊就應該考慮加入ECC（Error Correction Code）等保護機制。

### 公式 4：Initial / Total Failure Rate permanent [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | D2 |
| 公式原文 | `=IF(B2=0,0,SUM(BlockList!$C$2:$C$7))` |
| 應用範圍 | D2 到 D21（共 20 個儲存格） |
| 計算值 | 0 |

**參照來源**：
- 目前工作表的 `B2` 儲存格
- `BlockList` 工作表的 `$C$2:$C$7` 儲存格範圍

**計算邏輯（白話文）**：
此公式首先檢查同一列的 `B2` 儲存格。如果 `B2` 的值為 0（通常意味著該列沒有對應到一個有效的安全目標違規），則此儲存格的結果也為 0。若 `B2` 不為 0，則公式會加總 `BlockList` 工作表中從 `C2` 到 `C7` 儲存格範圍內的所有數值，這個總和代表了與該安全目標相關的所有硬體區塊的總永久失效率。

**物理意義**：
這個公式計算的是與特定安全目標違規（SGV）相關的所有硬體元件的 **初始總永久失效率（Initial Total Permanent Failure Rate）**，單位為 FIT（Failure In Time，每十億小時的失效次數）。這是整個安全分析的基礎，代表了在沒有任何安全機制介入的情況下，硬體因物理原因（如老化、電遷移）發生永久性失效的基礎概率。`BlockList!$C$2:$C$7` 範圍內的數值，是構成一個完整功能（如一個 IP）的各個子模組的永久失效率。

**標準依據**：
- **失效率數據來源**：計算 FIT 值的基礎數據通常來自業界公認的標準，如 **IEC TR 62380** 或 **SN 29500**。這些標準提供了基於元件類型、邏輯閘數（Gate Count）、溫度、製程等因素的失效率預測模型。
- **FMEDA 方法學**：此計算是執行 FMEDA（Failure Modes, Effects, and Diagnostic Analysis）的第一步，符合 **ISO 26262-5, Annex D** 的指導原則，即首先需要確定元件的基礎失效率。

**為什麼要這樣算**：
這是為了量化評估硬體隨機失效對安全目標的影響。首先計算出一個「裸」失效率，作為基準線。後續的分析將評估各種安全機制（Safety Mechanism）能將這個初始失效率降低多少，從而計算出 SPFM（單點故障指標）和 LFM（潛在故障指標）等關鍵的硬體架構指標。將永久（Permanent）和暫態（Transient）失效分開計算，是因為它們的物理成因和應對策略完全不同。

**ASIC 部門注意事項**：
- **提供準確的邏輯閘數**：`BlockList` 中 C 欄的失效率，其計算起點極大程度依賴於 ASIC 設計團隊提供的 **邏輯閘數（Gate Count）** 或 **晶片面積（Area）**。設計團隊必須為 FMEDA 分析提供準確、詳細的每個硬體模組的邏輯閘數。
- **理解設計與失效率的關係**：ASIC 工程師需要意識到，設計的複雜度、所選用的標準單元庫（Standard Cell Library）類型，都會直接影響到最終的 FIT Rate。例如，使用高密度但可靠性較低的單元庫，可能會增加 FIT Rate。
- **協同定義模組邊界**：ASIC 團隊需要與功能安全團隊緊密合作，清晰地定義 `BlockList` 中每一個硬體區塊的邊界，確保失效率計算的範圍與實際設計一致。

### 公式 5：Initial / Total Failure Rate transient [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | E2 |
| 公式原文 | `=IF(B2=0,0,SUM(BlockList!$L$2:$L$7))` |
| 應用範圍 | E2 到 E21（共 20 個儲存格） |
| 計算值 | 0 |

**參照來源**：
- 目前工作表的 `B2` 儲存格
- `BlockList` 工作表的 `$L$2:$L$7` 儲存格範圍

**計算邏輯（白話文）**：
此公式的邏輯與前一個公式（D2）相似。它檢查 `B2` 儲存格是否為 0。如果是，則結果為 0。否則，它會加總 `BlockList` 工作表中從 `L2` 到 `L7` 儲存格範圍內的所有數值。這個總和代表了與該安全目標相關的所有硬體區塊的總**暫態**失效率。

**物理意義**：
這個公式計算的是與特定安全目標違規（SGV）相關的所有硬體元件的 **初始總暫態失效率（Initial Total Transient Failure Rate）**，單位同樣是 FIT。與永久失效不同，暫態失效（Transient Failures 或 Soft Errors）是由於環境因素（如宇宙射線中的中子或 alpha 粒子）引起的暫時性錯誤，它們會導致記憶體單元（Flip-flop, SRAM）的狀態翻轉，但不會對硬體造成永久性損傷。此計算結果代表了在沒有任何安全機制時，硬體發生暫態失效的基礎概率。

**標準依據**：
- **失效率數據來源**：暫態失效率的評估比永久失效率更複雜，其數據可能來自 **IEC TR 62380**，但更常見的是依賴晶片供應商根據具體的製程技術和晶片特性進行的模擬和實測數據。ISO 26262 承認暫態失效是需要考慮的隨機硬體失效之一。
- **FMEDA 方法學**：將暫態失效納入考量是完整 FMEDA 的一部分，符合 **ISO 26262-5** 的要求，即必須分析所有可預見的隨機硬體失效。

**為什麼要這樣算**：
為了區分兩種根本不同的失效類型並採取不同的應對策略。永久失效通常需要透過硬體冗餘來應對，而暫態失效則可以透過錯誤偵測與校正碼（如 ECC）、三重模組冗餘（TMR）或通訊協定層級的重傳機制來緩解。獨立計算暫態失效率是評估這些特定安全機制有效性的前提。

**ASIC 部門注意事項**：
- **製程節點的影響**：暫態失效率對半導體製程的先進程度高度敏感。隨著製程節點的縮小（例如從 28nm 到 7nm），單一電晶體的尺寸和工作電壓降低，使其更容易受到高能粒子的影響。ASIC 團隊必須提供詳細的製程資訊以供評估。
- **記憶體與邏輯的區分**：暫態失效主要影響儲存元件。ASIC 團隊應提供設計中 SRAM 和 Flip-flop 的數量、類型和容量，因為它們是暫態失效分析的主要對象。
- **暫態失效的安全機制**：設計團隊應主動考慮針對暫態失效的安全機制。例如，在關鍵的暫存器上使用 TMR，在記憶體介面上使用 ECC 或 Parity Check。這些機制的覆蓋率將在後續的 FMEDA 分析中被量化評估。

### 公式 6：Initial / Total Failure Rate permanent and transient [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | F2 |
| 公式原文 | `=D2+E2` |
| 應用範圍 | F2 到 F23（共 21 個儲存格） |
| 計算值 | 0 |

**參照來源**：
- 目前工作表的 `D2` 儲存格 (永久失效率)
- 目前工作表的 `E2` 儲存格 (暫態失效率)

**計算邏輯（白話文）**：
此公式非常直接，就是將同一列的 `D2`（總永久失效率）和 `E2`（總暫態失效率）兩個儲存格的數值相加。

**物理意義**：
這個計算結果代表了與特定安全目標違規（SGV）相關的所有硬體元件的 **總體初始失效率（Overall Initial Failure Rate）**。它綜合了永久失效和暫態失效兩種隨機硬體失效的來源，給出了一個在沒有任何安全機制保護下的總失效率基準值。這個數值是後續計算所有安全指標（SPFM, LFM, PMHF）的基礎。

**標準依據**：
**ISO 26262-5, Clause 8** 要求對硬體架構進行量化評估，以衡量其對抗隨機硬體失效的能力。計算總失效率是進行此類評估的第一步，因為它定義了需要被安全機制所覆蓋的「總風險」。

**為什麼要這樣算**：
為了得到一個單一的、全面的失效率指標，以便於進行更高層次的分析。雖然永久和暫態失效的應對策略不同，但在計算總體的安全指標時，需要將它們的影響合併計算。這個總和代表了硬體元件「內在」的不可靠性程度。

**ASIC 部門注意事項**：
- **總體 FIT 預算**：在專案初期，功能安全團隊通常會根據目標 ASIL 等級，為整個晶片或關鍵 IP 設定一個總體 FIT 率的「預算」。ASIC 設計團隊需要將這個總 FIT 率作為一個關鍵的設計約束，並在設計過程中持續監控邏輯閘數和記憶體用量，以確保不超出預算。
- **設計權衡**：這個公式明確顯示了永久和暫態失效率對總失效率的貢獻。如果總 FIT 超標，ASIC 團隊可以與安全團隊討論，是應該優化邏輯設計以降低永久失效率，還是增加更多針對暫態失效的保護措施（如增加 ECC 或 TMR），這是一個重要的設計權衡（Trade-off）。

### 公式 7：Safety related Failure Rate permanent [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | G2 |
| 公式原文 | `=IF(B2=0,0,BlockList!$D$2)` |
| 應用範圍 | G2 到 G21（共 20 個儲存格） |
| 計算值 | 0 |

**參照來源**：
- 目前工作表的 `B2` 儲存格
- `BlockList` 工作表的 `$D$2` 儲存格

**計算邏輯（白話文）**：
此公式檢查 `B2` 儲存格是否為 0。如果是，則結果為 0。否則，它會從 `BlockList` 工作表中，複製儲存格 `$D$2` 的內容。這個值代表了被定義為「安全相關」的硬體區塊的永久失效率。

**物理意義**：
這個公式計算的是硬體元件總永久失效率中，被歸類為 **「安全相關」（Safety-related）** 的那一部分的失效率。一個失效如果可能導致安全目標被違反，那麼它就是安全相關的。將失效率分解為「安全相關」和「非安全相關」是功能安全分析的核心步驟，因為只有安全相關的失效才需要被安全機制所覆蓋，並納入後續的硬體架構指標（SPFM, LFM）計算中。

**標準依據**：
- **ISO 26262-1, Clause 3.141** 定義了「安全相關」（Safety-related）。如果一個元件實現了安全功能，或者其失效會違反安全目標，那麼它就是安全相關的。
- **ISO 26262-5, Clause 7.4.3** 描述了需要對硬體零件的失效率進行分配，區分出安全相關的部分。

**為什麼要這樣算**：
為了將分析的焦點集中在真正影響安全的失效上。非安全相關的失效（例如，一個僅用於除錯的介面發生故障）不會導致車輛層級的危害，因此在計算 SPFM 和 LFM 時可以被忽略。這種區分使得安全設計更具針對性，避免了在不必要的地方過度設計，從而節省成本和資源。

**ASIC 部門注意事項**：
- **協同進行分類**：一個硬體模組是否為「安全相關」，是一個需要由 ASIC 設計團隊和功能安全團隊共同決定的關鍵判斷。ASIC 團隊需要提供足夠的設計細節，以支持這個分類決策。
- **設計隔離**：在架構設計階段，應考慮將安全相關的邏輯與非安全相關的邏輯進行物理上或邏輯上的隔離。良好的隔離設計可以防止非安全相關部分的故障傳播到關鍵的安全電路中，從而簡化 FMEDA 分析。
- **理解 `BlockList` 的數據來源**：`BlockList!$D$2` 的值本身也是一個計算結果，ASIC 團隊需要清楚它是如何從更底層的邏輯閘數和失效率模型推導出來的，以確保數據的準確性。

| 應用範圍 | H2 到 H21（共 20 個儲存格） |

- `BlockList` 工作表的 `$M$2` 儲存格

**計算邏輯（白話文）**：
此公式的邏輯與 `G2` 儲存格的公式相同，但針對的是暫態失效。它檢查 `B2` 儲存格是否為 0。如果是，則結果為 0。否則，它會從 `BlockList` 工作表中複製儲存格 `$M$2` 的內容，這個值代表了被定義為「安全相關」的硬體區塊的暫態失效率。

**物理意義**：
這個公式計算的是硬體元件總暫態失效率中，被歸類為 **「安全相關」（Safety-related）** 的那一部分。它量化了因環境因素（如高能粒子）導致的、且可能違反安全目標的暫時性硬體錯誤的發生頻率。這是評估針對暫態失效的安全機制（如 ECC、TMR）有效性的關鍵輸入。

**標準依據**：
- **ISO 26262-5, Annex D**：FMEDA 方法論要求對所有相關的失效模式進行分析，包括暫態失效。
- **ISO 26262-11, Clause 5**：提供了在半導體層級應用 ISO 26262 的詳細指南，其中特別強調了對暫態失效（Soft Errors）的分析。

**為什麼要這樣算**：
為了精確評估和管理暫態失效對系統安全的風險。透過分離出安全相關的暫態失效率，可以更有針對性地部署和驗證安全機制。例如，對於一個 ASIL B 的目標，我們需要證明所部署的安全機制能夠充分覆蓋這部分失效率，從而滿足 LFM（潛在故障指標）的要求。

**ASIC 部門注意事項**：
- **精確定位安全相關的儲存單元**：ASIC 設計團隊需要與功能安全團隊合作，精確識別出設計中哪些正反器（Flip-flops）和靜態隨機存取記憶體（SRAM）的暫態失效是安全相關的。這不是一個粗略的估計，而應基於詳細的設計功能分析。
- **選擇合適的保護策略**：知道了哪些是關鍵的儲存單元後，ASIC 團隊可以做出更明智的設計決策。例如，對控制關鍵狀態機的正反器群組採用三重模組冗餘（TMR），而對較大的資料緩衝區則採用 ECC。這種差異化的保護策略是在滿足安全目標的同時，優化晶片面積、功耗和性能的關鍵。
- **提供保護機制的診斷覆蓋率**：當實施了 ECC 或 TMR 等機制時，ASIC 團隊還需要評估這些機制自身的診斷覆蓋率（Diagnostic Coverage, DC），即它們能夠偵測到錯誤的百分比。這個 DC 值將在後續的 LFM 計算中被使用。

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | I2 |
| 公式原文 | `=G2+H2` |
| 應用範圍 | I2 到 I23（共 21 個儲存格） |
| 計算值 | 0 |

**參照來源**：
- 目前工作表的 `G2` 儲存格 (安全相關永久失效率)

**計算邏輯（白話文）**：
此公式將同一列的 `G2`（安全相關的永久失效率）和 `H2`（安全相關的暫態失效率）兩個儲存格的數值相加。

**物理意義**：

**為什麼要這樣算**：
為了得到一個統一的、用於計算硬體架構指標的基準值。SPFM 和 LFM 旨在評估安全機制的有效性，而它們評估的對象正是這些「安全相關」的失效。將永久和暫態失效合併計算，可以得到一個全面的失效率指標，用以衡量架構的整體穩健性。

**ASIC 部門注意事項**：
- **理解指標的目標值**：ASIL B 等級對 SPFM 和 LFM 有明確的目標值要求（SPFM ≥ 90%, LFM ≥ 60%）。ASIC 設計團隊需要理解，他們設計的安全機制（如 ECC, Parity, TMR, Watchdog 等）的診斷覆蓋率（DC）將直接決定能否達到這些目標值。總安全相關失效率越高，對安全機制的覆蓋率要求就越嚴苛。
- **架構設計的早期輸入**：在設計初期，這個總安全相關失效率的估算值，可以幫助架構師決定需要採用何種等級的安全架構。如果預估的失效率很高，可能就需要考慮採用硬體冗餘等更強的架構方案。
- **成本與安全的平衡**：實現高診斷覆蓋率通常意味著更高的晶片面積和功耗。ASIC 團隊需要在滿足 SPFM 和 LFM 目標的前提下，尋求成本效益最高的安全機制組合。

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | J2 |
| 公式原文 | `=D2-G2` |
| 應用範圍 | J2 到 J23（共 21 個儲存格） |
| 計算值 | 0 |

**參照來源**：
- 目前工作表的 `D2` 儲存格 (總永久失效率)
- 目前工作表的 `G2` 儲存格 (安全相關永久失效率)

**計算邏輯（白話文）**：

**物理意義**：
這個計算結果代表了硬體元件的 **非安全相關永久失效率（Non-Safety-Related Permanent Failure Rate）**。它量化了那些即使發生了也不會導致安全目標被違反的永久性硬體失效的頻率。例如，一個控制除錯日誌輸出的邏輯區塊發生永久失效，它可能會影響除錯，但不會對車輛的行駛安全構成威脅，因此其失效率被歸類於此。

**ISO 26262-5, Clause 7.4.3** 要求對硬體零件的失效模式進行分類。這個公式正是該分類過程的直接體現，它將總失效率分解為「安全相關」和「非安全相關」兩部分。這是 FMEDA 分析中的一個基本步驟。

**為什麼要這樣算**：
為了將分析資源和設計精力集中在真正重要的部分。透過明確分離出非安全相關的失效，團隊可以確保不會在這些失效上投入不必要的安全機制或分析時間。這有助於控制專案成本和複雜性，同時確保滿足安全目標所需的所有要求都得到滿足。這些非安全相關的失效率在計算 SPFM 和 LFM 等硬體架構指標時將被排除在外。

**ASIC 部門注意事項**：
- **確認分類的準確性**：ASIC 設計團隊應與功能安全團隊一起審查失效模式的分類，確保所有可能間接影響安全功能的失效都被正確地識別為「安全相關」。一個常見的錯誤是，認為某個模組與安全無關，但其失效卻可能導致共享資源（如匯流排、記憶體）的鎖死，從而間接影響到安全功能的正常運作。
- **架構上的隔離**：這個計算再次凸顯了在晶片架構中實現「安全島」（Safety Islands）的重要性。將安全相關的邏輯與非安全相關的邏輯清晰地分開，可以讓這個減法運算（`D2-G2`）的結果更加可靠和易於驗證。
- **對非安全相關失效的處理**：儘管這些失效不影響功能安全，但它們可能影響晶片的功能或性能。ASIC 團隊仍需根據產品需求，決定是否需要為這些非安全相關的失效設計糾錯或容錯機制。

### 公式 13：Failure Rate (Safe) permanent [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | M2 |
| 公式原文 | `=FMEDA!CL2` |
| 應用範圍 | M2 |
| 計算值 | 0 |

**參照來源**：
- `FMEDA` 工作表的 `CL2` 儲存格

**計算邏輯（白話文）**：
此公式從更底層、更詳細的 `FMEDA` 工作表中，直接引用單元格 `CL2` 的計算結果。

**物理意義**：
這個值代表 **安全永久失效率（Safe Permanent Failure Rate）**。所謂「安全失效」（Safe Failure），根據 ISO 26262 的定義，是指雖然發生了失效，但其本身不會導致安全目標的違反，或者該失效能被已實現的安全機制所偵測並控制，使系統進入或維持在一個安全的狀態。這個數值量化了所有被歸類為安全的永久性硬體失效的發生頻率，是從 `FMEDA` 表中對每個具體失效模式分析後匯總得出的結果。

- **ISO 26262-1, Clause 3.138**：對「安全失效」（Safe Failure）的官方定義。
- **ISO 26262-5, Clause 8.4.4**：在計算單點故障指標（SPFM）時，安全失效的失效率（λ_sf）是公式分子中的一個關鍵組成部分，代表了架構的穩健性。

**為什麼要這樣算**：
為了量化安全機制的有效性。一個較高的安全失效率，正面地反映了設計中的安全機制（如 ECC、Parity Check、Watchdog 等）能夠成功地偵測和處理大部分潛在的危險故障，將它們轉化為不會導致危害的安全事件。這個數值是計算 SPFM 指標並證明硬體架構滿足 ASIL 目標的關鍵證據。

**ASIC 部門注意事項**：
- **診斷覆蓋率（DC）的直接體現**：`FMEDA!CL2` 這個值的根本來源，是 ASIC 設計團隊為其設計的各種安全機制所提供的診斷覆蓋率（DC）估算。例如，如果一個 ECC 電路的 DC 被評估為 99%，那麼它所保護的記憶體模組的相關失效率中，就有 99% 的部分會貢獻到這個「安全失效」的類別中。因此，提供準確且有依據的 DC 值至關重要。
- **安全機制的實現**：為了最大化安全失效率，ASIC 團隊需要在架構中實現高效的監控和偵測電路，例如電壓/時脈監控器、邏輯內建自測試（LBIST）、錯誤偵測碼以及各種看門狗計時器。
- **失效後的安全狀態**：設計團隊不僅要確保失效能被偵測，還必須保證在偵測到失效後，系統能夠可靠地進入預定義的安全狀態（例如，發出中斷、重置模組或停止操作）。

- `FMEDA` 工作表的 `MG2` 儲存格

**計算邏輯（白話文）**：

為了證明設計對於暫態失效的穩健性。隨著半導體製程越來越先進，暫態失效已成為影響晶片可靠性的主要因素之一。一個高的安全暫態失效率，證明了所採用的安全機制（如 ECC、TMR、Parity）是有效的，能夠應對現實世界中的軟錯誤（Soft Error）威脅。這個數值同樣是計算 SPFM 和 LFM 指標的關鍵輸入。

**ASIC 部門注意事項**：
- **ECC 和 TMR 的設計**：這個數值直接反映了 ASIC 團隊所設計的暫態失效保護機制的成效。高效的 ECC 引擎、精心佈局的 TMR 邏輯，以及對關鍵正反器的保護，都是提高此數值的關鍵。
- **提供準確的 DC**：與永久失效一樣，ASIC 團隊必須為其設計的暫態失效安全機制提供有充分理由支持的診斷覆蓋率（DC）。例如，一個 ECC 機制能偵測並校正單位元錯誤，但可能只能偵測雙位元錯誤，這些細節都會影響 DC 的計算。
- **考慮多位元翻轉（MBU）**：在先進製程中，單一高能粒子可能同時導致多個相鄰記憶體單元的狀態翻轉，即多位元翻轉（Multi-Bit Upset, MBU）。ASIC 團隊需要評估其所選的 ECC 方案是否能有效應對 MBU，並在 FMEDA 中反映出來。

### 公式 15：Failure Rate (Safe) permanent and transient [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | O2 |
| 公式原文 | `=M2+N2` |
| 應用範圍 | O2 到 O23（共 21 個儲存格） |
| 計算值 | 0 |

- 目前工作表的 `M2` 儲存格 (安全永久失效率)
- 目前工作表的 `N2` 儲存格 (安全暫態失效率)

**計算邏輯（白話文）**：
此公式將同一列的 `M2`（安全永久失效率）和 `N2`（安全暫態失效率）兩個儲存格的數值相加。

**物理意義**：
這個計算結果代表了硬體元件的 **總安全失效率（Total Safe Failure Rate）**。它綜合了所有被安全機制成功偵測並控制的硬體失效（包括永久和暫態）。這個數值是衡量整個安全架構有效性的關鍵指標，一個高的總安全失效率是一個正面的訊號，意味著設計中的安全機制是穩健且覆蓋率高的。

**為什麼要這樣算**：
為了得到一個單一的、全面的安全失效率指標。這簡化了硬體架構指標的計算，並提供了一個清晰的頂層視圖，展示了設計中所有安全機制的總體貢獻。

- **向管理者報告的關鍵數據**：在向專案管理層或客戶報告功能安全狀態時，這是一個非常有說服力的數據。它清晰地表明了設計中有多少潛在的失效已經被成功地轉化為無害的安全事件。
- **與診斷覆蓋率的關係**：ASIC 團隊應理解，這個數值與他們提供的診斷覆蓋率（DC）估算直接相關。更高的 DC 意味著更多的失效會被歸類為「安全失效」，從而提高這個總安全失效率，並最終有助於滿足 SPFM 的目標值。

### 公式 16：Failure Rate (Not Safe) permanent [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | P2 |
| 公式原文 | `=FMEDA!CO2-M2` |
| 應用範圍 | P2 |
| 計算值 | 0 |

**參照來源**：
- `FMEDA` 工作表的 `CO2` 儲存格
- 目前工作表的 `M2` 儲存格 (安全永久失效率)

**計算邏輯（白話文）**：
此公式從 `FMEDA` 工作表中的一個匯總值 (`CO2`)，減去本表中已計算出的「安全永久失效率」(`M2`)。這裡的 `FMEDA!CO2` 代表的是與安全目標相關的總永久失效率。

**物理意義**：
這個計算結果是 **不安全永久失效率（Not Safe Permanent Failure Rate）**。它代表了在所有與安全目標相關的永久性硬體失效中，**未能**被任何安全機制所偵測或控制的那一部分。這些失效是危險的，因為它們有潛力直接或間接地導致安全目標的違反。這個數值是後續分析單點失效（Single-Point Faults）和殘餘失效（Residual Faults）的基礎。

**標準依據**：
- **ISO 26262-5, Clause 8**：整個硬體架構指標的評估都建立在對失效進行正確分類的基礎上，即分為安全（Safe）、單點（Single-Point）、殘餘（Residual）和潛在（Latent）等類別。此公式是進行該分類的關鍵一步。
- **失效分類**：總安全相關失效率 (λ_SR) = 安全失效率 (λ_S) + 不安全失效率 (λ_NS)。因此，不安全失效率 = 總安全相關失效率 - 安全失效率。

**為什麼要這樣算**：
為了精確量化設計中剩餘的、未被安全機制覆蓋的風險。這個「不安全」的失效率是功能安全工程師最關注的數字之一，因為它直接關係到系統的殘餘風險。將它從總失效率中分離出來，可以讓團隊清楚地看到現有安全機制的覆蓋率缺口在哪裡，並據此決定是否需要增加或改進安全機制。

**ASIC 部門注意事項**：
- **診斷覆蓋率 (DC) 的反面**：這個數值直接反映了 ASIC 團隊所設計安全機制的診斷覆蓋率 (DC) 的不足之處。如果 DC 是 90%，那麼就有 10% 的失效率會落入這個「不安全」的類別。因此，降低這個數值是 ASIC 團隊在安全設計上的核心目標之一。
- **審查 FMEDA 表**：ASIC 團隊應與功能安全團隊一起，仔細審查底層的 `FMEDA` 工作表，特別是那些被標記為「不安全」的失效模式。需要逐一確認，是否真的沒有任何辦法可以偵測到這些失效，或者是否可以透過修改設計來提高對這些失效的覆蓋率。
- **對 PMHF 的直接貢獻**：這個不安全的失效率是計算 PMHF（Probabilistic Metric for Hardware Failure，硬體失效概率指標）的主要貢獻者之一。為了達到目標 ASIL 等級的 PMHF 要求（例如 ASIL B < 100 FIT），必須嚴格控制這個數值。

- **對 PMHF 的貢獻**：與永久失效一樣，不安全的暫態失效也是計算 PMHF 的一個主要來源。在先進製程中，暫態失效率甚至可能超過永久失效率，因此嚴格控制這個數值對於滿足 PMHF 目標至關重要。

### 公式 19：Failure Rate (Potential to violate SG directly - SPF) permanent [FIT]

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | S2 |
| 公式原文 | `=SUMIFS(FMEDA!$BO$4:$BO$1653,FMEDA!$L$4:$L$1653,"S")` |
| 應用範圍 | S2 |
| 計算值 | 0 |

**參照來源**：
- `FMEDA` 工作表的 `$BO$4:$BO$1653` 儲存格範圍
- `FMEDA` 工作表的 `$L$4:$L$1653` 儲存格範圍

**計算邏輯（白話文）**：
此公式在 `FMEDA` 工作表中進行條件加總。它會檢查 `$L$4:$L$1653` 這個範圍內的每個儲存格，如果儲存格的內容是 "S"（代表 Single-Point Fault，單點故障），它就會將同一列的 `$BO$4:$BO$1653` 範圍內對應的數值（即該單點故障的永久失效率）加總起來。

**物理意義**：
這個計算結果是 **單點故障永久失效率（Single-Point Failure Permanent Rate）**。單點故障（SPF）是功能安全中最危險的一種失效，因為它「單獨一個」就能直接導致安全目標的違反，並且系統中沒有任何安全機制能夠偵測到它。這個數值量化了所有此類危險的永久性硬體失效的總發生頻率。

- **ISO 26262-1, Clause 3.158**：對「單點故障」（Single-Point Fault）的官方定義。
- **ISO 26262-5, Clause 8.4.4**：在計算單點故障指標（SPFM）時，單點故障的失效率（λ_spf）是從總安全相關失效率中必須被減去的部分，代表了架構的弱點。

**為什麼要這樣算**：
為了精確識別和量化設計中最高風險的失效。單點故障是必須被優先處理和消除的。透過 `SUMIFS` 將所有被標記為 "S" 的失效模式的失效率加總，可以得到一個清晰的指標，顯示設計中存在多少未被覆蓋的、直接違反安全目標的風險。這個計算是評估硬體架構是否滿足 ASIL B 或更高等级 SPFM 指標（SPFM ≥ 90% for ASIL B）的關鍵步驟。

**ASIC 部門注意事項**：
- **零單點故障的目標**：對於任何有志於通過 ISO 26262 認證的設計，尤其是 ASIL B 及以上，其核心目標之一就是 **消除所有單點故障**。理想情況下，這個 `S2` 單元格的計算結果應該為 0。任何非零值都代表著一個嚴重的設計缺陷，必須被解決。
- **審查 FMEDA 中的 "S" 標記**：ASIC 團隊必須與功能安全團隊一起，對 `FMEDA` 表中每一個被標記為 "S" 的失效模式進行根源分析（Root Cause Analysis）。需要回答的問題是：為什麼這個失效會直接違反安全目標？以及，為什麼沒有任何安全機制能偵測到它？
- **增加診斷覆蓋率**：消除單點故障的唯一方法就是增加安全機制的診斷覆蓋率。如果一個失效被偵測到，它就不再是單點故障，而會被歸類為「殘餘故障」（Residual Fault）或「被偵測到的多點故障」（Detected Multi-Point Fault）。ASIC 團隊需要為這些被標記為 "S" 的失效模式設計專門的監控或偵測電路。

- **零單點故障目標**：與永久失效一樣，由暫態失效引起的單點故障也必須被消除。此計算結果應為 0。
- **審查 FMEDA 中的 "S" 標記**：ASIC 團隊需要審查每一個被標記為 "S" 的暫態失效模式，理解為何現有的軟錯誤保護機制（如 ECC, Parity, TMR）會錯過它。
- **強化軟錯誤保護**：如果此值不為 0，ASIC 團隊必須強化其軟錯誤保護策略。這可能意味著需要使用更強的 ECC 演算法，擴大 TMR 的應用範圍，或為關鍵的控制邏輯增加專門的監控器。
