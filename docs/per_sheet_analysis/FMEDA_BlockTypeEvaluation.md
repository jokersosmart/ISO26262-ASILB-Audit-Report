# BlockTypeEvaluation 公式分析報告

此報告針對 `BlockTypeEvaluation` 工作表中總共 139 個獨特公式進行詳細分析。

### 公式 1：引用功能區塊類型

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `A2` |
| 公式原文 | `=BlockList!B2` |
| 應用範圍 | A2 到 A7（共 6 個儲存格） |
| 計算值 | `Si MOS: Digital circuits, Micros, DSP` |

**參照來源**：
- `BlockList!B2`

**計算邏輯（白話文）**：
此公式直接從 `BlockList` 工作表中對應的儲存格（例如 B2）讀取預先定義好的功能區塊類型名稱。

**物理意義**：
在 FMEDA 分析中，此欄位用於標識當前正在分析的硬體元件或功能區塊的類別。這有助於對應到正確的基礎失效率模型（如 IEC TR 62380 或 SN 29500 中的分類）。

**標準依據**：
此為分析流程中的資料引用步驟，本身不直接對應特定標準章節，但其目的是為了後續依據 ISO 26262-5 附錄 D、IEC TR 62380 或 SN 29500 進行失效率估算。

**為什麼要這樣算**：
為了確保分析的一致性與可追溯性，所有功能區塊的名稱與分類都應集中在 `BlockList` 中管理，本工作表僅做引用，避免資料分散造成錯誤。

**ASIC 部門注意事項**：
ASIC 部門需確保 `BlockList` 中定義的區塊名稱與實際設計中的模組劃分保持一致。當設計發生變更（如模組合併、拆分或更名）時，必須同步更新 `BlockList`。

### 公式 2：引用功能區塊的基礎永久性 (permanent)失效率

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `B2` |
| 公式原文 | `=BlockList!C2` |
| 應用範圍 | B2 到 B7（共 6 個儲存格） |
| 計算值 | `0` |

**參照來源**：
- `BlockList!C2`

**計算邏輯（白話文）**：
此公式從 `BlockList` 工作表讀取指定功能區塊的基礎永久性 (permanent)失效率（FIT）。這個值本身可能是在 `BlockList` 或其他更底層的工作表（如 `FailureRateCalcIC`）中計算出來的。

**物理意義**：
代表該功能區塊在不考慮任何安全機制的覆蓋下，其固有的永久性 (permanent)故障率，單位為 FIT（Failure in Time），即每十億小時（10^9 hours）發生故障的次數。

**標準依據**：
失效率的估算方法依據 IEC TR 62380 或 SN 29500。ISO 26262-5 附錄 D 提供了將基礎失效率整合到 FMEDA 分析中的指導原則。

**為什麼要這樣算**：
這是 FMEDA 分析的起點。我們需要先知道一個元件的基礎永久性 (permanent)失效率，才能接著評估安全機制對其的影響，並計算殘餘風險。

**ASIC 部門注意事項**：
ASIC 部門提供的 Gate Count（邏輯閘數）或對應的晶片面積是計算基礎失效率的關鍵輸入。部門內應建立一套標準流程來準確統計不同 IP/模組的邏輯閘數，並註明其使用的製程技術與 library。

### 公式 4：通用失效率計算

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `D2` |
| 公式原文 | `=B2+C2` |
| 應用範圍 | D2 到 D7（共 6 個儲存格） |
| 計算值 | `#VALUE!` |

**參照來源**：
- `None!B2`
- `None!C2`

**計算邏輯（白話文）**：
這是一個通用計算公式，通常用於對不同類別的失效率進行加總或相減，以區分不同故障類別的貢獻。

**物理意義**：
其物理意義取決於輸入項。例如，`=B2-E2` 是計算『非安全相關』的永久失效率，即從總永久失效率中減去安全相關的部分。`=T2+AI2` 則是將『殘餘故障』和『潛在故障』的失效率相加，得到總的『不安全』失效率。

**標準依據**：
計算本身遵循基本的數學運算。其在功能安全中的應用需參考 ISO 26262-5 附錄 D 的失效率分類與計算指導。SPFM 和 LFM 的計算公式定義在 ISO 26262-5 的 8.4.4 和 8.4.5 章節。

**為什麼要這樣算**：
為了將底層的失效率數據逐步匯總成頂層的安全度量（SPFM, LFM, PMHF），需要進行一系列的加、減、乘、除運算，區分安全相關、非安全相關、單點故障、殘餘故障、潛在故障等不同類別。

**ASIC 部門注意事項**：
ASIC 部門應確保提供給功能安全團隊的基礎數據（如 Gate Count）是準確的，因為這些基礎數據的任何微小誤差都可能在逐層計算中被放大。同時，對於設計中的安全機制，需要提供其診斷覆蓋率（DC）的估算依據。

### 公式 5：匯總與安全相關的永久性 (permanent)失效率

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `E2` |
| 公式原文 | `=SUMIFS(BlockList!$F$10:$F$209,BlockList!$R$10:$R$209,A2)` |
| 應用範圍 | E2 到 E7（共 6 個儲存格） |
| 計算值 | `#DIV/0!` |

**參照來源**：
- `BlockList!$F$10:$F$209`
- `BlockList!$R$10:$R$209`
- `None!A2`

**計算邏輯（白話文）**：
此公式使用 `SUMIFS` 函數，在 `BlockList` 工作表中查找所有被標記為『與安全相關』且屬於特定區塊類型（條件為 A2）的模組，並將它們的永久性 (permanent)失效率（位於 F 欄或 G 欄）加總。

**物理意義**：
計算結果代表了所有與執行安全功能相關的硬體模組的總永久性 (permanent)失效率。這個值是後續計算 SPFM、LFM 等安全度量的基礎。

**標準依據**：
ISO 26262-1 將『與安全相關』定義為『如果失效可能導致違反安全目標』。此計算是遵循 ISO 26262-5 中關於硬體架構度量（SPFM, LFM）和隨機硬體失效機率度量（PMHF）的計算要求。

**為什麼要這樣算**：
FMEDA 的核心是評估『與安全相關』的部分。我們需要將這部分的失效率從總失效率中分離出來，因為只有這部分失效才會直接影響安全目標。非安全相關部分的失效則不在此計算範圍內。

**ASIC 部門注意事項**：
ASIC 部門必須與功能安全團隊密切合作，準確識別出設計中哪些模組是『與安全相關』的。這個識別過程（通常稱為 Safety Analysis 或 HARA）的準確性，直接決定了整個 FMEDA 分析的有效性。

- `FMEDA!$BR$4:$BR$1653`

**計算邏輯（白話文）**：
此公式使用 `SUMIFS` 函數，在 `BlockList` 工作表中查找所有被標記為『與安全相關』且屬於特定區塊類型（條件為 A2）的模組，並將它們的暫態 (transient)失效率（位於 F 欄或 G 欄）加總。

- `FMEDA!$BR$4:$BR$1653`

**計算邏輯（白話文）**：
此公式使用 `SUMIFS` 函數，在 `BlockList` 工作表中查找所有被標記為『與安全相關』且屬於特定區塊類型（條件為 A2）的模組，並將它們的暫態 (transient)失效率（位於 F 欄或 G 欄）加總。

**物理意義**：
計算結果代表了所有與執行安全功能相關的硬體模組的總暫態 (transient)失效率。這個值是後續計算 SPFM、LFM 等安全度量的基礎。

FMEDA 的核心是評估『與安全相關』的部分。我們需要將這部分的失效率從總失效率中分離出來，因為只有這部分失效才會直接影響安全目標。非安全相關部分的失效則不在此計算範圍內。

- `FMEDA!$BR$4:$BR$1653`

**計算邏輯（白話文）**：
此公式使用 `SUMIFS` 函數，在 `BlockList` 工作表中查找所有被標記為『與安全相關』且屬於特定區塊類型（條件為 A2）的模組，並將它們的暫態 (transient)失效率（位於 F 欄或 G 欄）加總。

**物理意義**：
計算結果代表了所有與執行安全功能相關的硬體模組的總暫態 (transient)失效率。這個值是後續計算 SPFM、LFM 等安全度量的基礎。
