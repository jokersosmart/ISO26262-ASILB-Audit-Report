# BlockEval-IEC61508_opt2 公式分析報告

此報告詳細分析了 **BlockEval-IEC61508_opt2** 工作表中包含的 **36** 個獨特公式。

### 公式 1：儲存格 A2 的計算邏輯

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `A2` |
| 公式原文 | `=BlockList!A10` |
| 應用範圍 | A2 到 A201（共 200 個儲存格） |
| 計算值 | 1 |

**參照來源**：
- `BlockList!A10`

**計算邏輯（白話文）**：
此公式直接從 `BlockList` 工作表的 `A10` 儲存格中提取一個序號。這是一個參照操作，用於在當前工作表中標識和追蹤對應的硬體組件。

**物理意義**：
這個值代表了 FMEDA 分析中特定硬體組件（Block/Subblock）的唯一識別碼。在複雜的晶片設計中，每個功能單元都需要一個獨立的編號，以便於追蹤其失效模式、安全機制以及對整體安全目標的影響。

**標準依據**：
此操作本身不直接源於特定標準的計算章節，而是遵循了 ISO 26262-8:2018 Clause 6 (Configuration management) 和 Clause 7 (Change management) 中對可追溯性（Traceability）的要求。為每個組件分配唯一標識符是實現可追溯性的基礎。

**為什麼要這樣算**：
為了確保功能安全分析的完整性和可追溯性，每一個被分析的硬體組件都必須能夠被唯一識別。透過從 `BlockList`（應為組件清單）中引用編號，可以確保 `BlockEval`（組件評估）工作表中的每一行都與一個明確定義的硬體單元相關聯，避免混淆或遺漏。

**ASIC 部門注意事項**：
ASIC 部門需要確保 `BlockList` 中定義的每一個組件都與實際的 RTL (Register-Transfer Level) 設計模組或主要實體 IP (Intellectual Property) 一一對應。當設計發生變更時，必須同步更新 `BlockList`，以確保 FMEDA 分析始終基於最新的晶片架構。序號的連續性和唯一性也需要工具或流程來保證。

**參照來源**：

**計算邏輯（白話文）**：
此公式從 `BlockList` 工作表的 `B10` 儲存格中提取硬體組件（Block / Subblock）的名稱。與公式 1 類似，這是一個直接的參照操作。

**物理意義**：
組件名稱是將 FMEDA 分析與實際晶片設計文件（如架構圖、RTL 代碼）聯繫起來的關鍵。它為後續的失效率計算和安全機制評估提供了明確的對象，確保分析的內容與設計實體相符。計算結果為 `-` 表示該行可能是分隔符或註釋，不代表一個實際的硬體組件。

**標準依據**：
與公式 1 相同，此操作主要遵循 ISO 26262-8:2018 中關於可追溯性（Traceability）和配置管理（Configuration Management）的要求。一個清晰、人類可讀的名稱是管理硬體組件生命週期的基礎。

**為什麼要這樣算**：
採用直接參照的方式可以確保資料的一致性。當硬體組件的名稱在中央清單 `BlockList` 中被更新時，此變更會自動反映到所有相關的評估工作表中，從而減少手動輸入錯誤的風險，並維持單一事實來源（Single Source of Truth）。

**ASIC 部門注意事項**：
應建立一套標準化且清晰的組件命名規則。命名應能輕易地將 FMEDA 中的名稱對應到 Verilog/VHDL 程式碼中的模組。避免使用模糊或過於通用的名稱，並建議在命名中體現設計的層次結構（例如 `CPU_CORE.FPU.ADDER`）。

**參照來源**：
- `BlockList!C10`

**計算邏輯（白話文）**：
此公式從 `BlockList` 工作表的 `C10` 儲存格中提取硬體組件的類型（Block / Component Type）或其所屬的群組（Block Group）。這同樣是一個直接的資料參照操作。

**物理意義**：
此欄位用於將硬體組件進行分類，例如將其歸類為「邏輯單元（Logic）」、「記憶體（Memory）」或「介面（Interface）」。這種分類對於後續套用統一的分析假設至關重要，例如針對不同類型的組件使用不同的失效率估算模型（例如，對邏輯單元使用基於門數的計算，而對記憶體則使用基於面積的計算）。計算結果為 `-` 表示該行可能是一個分隔符或註釋行，不代表一個實際的硬體組件。

**標準依據**：
這種分層和分類的分析方法符合 ISO 26262 和 IEC 61508 的指導原則。特別是 ISO 26262-5:2018 的附錄 D（Annex D），在說明硬體架構度量指標的計算範例時，就體現了將組件分組處理的概念。對於複雜的系統單晶片（SoC），這種分組是簡化分析的必要手段。

**為什麼要這樣算**：
將組件按類型分組，可以在分析中對相似的組件套用一致的參數，例如相同的失效率模型或診斷覆蓋率（Diagnostic Coverage）。這不僅提高了 FMEDA 的分析效率，也確保了結果的一致性。透過參照中央清單，任何對組件分類的修改都能自動更新到所有相關的分析中。

**ASIC 部門注意事項**：
ASIC 設計團隊必須與功能安全團隊密切合作，定義一套清晰且全面的組件類型。此分類應基於底層的技術和設計特性（例如，標準邏輯單元、SRAM、類比巨集等）。所選擇的組件類型（Block Group）將直接影響其後續使用的失效率預測模型（例如 IEC TR 62380 或 SN 29500），因此這個分類對分析結果的準確性至關重要。

### 公式 4：儲存格 D2 的計算邏輯

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `D2` |
| 公式原文 | `=IF(BlockList!L10=0,0,IF(BlockList!P10="",0,IF(BlockList!$R10=BlockList!$B$2,BlockList!$C$2,IF(BlockList!$R10=BlockList!$B$3,BlockList!$C$3,IF(BlockList!$R10=BlockList!$B$4,BlockList!$C$4,IF(BlockList!$R10=BlockList!$B$5,BlockList!$C$5,IF(BlockList!$R10=BlockList!$B$6,BlockList!$C$6,IF(BlockList!$R10=BlockList!$B$7,BlockList!$C$7,0))))))*IF(BlockList!$R$2="Area",BlockList!$M10/IF(BlockList!$R10=BlockList!$B$2,BlockList!$F$2,IF(BlockList!$R10=BlockList!$B$3,BlockList!$F$3,IF(BlockList!$R10=BlockList!$B$4,BlockList!$F$4,IF(BlockList!$R10=BlockList!$B$5,BlockList!$F$5,IF(BlockList!$R10=BlockList!$B$6,BlockList!$F$6,IF(BlockList!$R10=BlockList!$B$7,BlockList!$F$7,0)))))),BlockList!$L10/IF(BlockList!$R10=BlockList!$B$2,BlockList!$E$2,IF(BlockList!$R10=BlockList!$B$3,BlockList!$E$3,IF(BlockList!$R10=BlockList!$B$4,BlockList!$E$4,IF(BlockList!$R10=BlockList!$B$5,BlockList!$E$5,IF(BlockList!$R10=BlockList!$B$6,BlockList!$E$6,IF(BlockList!$R10=BlockList!$B$7,BlockList!$E$7,0)))))))))` |

**計算邏輯（白話文）**：
此公式用於計算一個硬體組件的「初始總失效率（Total Initial Failure Rate）」，單位為 FIT。計算邏輯非常複雜，可拆解如下：
1.  **前置檢查**：首先檢查 `BlockList!L10`（可能是門數或一個啟用標誌）是否為 0，或 `BlockList!P10`（可能是組件類型）是否為空。若是，則此組件不進行計算，失效率為 0。
2.  **查找基礎失效率因子**：公式的第一部分是一個手動實現的查找功能。它根據 `BlockList!$R10` 的值（可能是組件的分類），去 `BlockList` 的 `$B$2:$B$7` 範圍內查找對應的 `$C$2:$C$7` 值。這個值可視為一個基礎失效率或換算因子。
3.  **選擇計算模型（面積 vs. 門數）**：公式的第二部分根據 `BlockList!$R$2` 的值來決定計算方法：
    *   如果 `BlockList!$R$2` 的值是 "Area"，則採用**基於面積的失效率模型**。它會用 `BlockList!$M10`（該組件的面積）除以一個從 `BlockList` 的 `$F$2:$F$7` 中查找到的「總面積」值。
    *   如果不是 "Area"，則採用**基於門數的失效率模型**。它會用 `BlockList!$L10`（該組件的門數）除以一個從 `BlockList` 的 `$E$2:$E$7` 中查找到的「總門數」值。
4.  **最終計算**：將步驟 2 得到的「基礎失效率因子」與步驟 3 得到的「面積或門數比例」相乘，得出該組件的最終失效率。

**物理意義**：
這個公式計算的是一個硬體組件在沒有任何安全機制保護的情況下的「原始失效率」，單位是 FIT（Failure In Time，每十億小時的失效次數）。這是 FMEDA 分析的起點，代表了組件的固有可靠性水平。公式的核心思想是「分配」，即將一個已知的、更高層級（例如，整個晶片或一個 IP）的總失效率，按照一定比例（門數或面積）分配給其內部的各個子組件。這種自頂向下的分配方法在缺乏底層詳細數據時非常實用。

**標準依據**：
-   **失效率分配**：這種按比例分配失效率的方法，雖然沒有在 ISO 26262 中被明確規定為公式，但其邏輯符合 ISO 26262-5:2018 附錄 D 和附錄 E 中對硬體失效率進行估算和分配的原則。它是一種業界常用的實務做法。
-   **FIT/門數關係**：使用門數（Gate Count）或面積（Area）來估算失效率的依據，通常來自於 IEC TR 62380 或 SN 29500 等可靠性數據手冊。例如，IEC TR 62380 的 Table 16 提供了不同製程技術下，每百萬門的基礎失效率參考值。SN 29500 也提供了類似的數據。公式中的 `BlockList!C` 欄位很可能就是基於這些標準計算得出的更高層級 IP 的總失效率。
-   **IF 條件判斷**：使用 `IF` 語句來處理不同情況（例如，門數為零、不同計算模型）是實現 FMEDA 分析靈活性和準確性的基本邏輯操作，符合 ISO 26262 對於分析過程需嚴謹、系統化的要求。

**為什麼要這樣算**：
在晶片設計的早期階段，往往無法獲得每個最底層邏輯單元的精確失效率數據。然而，對於一些複用的 IP（例如 ARM 核心、PCIe 控制器），其供應商通常會提供一個總的失效率 FIT 值。此公式的目的，就是將這個已知的、較為準確的總 FIT 值，合理地「攤分」到構成該 IP 的各個子模組上。使用門數或面積作為分配依據，是因為通常可以假設在相同的製程和工作條件下，一個組件的失效率與其複雜度（以門數或面積衡量）成正比。這是在缺乏更精確數據時，進行定量安全分析的最佳實踐。

**ASIC 部門注意事項**：
-   **提供準確的門數/面積**：ASIC 部門的核心任務是為 FMEDA 提供準確的門數（Gate Count）和面積（Area）數據。這些數據應來自於晶片佈局（Layout）後的報告，並且需要明確其單位（例如，等效兩輸入 NAND 門數）。數據的準確性直接影響失效率分配的合理性。
-   **確認 IP 邊界**：在進行失效率分配時，必須清晰地定義 IP 或子系統的邊界。ASIC 團隊需要與安全團隊確認，`BlockList` 中 `$E$2:$E$7`（總門數）或 `$F$2:$F$7`（總面積）所代表的範圍，與設計層次結構中的模組邊界是否一致。
-   **標示非標準單元**：如果晶片中包含特殊的客製化單元、類比電路或非標準的記憶體，它們的失效率模型可能與標準邏輯單元不同。ASIC 團隊需要主動標示出這些組件，以便功能安全團隊採用更合適的失效率估算方法，而不是簡單地按門數/面積分配。

**計算邏輯（白話文）**：
此公式的結構與用於計算永久性故障率的「公式 4」非常相似，但其目的是計算「暫態故障（Transient Fault）」的初始總失效率。其邏輯如下：
1.  **前置檢查**：檢查 `BlockList!N10`（可能是與暫態故障相關的門數或面積）是否為 0，或 `BlockList!R10`（組件類型）是否為空。若是，則失效率為 0。
2.  **查找基礎失效率**：根據 `BlockList!$R10` 的組件類型，從 `BlockList` 的 `$L$2:$L$7` 範圍內查找一個基礎失效率值。這個值很可能是針對暫態故障的 FIT 費率。
3.  **按比例分配**：將查找到的基礎失效率，乘以一個比例。這個比例是當前組件的規模（`BlockList!$N10`）除以該類型組件的總規模（從 `BlockList` 的 `$N$2:$N$7` 中查找）。
總體而言，此公式將一個更高層級的、針對暫態故障的總失效率，按比例分配給其下的各個子組件。

**物理意義**：
此計算結果代表了硬體組件的「暫態故障失效率」，單位為 FIT。暫態故障（也稱為軟錯誤，Soft Error）是由於外部環境因素（如宇宙射線中的中子或 α 粒子）導致的非破壞性、暫時性的邏輯狀態翻轉。與會造成永久性損壞的永久性故障不同，暫態故障在系統重置或數據刷新後即可恢復。在對可靠性要求極高的車用晶片中，評估暫態故障率對於確保系統的穩定運行至關重要。

**標準依據**：
-   **暫態故障分析**：ISO 26262-11:2018 附錄 B (Annex B) 中詳細討論了半導體中由隨機硬體故障引起的暫態故障。該標準建議對暫態故障進行分析，並評估其對安全目標的影響。此公式的計算是實現這一要求的第一步。
-   **失效率估算**：暫態故障的失效率估算比永久性故障更為複雜，它高度依賴於製程技術、晶片海拔、中子通量等因素。雖然沒有單一的標準公式，但業界通常會使用 JEDEC 的 JESD89A 標準作為測試和計算的基礎。此處使用的分配方法，是一種在缺乏精確模型時，基於已有數據進行工程估算的合理簡化。

**為什麼要這樣算**：
與永久性故障率的計算邏輯相同，此公式採用自頂向下的分配法，是為了在設計初期，將從 IP 供應商或歷史數據中獲得的、針對暫態故障的總體 FIT 值，合理地分配到各個子組件上。這使得在缺乏詳細數據的情況下，也能對暫態故障的影響進行初步的量化評估。

**ASIC 部門注意事項**：

計算結果代表了該組件的「暫態危險失效（Transient Dangerous Fault）」的失效率，即 λ_DT。這是指那些可能導致安全目標被違反的暫態硬體故障。與永久性危險失效一樣，這是 FMEDA 分析的核心關注點。

**標準依據**：
此計算同樣基於 ISO 26262-5:2018, Clause 8.4.2 的故障分類原則，將其應用於暫態故障領域。λ_transient = λ_transient_safe + λ_transient_dangerous。

**為什麼要這樣算**：
為了全面評估 PMHF，必須考慮所有危險失效的來源，包括暫態故障。單獨計算暫態危險失效率，是為了後續將其與永久性危險失效率匯總，得到總的危險失效率，從而進行準確的 PMHF 計算。

**ASIC 部門注意事項**：
- **協助判斷暫態失效的危險性**：ASIC 團隊需要分析暫態故障（如單粒子翻轉，SEU）對模組功能的具體影響。例如，一個發生在控制暫存器中的 SEU 可能比發生在數據路徑中的 SEU 更危險。這種專業判斷是準確分類暫態危險失效的基礎。
- **提供緩解措施資訊**：如果設計中包含了針對暫態故障的緩解措施（如 ECC、冗餘邏輯），ASIC 團隊需要提供這些措施的詳細資訊及其預期的錯誤偵測/校正能力，這將影響後續對殘餘失效率的計算。

- **提供設計證據**：對於每一個被歸類為單點失效的故障模式，ASIC 團隊都需要提供證據，證明確實沒有任何機制可以偵測或控制它。反之，對於那些被認為已覆蓋的故障，也需要提供安全機制的設計文檔和驗證報告。
- **追查計算錯誤**：ASIC 團隊需要協助追查導致 `BlockList!I10` 計算錯誤的原因，以確保 FMEDA 分析的順利進行。

### 公式 15：儲存格 O2 的計算邏輯

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `O2` |
| 公式原文 | `=(G2+H2)/(SUM($G$2:$G$201)+SUM($H$2:$H$201))` |
| 應用範圍 | O2 到 O201（共 200 個儲存格） |
| 計算值 | #DIV/0! |

**參照來源**：
- `目前工作表!G2`
- `目前工作表!H2`
- `目前工作表!$G$2:$G$201`
- `目前工作表!$H$2:$H$201`

**計算邏輯（白話文）**：
此公式計算當前組件的「總安全失效率」（G2+H2），然後除以整個晶片（或被分析的所有組件）的「總安全失效率之和」（SUM($G$2:$G$201)+SUM($H$2:$H$201)）。

**物理意義**：
計算結果代表了該組件的「總失效率（Total Failure Rate, λ）」。這個值是後續所有安全度量（如 SPFM, LFM, PMHF）計算的基礎，它整合了由物理損壞（永久性）和外部干擾（暫態性）引起的所有隨機硬體故障風險。

**標準依據**：
ISO 26262-5:2018, Clause 7.4.3.2 中提到，失效率估算應考慮所有相關的故障模式（failure modes）。將永久性故障和暫態故障的失效率相加，是為了得到一個全面的總體失效率，這完全符合標準的精神。總失效率 λ 是計算 λ_SPF, λ_RF, λ_MPF 等後續指標的起點。

**為什麼要這樣算**：
為了全面評估一個組件的可靠性，必須將所有已知的、顯著的隨機故障來源都納入考量。在先進製程的晶片中，暫態故障的比例可能與永久性故障相當，甚至更高。如果忽略暫態故障，將會嚴重低估整體的安全風險，導致計算出的安全指標（SPFM, LFM, PMHF）過於樂觀，無法真實反映產品的安全性。

**ASIC 部門注意事項**：
ASIC 部門需要確保用於計算永久性故障的數據基礎（例如，總邏輯門數）和用於計算暫態故障的數據基礎（例如，Flip-flop 和 SRAM 的位元數）是清晰且沒有重疊的。在提交數據時，應明確指出各項數據的來源和範圍，以確保功能安全團隊在進行加總時不會重複計算或遺漏某些部分的失效率。

- **SPFM 是設計目標**：ASIC 團隊需要將滿足 SPFM 目標（如 ASIL B >= 97%）作為一個明確的設計目標。這意味著需要盡力消除單點失效，並提高安全機制的診斷覆蓋率以減少殘餘失效。

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `AI2` |
| 公式原文 | `=S2+V2+AE2` |
| 應用範圍 | AI2 到 AI201（共 200 個儲存格） |
| 計算值 | #VALUE! |

**參照來源**：
- `目前工作表!S2`
- `目前工作表!V2`
- `目前工作表!AE2`

**計算邏輯（白話文）**：
此公式將總單點失效率（S2）、總殘餘失效率（V2）和總潛在多點失效率（AE2）相加。

**物理意義**：

**為什麼要這樣算**：
PMHF（Probabilistic Metric for Hardware Failure）是 ISO 26262 中最重要的定量指標，它直接估算了在車輛生命週期內，由於隨機硬體故障導致安全目標被違反的平均概率。為了計算 PMHF，必須將所有可能導致安全目標被違反的故障類型（單點、殘餘、潛在）的失效率都匯總起來。

**ASIC 部門注意事項**：
- **PMHF 是最終目標**：ASIC 團隊的所有安全相關活動，最終都是為了確保整個晶片的 PMHF 值滿足目標要求（例如，ASIL B < 100 FIT）。
- **提供準確的基礎數據**：PMHF 的計算結果直接取決於 ASIC 團隊提供的基礎失效率數據的準確性。任何數據上的偏差都會被直接傳導到最終的 PMHF 結果中。

### 公式 17：儲存格 Q2 的計算邏輯

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | `Q2` |
| 公式原文 | `=J2-M2-P2` |
| 應用範圍 | Q2 到 Q201（共 200 個儲存格） |
| 計算值 | #DIV/0! |

**參照來源**：
- `目前工作表!J2`
- `目前工作表!M2`
- `目前工作表!P2`**計算邏輯（白話文）**：
此公式從「永久性危險失效」的失效率（來自儲存格 J2）中，減去「永久性單點失效」的失效率（來自儲存格 M2）和另一個未知來源的失效率（來自儲存格 P2）。

**物理意義**：
此計算的目的是為了分離出「永久性多點失效（Permanent Multi-Point Fault）」的失效率。根據 ISO 26262，危險失效（λ_D）可以被分解為單點失效（λ_SPF）、殘餘失效（λ_RF）和多點失效（λ_MPF）。因此，λ_MPF = λ_D - λ_SPF - λ_RF。在這個公式中，J2 代表 λ_D（永久性部分），M2 代表 λ_SPF（永久性部分），因此 P2 很可能代表了「永久性殘餘失效（Permanent Residual Fault）」的失效率 λ_RF。計算結果 `#DIV/0!` 源於其輸入 J2 和 M2 的計算錯誤。

**標準依據**：
此公式完全遵循 ISO 26262-5:2018, Clause 8.4.4 中對危險失效的分類方法。該條款定義了 λ_dangerous = λ_SPF + λ_RF + λ_MPF。移項後即可得到此公式的計算邏輯。這是計算 LFM（Latent Fault Metric）和 PMHF 的基礎步驟。
**為什麼要這樣算**：
為了評估系統對潛在失效（Latent Fault）的抵禦能力（LFM 指標），以及計算總體的 PMHF，必須將不同類型的危險失效（單點、殘餘、多點）精確地分開。多點失效是指那些本身不會導致安全目標被違反，但與其他獨立故障組合後可能導致安全目標被違反的故障。識別並量化這部分失效率，對於評估複雜系統的安全性至關重要。

**ASIC 部門注意事項**：
- **協助識別多點失效**：ASIC 團隊需要協助分析，哪些故障模式屬於多點失效。這通常涉及對冗餘架構（如雙核鎖步 Lock-step）或共享資源的分析。例如，在一個雙核鎖步系統中，單個核心的故障如果能被比較器檢測到，那麼它就不是單點失效，而可能是一個多點失效（如果比較器也失效）。
- **澄清 P2 的來源**：由於 P2 存在循環參照問題，ASIC 團隊需要協助澄清此處應有的正確輸入，即「永久性殘餘失效」的失效率是如何得出的。這通常與安全機制的診斷覆蓋率（Diagnostic Coverage, DC）有關（λ_RF = λ_D * (1 - DC)）。
