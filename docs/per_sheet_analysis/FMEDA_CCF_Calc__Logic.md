# FMEDA CCF Calc. Logic 公式分析報告


### 公式 1：計算防禦措施的加權分數 X

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | F5 |
| 公式原文 | `=B5*H5` |
| 應用範圍 | F5 到 F49（共 29 個儲存格） |
| 計算值 | 0 |

**參照來源**：

*   `B5`: 同一張工作表中的儲存格，代表特定防禦措施的評分（Score）。
*   `H5`: 同一張工作表中的儲存格，代表該防禦措施的權重（Weight）。

**計算邏輯（白話文）**：

此公式將 B5 儲存格的「評分」與 H5 儲存格的「權重」相乘，得到一個加權後的分數。

**物理意義**：

在共因失效（Common Cause Failure, CCF）分析中，這個計算代表了某一個具體防禦措施（例如，電路設計上的隔離、冗餘、時脈監控等）對於抵抗潛在共因事件的貢獻程度。不同的防禦措施有不同的重要性（權重），此公式旨在量化每一個措施的有效性。

**標準依據**：

此計算方法屬於 **ISO 26262-5:2018 Annex D** 中描述的「相依性失效分析」（Dependent Failure Analysis, DFA）的一部分，特別是針對共因失效的量化評估。雖然標準沒有規定具體的公式，但採用加權評分法（Weighted Scoring Method）是業界評估 CCF 防禦措施的常見實踐，其精神與 **IEC 61508** 的 β-factor 模型相似，都是為了量化多個組件同時失效的可能性。

**為什麼要這樣算**：

為了系統性地、客觀地評估一組防禦措施的整體強度。透過為每個措施分配權重和評分，安全工程師可以識別出哪些防禦措施最為關鍵，哪些可能需要加強。這個量化結果（β-factor）後續會用於計算 SPFM、LFM 和 PMHF 等安全度量，以判斷系統是否達到目標 ASIL 等級。

**ASIC 部門注意事項**：

*   **權重（H5）的合理性**：ASIC 設計團隊需要與功能安全團隊密切合作，確保分配給每個防禦措施的權重是合理且有依據的。例如，物理隔離（Physical Separation）的權重通常會高於邏輯檢查（Logic Check）。
*   **評分（B5）的客觀性**：評分的依據必須明確。是因為設計本身、驗證結果、還是來自於歷史數據？ASIC 團隊需要提供具體的設計實現證據來支持評分。
*   **設計實現**：ASIC 團隊必須確保在晶片設計中，確實實現了這些被評分的防禦措施，並且其實現方式符合預期，能夠達到宣稱的防禦效果。

### 公式 2：計算防禦措施的加權分數 Y

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | G5 |
| 公式原文 | `=C5*H5` |
| 應用範圍 | G5 到 G49（共 27 個儲存格） |
| 計算值 | 0 |

**參照來源**：

*   `C5`: 同一張工作表中的儲存格，代表特定防禦措施在另一個維度上的評分。
*   `H5`: 同一張工作表中的儲存格，代表該防禦措施的權重（與公式 1 相同）。

**計算邏輯（白話文）**：

此公式將 C5 儲存格的「評分」與 H5 儲存格的「權重」相乘，得到另一個加權後的分數。

**物理意義**：

與公式 1 類似，這個計算同樣是為了量化特定防禦措施對抵抗共因失效的貢獻。欄位 F (X) 和欄位 G (Y) 很可能代表了對防禦措施有效性的兩種不同評估維度。例如，X 可能代表「防禦覆蓋率」，而 Y 可能代表「防禦機制的獨立性」。將兩者分開計算可以提供更細緻的分析視角。

**標準依據**：

此方法同樣基於 **ISO 26262-5:2018 Annex D** 的相依性失效分析（DFA）原則。標準鼓勵對防禦措施進行系統性的評估，而使用多維度加權評分是實現此目標的有效手段之一。這有助於更全面地理解和量化共因失效的耦合因子（Coupling Factor）。

**為什麼要這樣算**：

為了對防禦措施進行更全面的評估。單一維度的評分可能無法完全捕捉一個防禦措施的全部特性。例如，一個措施可能覆蓋範圍很廣（高 X 分），但與其他措施的獨立性較差（低 Y 分）。將這些維度分開評估，可以更精準地識別系統的潛在弱點。

**ASIC 部門注意事項**：

*   **理解 X 和 Y 的區別**：ASIC 團隊必須清楚 X (`B5`) 和 Y (`C5`) 評分所代表的具體工程意義。這會直接影響到設計決策。例如，如果 Y 代表「時序獨立性」，那麼設計團隊就需要考慮使用不同的時脈域（Clock Domain）或非同步介面（Asynchronous Interface）來實現冗餘功能。
*   **避免重複計算**：團隊需要確保為 X 和 Y 提供的設計證據是正交的，避免將同一個設計特性重複用於兩個維度的評分，這會導致對防禦效果的高估。
*   **提供對應的設計證據**：針對 Y 維度的評分，必須提供相應的晶片設計證據。例如，如果評分基於「不同的觸發條件」，則需要展示在 RTL Code 或電路圖中，冗餘模組是如何由不同的事件或訊號觸發的。

### 公式 3-9：計算各項評分的總和

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | B51, C51, D51, E51, F51, G51, H51 |
| 公式原文 | `=SUM(B5:B49)`, `=SUM(C5:C49)`, `=SUM(D5:D49)`, `=SUM(E5:E49)`, `=SUM(F5:F49)`, `=SUM(G5:G49)`, `=SUM(H5:H49)` |
| 應用範圍 | 每個公式僅應用於單一儲存格 |
| 計算值 | B51: 62.5, C51: 50, D51: 55, E51: 46.5, F51: 0, G51: 0, H51: 0 |

**參照來源**：

*   `B5:B49`, `C5:C49`, `D5:D49`, `E5:E49`: 各項防禦措施的原始評分（未加權）。
*   `F5:F49`, `G5:G49`: 各項防禦措施的加權分數 X 和 Y。
*   `H5:H49`: 各項防禦措施的權重。

**計算邏輯（白話文）**：

這些公式分別將 B、C、D、E、F、G、H 各欄位從第 5 行到第 49 行的數值全部加總起來，得到各欄位的總分。

**物理意義**：

*   `F51` 和 `G51` 的總和代表了所有已實施的防禦措施，在 X 和 Y 兩個維度上的**總體加權分數**。這是對整個系統抵抗共因失效能力的初步量化指標。
*   `B51` 到 `E51` 的總和是原始評分的累加，主要用於中間計算或數據核對，其直接物理意義較弱。
*   `H51` 的總和（在此案例中為 0，可能因為權重已在 F/G 欄計算中體現）理論上代表總權重，但在這個工作表中似乎未使用。

**標準依據**：

這是標準加權評分法中的一個步驟，符合 **ISO 26262-5:2018 Annex D** 對於系統性評估 CCF 的要求。將各個獨立措施的貢獻匯總，是得出最終量化指標（如 β-factor）的必要過程。

**為什麼要這樣算**：

為了將分散在數十個獨立防禦措施中的評分，匯總成一個或數個能夠代表系統整體水平的宏觀指標。如果沒有這個加總步驟，將無法對系統的整體 CCF 防禦能力進行評估和比較。

**ASIC 部門注意事項**：

*   **確保完整性**：ASIC 團隊需要與安全團隊確認，B5:H49 的範圍是否涵蓋了所有為抵抗 CCF 而設計的相關電路與邏輯。若有遺漏，將導致總分被低估，從而可能誤判系統的安全性。
*   **追溯性**：對於 F51 和 G51 這兩個關鍵的總分，ASIC 團隊應能追溯到其貢獻最大的幾個具體防禦措施。這有助於在後期設計優化或問題排查時，快速定位關鍵電路。

### 公式 10：計算防禦措施的總體分數 S

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | B53 |
| 公式原文 | `=(F51+G51)` |
| 應用範圍 | 僅應用於此儲存格 |
| 計算值 | 0 |

**參照來源**：

*   `F51`: 所有防禦措施的加權分數 X 的總和。
*   `G51`: 所有防禦措施的加權分數 Y 的總和。

**計算邏輯（白話文）**：

此公式將所有防禦措施在 X 維度和 Y 維度上的加權總分相加，得到一個最終的、單一的整體評分 S。

**物理意義**：

分數 S (Score S = x + y) 代表了 SSD Controller 晶片中，針對共因失效（CCF）所實施的**所有防禦措施的總體有效性**。它是一個宏觀指標，將數十個微觀的設計措施（如冗餘、監控、隔離）的貢獻匯總成一個單一的量化數值，用以評估系統整體的 CCF 防禦強度。

**標準依據**：

這個計算是 **ISO 26262-5:2018 Annex D** 中「β-factor 模型」的應用實踐。標準本身沒有定義 `S = x + y` 這個具體公式，但它提供了一個定性的評估表（Table D.1），其中包含多項影響 CCF 的因素。業界普遍採用的方法就是將這些因素量化為加權分數，然後匯總，最終映射到一個 β-factor 值。此公式是這個匯總過程的核心步驟。

**為什麼要這樣算**：

為了得到一個可以用於後續計算的、簡潔的量化指標。單獨看每個防禦措施的分數是難以評估系統整體安全性的。透過將所有分數加總，可以得到一個能夠代表「系統整體」抵抗共因失效能力的數值，這個數值 S 將直接用於計算最終的 β-factor。

**ASIC 部門注意事項**：

*   **分數 S 是設計質量的直接體現**：ASIC 團隊應理解，這個分數 S 直接反映了其安全設計的質量和完整性。如果最終 S 值過低，意味著 ASIC 設計中實現的安全機制不足或效果不佳，需要重新審視設計。
*   **與安全目標的關聯**：這個分數將直接影響到 PMHF（Probabilistic Metric for Hardware Failure）等頂層安全度量的計算結果。一個較低的 S 分數會導致較高的 β-factor，進而推高 PMHF，可能導致無法滿足 ASIL B 的目標值（< 10 FIT）。

### 公式 11：計算考慮到設計多樣性的總體分數 SD

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | B57 |
| 公式原文 | `=F51*(B55+1)+G51` |
| 應用範圍 | 僅應用於此儲存格 |
| 計算值 | 0 |

**參照來源**：

*   `F51`: 所有防禦措施的加權分數 X 的總和。
*   `G51`: 所有防禦措施的加權分數 Y 的總和。
*   `B55`: 一個代表「設計多樣性」（Design Diversity）或類似概念的因子，在此標示為 `z`。

**計算邏輯（白話文）**：

此公式在 `Score S` 的基礎上，對 X 維度的總分 `F51` 乘以一個由 `z` (`B55`) 調整的係數 `(z+1)`，然後再加上 Y 維度的總分 `G51`。這意味著 `z` 因子特別用來增強或調整 X 維度分數的影響力。

**物理意義**：

`Score SD = x * (z + 1) + y` 這個公式計算的是一個考慮了**設計多樣性**後的增強型總分。在功能安全中，如果冗餘的組件不僅僅是簡單的複製，而是採用了不同的設計（例如，由不同的團隊設計、使用不同的演算法或硬體實現），那麼它們同時因為同一個未知設計缺陷而失效的可能性就會大大降低。`z` 因子 (`B55`) 就是用來量化這種設計多樣性的程度。一個較高的 `z` 值，意味著設計多樣性更好，從而會顯著提高總體分數 SD，反映出系統對共因失效更強的抵抗能力。

**標準依據**：

這個公式是 **ISO 26262-5:2018 Annex D** 中關於「透過設計多樣性降低共因失效」思想的具體量化實現。Annex D.2.4 明確指出「不同技術或設計的使用」是防禦 CCF 的有效措施。公式中的 `(z+1)` 是一個放大係數，這種透過特定因子來調整分數權重的方法，是業界在實踐標準要求時，自行定義的量化模型，旨在更精確地反映不同防禦策略的貢-獻。

**為什麼要這樣算**：

因為簡單的冗餘（同構冗餘）和帶有設計多樣性的冗餘（異構冗餘）在抵抗 CCF 方面的能力是不同的。如果只是將所有防禦措施的分數簡單相加（如公式 10），就無法體現出「設計多樣性」這一強力措施的額外價值。此公式透過引入 `z` 因子，特別放大了與設計多樣性相關的評分維度（此處為 X 維度），使得最終的總分能更準確地反映系統的真實安全水平。

**ASIC 部門注意事項**：

*   **`z` 因子的依據**：ASIC 團隊必須為 `z` (`B55`) 的取值提供強有力的證據。例如，如果宣稱兩個冗餘的處理器內核採用了不同的微架構，就需要提供設計文檔、驗證報告來證明其「多樣性」。不能隨意設定一個高的 `z` 值。
*   **X 維度的設計重點**：既然 X 維度的分數會被 `z` 因子放大，那麼 ASIC 團隊在設計時，應重點投入資源來提升與 X 維度相關的防禦措施。需要明確 X 維度到底代表什麼，是演算法多樣性、邏輯實現多樣性、還是時序多樣性？
*   **避免「偽多樣性」**：ASIC 團隊需要警惕「表面上」的多樣性。例如，僅僅交換了兩個相同邏輯單元的輸入順序，並不能算是真正的設計多樣性。真正的多樣性應能在功能、邏輯、時序或物理層面提供實質性的差異。

### 公式 12：根據總分 S 查表計算 β-factor

| 項目 | 內容 |
|:-----|:-----|
| 儲存格 | B59 |
| 公式原文 | `=IF(B53<45, 5, IF(B53<70, 2, IF(B53<120, 1, 0.5)))` |
| 應用範圍 | B59, B61 |
| 計算值 | 5 |

**參照來源**：

*   `B53`: 前面計算出的防禦措施總體分數 S (`Score S`)。

**計算邏輯（白話文）**：

這是一個查表（Lookup）功能的實現。它根據總分 S (`B53`) 的值，返回一個對應的 β-factor（以百分比形式表示，例如 5 代表 5%）。

*   如果總分 S 小於 45，則 β 值為 5%。
*   如果總分 S 在 45 到 69 之間，則 β 值為 2%。
*   如果總分 S 在 70 到 119 之間，則 β 值為 1%。
*   如果總分 S 大於等於 120，則 β 值為 0.5%。

**物理意義**：

此公式是整個 CCF 分析的核心步驟之一，它將前面所有對防禦措施的定性、定量評估（匯總為分數 S），最終轉化為一個在失效率計算中可以直接使用的**共因失效因子（β-factor）**。β-factor 代表了「在一個組件失效的事件中，該失效是由於一個共因事件（導致多個組件同時失效）所引起的機率」。

一個較高的分數 S，意味著防禦措施做得好，系統抵抗 CCF 的能力強，因此會對應一個較低的 β-factor，表示共因失效的機率很低。反之，一個較低的分數 S，則會得到一個較高的 β-factor，意味著系統更容易因為單一原因而導致多重失效。

**標準依據**：

這個分段查表的方法直接源自 **ISO 26262-5:2018, Annex D, Table D.1**。該表格提供了一個定性的評估框架，列出了多項防禦 CCF 的措施，並根據措施的實施程度給出從「-」（無貢獻）到「++」（高貢獻）的評級。此 Excel 中的公式，是將該定性表格**量化**後的一個具體實踐。分數區間（45, 70, 120）和對應的 β 值（5%, 2%, 1%, 0.5%）是公司根據自身經驗、產品特性和對標準的解讀而**自行定義**的，旨在將標準的指導原則轉化為可操作的計算流程。

**為什麼要這樣算**：

為了將抽象的「防禦措施強度」轉化為一個可以用於數學計算的具體數值。功能安全的硬體度量（SPFM, LFM, PMHF）都需要輸入失效率數據。對於冗餘系統，其總失效率不僅包括獨立失效，還必須考慮共因失效。β-factor 就是用來計算這部分共因失效所貢獻的失效率（λ_CCF = λ_total * β）。如果沒有這個轉換步驟，FMEDA 將無法完成對冗餘系統的量化分析。

**ASIC 部門注意事項**：

*   **分數目標的明確化**：ASIC 團隊的設計目標變得非常明確：必須實現足夠多、足夠好的安全機制，使得總分 S 超過某個閾值（例如，達到 70 分以上），以獲得一個足夠低的 β-factor（例如，1% 或更低）。這個目標應在設計初期就確定。
*   **閾值的敏感性**：團隊應注意分數區間的邊界。例如，如果當前設計的總分是 68 分，只差 2 分就能讓 β-factor 從 2% 降至 1%，這可能會對最終的 PMHF 結果產生顯著影響。在這種情況下，增加一個小的、低成本的防禦措施可能會有很高的投資回報率。
*   **公司內部標準的理解**：ASIC 團隊需要理解，這個查表公式是公司內部的標準實踐，而非 ISO 26262 的強制規定。在與外部審核員溝通時，需要能夠解釋這個量化模型的合理性，並提供支持其有效性的證據。
